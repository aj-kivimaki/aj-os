# Architectural Goal

The AJ-OS Schema Engine already models business relationships.

The Module Registry validates them.

However, the Notion Translation Layer currently ignores them.

This milestone teaches the Translation Layer how to convert schema relations into valid Notion relation properties.

Workspace synchronization is still responsible only for creating databases.

Relation values are not synchronized yet.

---

# Your Role

You are a senior TypeScript engineer extending the Notion Translation Layer.

Reuse the existing architecture.

Keep business logic inside the Schema Engine.

Keep infrastructure logic inside the Translation Layer.

---

# Objective

Implement translation of schema relations into Notion relation properties.

The translation should remain entirely automatic.

Business modules continue describing relations using RelationDefinition.

No Notion-specific code should appear inside business modules.

---

# Requirements

Implement relation translation.

Create:

```
src/notion/properties/

relation.ts
```

Update the Translation Layer so that:

- every RelationDefinition becomes a Notion relation property
- relation names become Notion property names
- relation targets resolve through the Module Registry

The translator should prepare relation payloads even if target database IDs are not yet available.

Use placeholders or deferred references if required.

The implementation should make future synchronization straightforward.

---

# Validation

Extend validation.

Verify:

- referenced modules exist
- duplicate relation names are rejected
- relation definitions are internally consistent

Do not require existing Notion databases.

---

# Workspace Synchronization

Workspace synchronization should continue functioning.

Projects and CRM databases should still synchronize successfully.

Relation creation may be skipped gracefully if target database IDs are unavailable.

The application should never fail because relations cannot yet be resolved.

---

# Constraints

Do NOT implement:

- relation synchronization
- relation updates
- rollups
- formula properties
- page linking
- dashboard integration

Infrastructure support only.

---

# Design Goals

The implementation should be:

- deterministic
- reusable
- strongly typed
- infrastructure-specific

Business modules must remain unaware of Notion.
