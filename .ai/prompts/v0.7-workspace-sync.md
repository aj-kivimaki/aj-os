# Architectural Requirement

This milestone officially adopts the Application Layer described in ADR-007.

The existing `src/sync/` directory should be migrated to `src/application/`.

This is a refactoring only.

No existing behavior should change as part of the migration.

Expected structure after the refactor:

```
src/application/

create-database.ts
find-database.ts
workspace-sync.ts
index.ts
```

Update imports, exports and references as necessary.

The Application Layer should become the orchestration layer for AJ-OS.

# Your Role

You are a senior TypeScript engineer extending AJ-OS into a workspace synchronization system.

Reuse the existing architecture.

Avoid unnecessary abstractions.

---

# Objective

Implement the first version of Workspace Synchronization.

The goal is to make database creation idempotent.

Running synchronization multiple times should never create duplicate databases.

---

# Business Context

AJ-OS manages an entire Notion workspace.

It should behave similarly to Infrastructure as Code tools.

Business Modules define the desired workspace.

The Application Layer compares the desired workspace with the existing Notion workspace and performs only the required actions.

This milestone only detects existing databases and prevents duplicates.

Schema updates belong to a future milestone.

---

# Requirements

Implement a synchronization workflow that:

1. Knows which business modules exist.

Initially:

- Projects
- CRM

2. Queries the configured parent page.

3. Detects existing child databases.

4. Matches databases by title.

5. If the database exists:

- Skip creation.

6. If the database does not exist:

- Create it.

7. Print a synchronization summary.

Example:

```
Workspace Synchronization

Projects
✓ Already exists

CRM
✓ Created

Summary

Created: 1
Skipped: 1
Failed: 0
```

---

# Constraints

Do NOT implement:

- Schema comparison
- Property updates
- Relation synchronization
- Database deletion
- Template synchronization
- Dashboard synchronization
- Page synchronization

Only prevent duplicate database creation.

---

# Design Goals

The synchronization process should be:

- deterministic
- idempotent
- modular
- easy to extend

Reuse:

- Business Modules
- Schema Engine
- Translation Layer
- Notion Client

No duplicate logic.

# Definition of Done

The Application Layer is fully established.

The former `sync/` directory has been migrated to `application/`.

Running AJ-OS repeatedly results in the same workspace state.

Projects and CRM are created only once.

Subsequent executions report that they already exist instead of creating duplicates.

The Application Layer becomes the foundation for future workspace management, schema updates, relation synchronization and deployment workflows.
