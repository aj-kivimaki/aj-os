# Overview

Version 1.0.0 is the first complete release of the Knowledge Assistant — and the
first complete product built on AJ-OS. It lets a user ask natural-language
questions about a configured handbook and receive answers that are grounded in
that handbook's generated wiki, each with citations to the exact articles behind
it. It is delivered as a command-line tool, `aj ask`.

This release closes the loop from idea to usable product: the full pipeline —
configuration, handbook discovery, retrieval, context assembly, prompt rendering,
and answer generation — runs end to end behind a single command. For the history
of how it got here, see
[architecture-timeline.md](../architecture-timeline.md); this document does not
retell it.

# New capabilities

User-facing capabilities delivered in 1.0.0:

- **`aj ask` — one-shot mode.** Answer a single question and exit:
  `aj ask "What is the AJ-OS knowledge pipeline?"`.
- **`aj ask` — interactive mode.** Start a session and ask multiple questions in a
  row; leave with `exit`, `quit`, or Ctrl-D.
- **Natural-language retrieval over the handbook wiki.** The assistant finds the
  most relevant wiki articles for a question.
- **Grounded answers with citations.** Answers are written only from the retrieved
  context, with a numbered citation list.
- **Honest "nothing found" handling.** A clear notice when no relevant knowledge
  exists, instead of a fabricated answer.
- **`--debug` diagnostics.** An optional view of what happened behind an answer —
  retrieved articles, context size, model, and per-stage timings — without changing
  the answer itself.
- **Plain-language error handling.** Missing configuration, a missing wiki, or a
  missing API key each produce a clear, actionable message.
- **Configurable handbook and model.** `aj.config.json` selects the handbook;
  `ANTHROPIC_MODEL` optionally selects the model.

For how to use each of these, see [usage.md](../usage.md).

# Architectural milestones

Version 1.0 is also the first release of several durable architectural outcomes.
These are summarized here and detailed in
[architecture-timeline.md](../architecture-timeline.md) and
[decisions.md](../decisions.md):

- **A three-layer architecture** — CLI → Product → Platform — with a strict one-way
  dependency direction.
- **Five new reusable platform capabilities** — Config, Handbook, Retrieval, Prompt
  Renderer, AI Client — plus a **sixth (Context Builder) proven reusable** by being
  consumed through its public contract (with one small assembly enhancement).
- **The render/generate seam** — a pure Prompt Renderer separated from a
  provider-isolated AI Client — which makes the product model-agnostic.
- **Index-driven retrieval** — the generated `wiki/index.md` defines the searchable
  corpus, not a filesystem scan.

# Known limitations

Version 1.0 has deliberate boundaries. These are limitations to be aware of, not
defects:

- **No conversation memory.** Each question is answered independently, even in an
  interactive session.
- **Keyword-based retrieval.** Relevance is term-frequency matching, not semantic
  search; phrasing questions with the handbook's own vocabulary yields the best
  results.
- **Read-only.** The assistant never modifies the handbook or wiki.
- **Handbook-only knowledge.** No internet search or outside knowledge.
- **One question at a time**, with the full answer shown at once (no streaming).
- **No clarifying questions.** Ambiguous questions are answered as best the
  retrieved context allows rather than paused for disambiguation.

### Known issues

- **Interactive banner shows "Version 0.1".** The interactive welcome banner still
  prints *Version 0.1*. This is a cosmetic label only — the functionality is 1.0.0 —
  and is scheduled to be corrected. Tracked for the finalization of this release.
- **Context Builder configuration is not yet honored.** The product requests a
  `documentation` profile, explainability, and Markdown output from the Context
  Builder, but v1.0 Assembly is structural only, so those options currently have no
  effect. See [decisions.md](../decisions.md) (KA-AD-03).

# Intentionally deferred features

The following were consciously postponed — scoped out of 1.0.0 with eyes open, not
overlooked (see [lessons-learned.md](../lessons-learned.md), "Mistakes vs.
conscious trade-offs"):

- **Conversation memory** — threading prior turns into each prompt. The
  render/generate seam already provides its natural home.
- **Semantic / ranked retrieval** (BM25, embeddings) — the naive scorer is a
  deliberate placeholder behind a stable contract.
- **A provider abstraction** — declined while there is a single provider.
- **Streaming, retries, and rate-limit handling** in the AI client.
- **Knowledge capture / editing** — writing back to the handbook belongs to a future
  product.
- **Graphical or web interfaces, and multi-user support.**

# Engineering significance

Version 1.0 matters for reasons beyond its feature list. It is a set of firsts for
AJ-OS:

- **The first complete AJ-OS product.** The point where the platform stopped being
  specifications and started carrying a real, usable product end to end.
- **The first reusable platform.** Six composable platform capabilities now exist,
  validated by an actual consumer rather than by anticipation.
- **The first stable Product / Platform architecture.** The three layers and the
  one-way dependency rule are now a proven template, not a proposal.
- **The foundation for future AJ-OS products.** PRODUCT-002 is expected to begin by
  composing these capabilities and to measure how much it reuses.

In short: the lasting deliverable of 1.0 is not `aj ask` — it is the reusable
platform and the way of working underneath it. The product is how we proved they
work.

# Why Version 1.0 matters

Beyond what it delivers, this release is a historical turning point for the project.

Version 1.0 marks the moment AJ-OS changed from an **architectural experiment** into
a **usable software product**. Everything before it was foundations, standards, and
a platform service with no consumer; Version 1.0 is the first time a person can run
a command and get real value from all of it.

The distinction worth preserving is this: **future releases will *expand* the
platform, not *establish* it.** The hard, one-time work of proving that the
product/platform architecture holds together is done. From here, growth is
addition on a proven base rather than invention of a new one — and that is exactly
what a "1.0" should mean.
