SPEC-003 → SPEC-004 boundary.

# Objective

Define an immutable, validated **`CandidateKnowledge`** contract as the canonical
unit of proposed knowledge, carrying enough metadata for SPEC-004 to group,
deduplicate, and compare against the Handbook.

Proposed shape (finalize during the task, then freeze):

- `id` — stable, namespaced (decision in this task: `session:<sessionId>:<n>` vs.
  content-hash)
- `kind` — lenient enum spanning the **full SPEC-003 §8 candidate taxonomy**:
  `handbook-entry | playbook | wiki-publication | lesson-learned | doc-update |
  automation-idea` (`.catch` to a safe default). The declared domain is the durable
  boundary *vocabulary* shared with SPEC-004, not a v1 capability list — v1 need not
  emit every kind, and a future analyzer can produce a declared kind without
  revising this boundary contract (EOS-003 decision, reviewer AJ). This aligns the
  task with SPEC-003 §8, which lists all six candidate outputs.
- `title`, `body` (markdown), `rationale` (why reusable)
- `provenance` — `{ sessionId, sourceChangeIds[], sourcePaths[], commitHash?,
  generatedAt, generator }`
- `governanceState` — literal `"candidate"` (AJS-006)
- `tags[]`, `related[]` (for SPEC-004 grouping/dedup)
- `confidence?` (optional)

# Implementation Notes

- **This is a boundary contract — bias toward completeness of provenance and
  grouping metadata**, since SPEC-004 cannot recover fields SPEC-003 omits.
- Reuse the compiler's schema idioms: lenient enums via `.catch`, `.min(1)` on
  required strings, arrays defaulting to `[]`, and a `parse…` function that
  strips/validates and deep-freezes (mirror `parseExtraction` /
  `parseContextPackage`).
- `governanceState` is a literal, not free text — it pins the artifact to the
  AJS-006 `Candidate` state.
- Do **not** add fields for SPEC-004's *decisions* (approve/reject/etc.) — those
  belong to SPEC-004's output, not SPEC-003's.
- Record the `id` scheme choice with rationale (stable across re-runs? unique per
  candidate? collision behavior?).

literal, provenance sanity, confidence range, strict unknown-key rejection,
deep immutability), all through the public surface.

Fields: `id`, `kind`, `title`, `body` (markdown), `rationale`,
`provenance {sessionId, sourceChangeIds[], sourcePaths[], commitHash?,
generatedAt, generator}`, `governanceState: "candidate"` (AJS-006 literal),
`tags[]`, `related[]`, `confidence?` (`[0,1]`).

## Decisions (finalized this task; reviewer AJ, distinct from author — AJS-007 §5)

**D-1 — `kind` domain = full SPEC-003 §8 taxonomy (six kinds).**
`handbook-entry | playbook | wiki-publication | lesson-learned | doc-update |
automation-idea`. The declared domain is the durable *boundary vocabulary* shared
with SPEC-004, not a v1 capability list: v1 need not emit every kind, and a future
analyzer can produce a declared kind without revising this frozen contract. This
also **resolved a documentation inconsistency** — the task's original proposed enum
omitted `playbook` (declared in §8) while including deferred kinds; the task doc was
realigned to §8 rather than inventing a third interpretation.

**D-2 — `kind` is required, leniency is for *unknown values only*.** Implemented
with `z.preprocess` (not `.catch`): a present but out-of-domain *string* falls back
to `handbook-entry` (forward-compatible with future kinds), while a **missing** or
non-string `kind` fails validation — so the boundary detects a producer bug (M4
omitting `kind`) instead of silently masking it. `.catch` was rejected because it
cannot distinguish a missing value from an unknown one.

**D-3 — `id` scheme = `session:<sessionId>:<n>`.** An identity/provenance handle,
stable within a session. It deliberately does **not** encode semantic identity or
deduplication (those remain SPEC-004's responsibility). The schema validates `id`
as a non-empty string and documents the scheme rather than enforcing its format, so
the boundary stays stable if id generation evolves (mirrors opaque `Session.id`);
generation itself is M4. Referential sanity is enforced via required non-empty
`provenance.sessionId`.

**Validation.** `npm run typecheck`, `npm run build` green; full suite
378 tests / 35 files (18 new). Reviewed against all 8 code-review angles (no
correctness defects); the one requested change (required `kind`) was applied and
re-validated.

---

# Lessons Learned

`.catch` is the wrong tool when a field must be *present but forward-compatible*:
it collapses "missing" and "unknown" into one silent default, which hides producer
bugs on a boundary contract. `z.preprocess(map-unknown-strings, z.enum(...))` keeps
forward-compatibility for genuinely new values while letting the required enum
reject a missing/garbage value — the pattern to prefer for any lenient-but-required
boundary enum (contrast the internal `Session.trigger`, where `.catch` leniency for
a missing value is acceptable). Surfacing the SPEC-003 §8 vs. task/MILESTONES
`kind` inconsistency for a decision — rather than silently picking — kept the frozen
contract aligned with the specification.
