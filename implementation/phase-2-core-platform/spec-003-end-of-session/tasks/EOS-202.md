# Purpose

Deliver the pipeline's **single non-deterministic stage** — the Knowledge
Extractor — which reads a `ChangeSet`, asks a text-generation model to identify
reusable knowledge, and returns a validated `KnowledgeExtraction`. All model
non-determinism is confined behind an **injected `TextGenerator` port**, so every
surrounding stage — and this stage's own *structure* — stays deterministic and
testable without the network.

> **Why does this task exist?** EOS-201 defined the extraction *contract* and its
> parser. This task supplies the *behavior*: build a deterministic prompt from the
> `ChangeSet`, generate through the port, and parse the response into a
> `KnowledgeExtraction`. It is the direct End-of-Session analog of
> `AnthropicKnowledgeCompiler` (`buildCompilePrompt` → `generator.complete` →
> `parseExtraction`), isolating the LLM exactly as the compiler and the SPEC-002
> provider model do.

---

# Objective

Implement, behind the injected port:

- **`TextGenerator`** — a narrow, injectable text-generation port (structurally
  satisfied by the platform `AIClient`; a stub satisfies it in tests). Owned
  **locally** in the End-of-Session module (see *Design Notes* — port ownership).
- **`KnowledgeExtractor`** — a behavioral interface with
  `extract(changeSet: ChangeSet): Promise<KnowledgeExtraction>`.
- **`buildExtractionPrompt(changeSet: ChangeSet): RenderedPrompt`** — a pure,
  deterministic function that renders the changes into a system+user prompt asking
  for reusable knowledge as strict JSON matching the EOS-201 schema.
- **`createKnowledgeExtractor({ generator }): KnowledgeExtractor`** — wires
  `buildExtractionPrompt` → `generator.complete` → `parseExtractionResponse`,
  returning a frozen handle (the module's factory convention).

The extractor consumes the `ChangeSet` **verbatim** and returns the parsed
extraction; it does **not** generate candidates, attach provenance, compare
against the Handbook, or persist.

## Included

- `src/end-of-session/extraction/` — `TextGenerator.ts` (port + `KnowledgeExtractor`
  interface), `prompt.ts` (`buildExtractionPrompt`), `createKnowledgeExtractor.ts`,
  `index.ts` (mirrors the `analyzers/git/` layout).
- The `TextGenerator` port referencing the platform `RenderedPrompt` / `AIResponse`
  shapes (platform infrastructure, not a cross-spec-module import).
- Deterministic prompt construction from the `ChangeSet` (changes rendered in the
  set's given order; no timestamps or nondeterministic content in the prompt).
- Wiring that passes a sensible `maxTokens` budget (the compiler's
  `COMPILE_MAX_TOKENS` precedent — one session yields a summary plus several
  findings).
- Frozen factory handle; public exports (`createKnowledgeExtractor`,
  `KnowledgeExtractor`, `TextGenerator`, `buildExtractionPrompt`) from the module
  barrel; EOS-007 drift-guard manifest updated.
- **Behaviour & determinism tests with a stub `TextGenerator`** (no network):
  - a canned response → the expected `KnowledgeExtraction`;
  - a **fixed** stub + identical `ChangeSet` ⇒ **deep-equal** extraction across
    runs (structure determinism);
  - `buildExtractionPrompt` is deterministic (same `ChangeSet` ⇒ identical prompt)
    and includes each change's path/type/summary;
  - an empty `ChangeSet` (no changes) and a partial one (changes **and**
    `AnalyzerError`s) both extract without error;
  - a stub returning invalid JSON / a schema-violating object surfaces as
    `ExtractionError` (the extractor does not swallow it);
  - a rejecting stub propagates as a rejection from `extract`.

## Not Included

- The `KnowledgeExtraction` schema/parser/`ExtractionError` — that is EOS-201.
- Any real/Anthropic `TextGenerator` implementation — the platform `AIClient`
  already satisfies the port; the composition root (M5) injects it. No network is
  used in this task's tests (MILESTONES: content isolated behind the port and
  stubbed).
- Candidate generation, provenance attachment, Handbook comparison, persistence,
  projection — M4/M5.
- Prompt-tuning/quality iteration on model output — out of scope for the vertical
  slice; the test asserts *structure*, not prose quality.

# Extractor Invariant (orchestration + structural validation only)

The Knowledge Extractor performs **orchestration and structural validation only**.
Its single responsibility is: render the `ChangeSet` into a prompt, generate
through the injected `TextGenerator`, and parse the response into a validated,
deep-frozen `KnowledgeExtraction`. It **must not** interpret the extracted
knowledge beyond validating the contract. Concretely, the extractor **does not**:

- **classify** — it does not map a finding's soft `kind` hint onto the closed
  `CandidateKind`, re-label, or otherwise decide the meaning of a finding;
- **deduplicate** — it does not detect or collapse repeated/overlapping findings,
  within a run or against anything else;
- **merge** — it does not combine findings, or reconcile the extraction against the
  Handbook, prior sessions, or any external knowledge;
- **score / rank** — it does not compute, adjust, or order by confidence or
  relevance (a model-supplied `confidence` is carried **verbatim**, never derived);
- **enrich / interpret** — it does not add provenance, infer relationships, filter,
  sort, or fill in fields the model did not supply.

The extraction is returned **as the model produced it**, changed only by
fence-stripping, validation, and freezing (EOS-201). These excluded
responsibilities belong **downstream and outside M3**: authoritative kind
classification and provenance attachment are **M4** (candidate generation);
deduplication and Handbook comparison are **SPEC-004**. This invariant is what
keeps the one non-deterministic stage narrow and every surrounding stage
deterministic, and it is asserted in this task's tests (the parsed extraction is
deep-equal to the model's structured content — no field is reordered, derived,
merged, or dropped by the extractor).

# Design Notes

- **Port ownership.** The compiler defines its own `TextGenerator` in
  `KnowledgeCompiler.ts`. The End-of-Session module has so far been deliberately
  **self-contained** (EOS-005/EOS-101 duplicated the `describeFailure` idiom
  "parallel, not shared, to avoid premature cross-module coupling"). Consistent
  with that precedent, this task defines an **EOS-local `TextGenerator`**
  (structurally identical, satisfied by the same platform `AIClient`) rather than
  importing the compiler's. The Planning Review **ratified** this and judged **no
  decision record (EOS-D6) warranted**: the ownership follows the existing
  architectural pattern, so there is no genuine architectural choice to record.
  Promoting a single shared `TextGenerator` remains a future option **only if** a
  second consumer justifies it — a promotion, not a fix.
- **Referencing platform shapes is not cross-spec coupling.** Using `RenderedPrompt`
  and `AIResponse` from `src/platform/` is depending on shared *infrastructure*
  (the same way the compiler does), not on another *specification's* module — it
  respects the dependency-direction rule.
- **Determinism boundary.** `buildExtractionPrompt` and the parse step are fully
  deterministic; the sole non-determinism is `generator.complete`. Tests pin the
  determinism by stubbing the generator and asserting deep-equal structure — the
  milestone's central risk mitigation ("LLM non-determinism leaking into tests").
- **Verbatim consumption.** The extractor renders the `ChangeSet` in its given
  order and does not reorder/dedupe; intra-set ordering was fixed upstream (M2).
- **Errors are typed, not swallowed.** A malformed model response becomes
  `ExtractionError` (from EOS-201); a transport rejection propagates. The extractor
  adds no catch — M5 orchestration decides how a failed extraction maps to the
  `SessionReport` (recoverable vs fatal, SPEC-003 §15).
- **No engine side effects.** The stage reads the `ChangeSet` and calls the port;
  it writes nothing (ADR-002 / AJS-005 §7).
- Suggested layout: `src/end-of-session/extraction/`; tests in
  `tests/end-of-session/knowledge-extractor.test.ts`.

  a rejecting stub → rejection from `extract`. No real generator is invoked.

# Worklog Summary

Delivered the pipeline's single non-deterministic stage as a thin orchestrator
behind an injected port (`src/end-of-session/extraction/`), the direct analog of
`createAnthropicKnowledgeCompiler`:

- **`TextGenerator`** (`TextGenerator.ts`) — the narrow text-generation port
  (`complete(prompt, { maxTokens? }) → AIResponse`), plus the **`KnowledgeExtractor`**
  interface (`extract(changeSet) → Promise<KnowledgeExtraction>`). Defined **locally**
  (not imported from `knowledge/compiler`) per the module's parallel-not-shared
  precedent; it references only the platform `RenderedPrompt`/`AIResponse` *shapes*.
- **`buildExtractionPrompt(changeSet)`** (`prompt.ts`) — a **pure, deterministic**
  function: a fixed system instruction (asking for a session summary + reusable-
  knowledge findings as strict JSON matching the EOS-201 schema, kinds drawn from
  `EXTRACTION_KINDS`), and a user turn rendering each change
  (`id [changeType, kind] path — summary`) in the set's given order. It states the
  `sessionId` and asks the model to echo it verbatim, so the extractor never stamps
  or interprets provenance itself.
- **`createKnowledgeExtractor({ generator })`** (`createKnowledgeExtractor.ts`) —
  wires `buildExtractionPrompt → generator.complete({ maxTokens: 4096 }) →
  parseExtractionResponse`, returning a **frozen** handle. A missing generator
  throws. No retry, fallback, classification, dedupe, scoring, merge, or
  interpretation — orchestration + structural validation only (the Extractor
  Invariant). A malformed response surfaces as `ExtractionError`; a transport
  rejection propagates. No engine side effect; `run` unchanged.

Exported the stage from the module barrel and recorded the two new public
operations (`createKnowledgeExtractor`, `buildExtractionPrompt`) in the EOS-007
drift-guard manifest.

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/knowledge-extractor.test.ts` — **stub-driven, no network** —
covers construction guards + frozen handle, valid extraction + immutability,
sessionId pass-through, prompt hand-off + token budget, empty and partial
`ChangeSet`s, the **Extractor Invariant** (verbatim order, duplicates kept, kinds
untouched save the schema's soft-hint fallback), determinism (deep-equal across runs;
identical prompt containing each change), and error handling (malformed JSON /
schema-violation → `ExtractionError`; transport rejection propagates). End-of-Session
suite: **15 files / 167 tests**; full platform suite green: **507 tests / 47 files**.

**Code review** (high effort, 8 angles): no correctness defects. Two observations,
both deliberately at parity with the ratified precedent and left unchanged: the
local `TextGenerator` port + `EXTRACT_MAX_TOKENS` parallel the compiler
(parallel-not-shared, EOS-005), and the model-echoes-`sessionId` dependency is the
frozen `→ parseExtractionResponse` wiring, which fails safe (omission → typed
`ExtractionError`, never silent bad provenance).

# Lessons Learned

Keeping the stage a thin orchestrator made the Extractor Invariant enforce itself:
because the extractor only builds a prompt, calls the port, and returns
`parseExtractionResponse`, there is nowhere for classification/dedupe/scoring to
creep in — the invariant is a property of the shape, not a rule to remember. The one
genuine design choice was provenance: rather than have the extractor stamp
`sessionId` onto the parsed object (which would be interpretation and would break the
clean `→ parseExtractionResponse` wiring), the prompt supplies the id and asks the
model to echo it, so the stage stays pure-validate and a bad echo fails safe as a
typed error. Stub-driving every test kept the suite fully provider-independent and
let a single fixed canned response pin determinism cheaply (deep-equal across runs).
