# Purpose

Deliver the **Session** stage: turn the input request (`SessionContext`) into the
**identified run** (`Session`) — a stable opaque id plus the metadata every downstream
artifact keys on.

> **Why does this task exist?** `Session` is the provenance anchor of the entire workflow:
> candidates are `session:<id>:<n>`, the review store is `pending/<session-id>/`, and the
> report references `Session.id`. EOS-002 froze the *contract* and deliberately deferred
> *creation* — "Session creation logic / id generation (M5 composition)". Today nothing in
> production constructs a `Session`; only tests and the schema do. This task closes that
> gap: it is the one place identity is minted and git observations become `gitState`.

---

# Objective

Implement, under `src/end-of-session/session/`:

- **`createSessionFactory(deps): SessionFactory`** — with
  `create(context: SessionContext, options): Promise<Session>`.

It assembles exactly the frozen `Session` shape:

| Field | Source |
| --- | --- |
| `id` | Minted here — a stable, opaque identifier (see Open Decisions) |
| `startedAt` / `endedAt` | The injected clock (see Open Decisions) |
| `trigger` | The `TriggerSource.trigger` kind, stamped by composition (EOS-D3) |
| `gitState.head` / `.dirty` | Observed via the injected `GitPort` (EOS-401) |
| `gitState.range` | **Constructed here** from the optional `since` ref |
| `branch` | Observed via the injected `GitPort` (EOS-401); `null` ⇒ synthesized per the [Branch Policy](#branch-policy--detached-head-frozen) |

Built through `parseSession` — the single `Session` validator (EOS-002) — so the result is
validated and deep-frozen.

# Branch Policy — detached HEAD (frozen)

> **Ratified by the reviewer (AJ) on 2026-07-16**, resolving the question EOS-401 raised
> when `GitPort.branch()` became `Promise<string | null>` (EOS-D7 v1.1).

When the `GitPort` reports **no current branch** (`null` — HEAD is detached during a rebase
or bisect, or on a checked-out tag/commit), the Session factory **synthesizes** a descriptive
value:

```text
detached@<short-head>
```

where `<short-head>` is derived from the session's observed HEAD commit.

The three properties this holds together, in the reviewer's terms:

1. **Detached HEAD is not fatal.** The workflow must still capture knowledge from a detached
   session — a rebase is exactly when a session's lessons are worth keeping. Session creation
   proceeds normally.
2. **The `Session` contract is not weakened.** `branch` stays **required and non-empty**. No
   contract change, no `.optional()`, no nullable field.
3. **The policy lives here, and only here.** The port reports git's actual state (`null`); the
   *factory* decides what a session with no branch records. Nullable branch handling therefore
   **stops at this stage** and never propagates into candidates, the projection, the report, or
   SPEC-004.

The synthesized value is honest rather than invented: it states *that* the session was
detached and *where* it was detached, so a reviewer reading provenance sees the real
situation. It is deliberately **not** a real-looking branch name, and is distinguishable from
`git rev-parse --abbrev-ref HEAD`'s literal `"HEAD"` — the value EOS-401 rejected precisely
because it is indistinguishable from a branch genuinely named `HEAD`.

`<short-head>` reuses the `head` the factory already observed (no extra git read).

# Session Factory Invariant (identity and assembly only)

The factory's single responsibility is to **produce one identified run**. It:

- **mints identity, never derives it** — the `id` is opaque and is not a function of
  branch, timestamp, or context (EOS-D3); two runs over an identical `SessionContext`
  yield two distinct ids;
- **assembles metadata, never interprets it** — it records what the ports observe; it does
  not classify the git state, judge whether a session is "worth" capturing, or filter;
- **reads only through injected ports** — the `GitPort` and the clock are injected; the
  factory performs no ambient I/O, reads no config, and shells out to nothing;
- **collects no changes and extracts nothing** — collection is EOS-101, extraction EOS-202;
- **is deterministic given its inputs** — with a pinned clock, a stubbed port, and a
  supplied id source, `create` is a pure function of its inputs. The only intentional
  non-determinism is the id source and the clock, both injected.

# Scope

## Included

- `src/end-of-session/session/` — `SessionFactory.ts` (the interface),
  `createSessionFactory.ts`, `index.ts`.
- **Range construction** from the optional `since` ref (EOS-102 §68–70 deferred it here):
  - `--since <ref>` supplied ⇒ `range = "<ref>..HEAD"`;
  - absent ⇒ `range = "HEAD"` — which makes `git diff --name-status -M HEAD` report the
    **uncommitted + staged** changes, the default session semantics proposed in EOS-002/M2
    and confirmed here.
- Injected dependencies: `gitPort` (EOS-401), `now` (clock), and an id source — each
  defaulting to the real implementation, each overridable in tests (the
  `createCandidateGenerator` / `createKnowledgePipeline` precedent).
- Frozen factory handle; module-barrel export (`createSessionFactory`); EOS-007
  drift-guard manifest updated (+1 operation).
- The **[Branch Policy](#branch-policy--detached-head-frozen)**: `branch ?? detached@<short-head>`,
  reusing the observed `head`.
- Behaviour tests with a **stub `GitPort`** and a pinned clock: field mapping; range
  construction with and without `since`; two `create` calls over one context yield distinct
  ids; **detached HEAD yields `detached@<short-head>` and is not fatal**; the result validates
  and is deep-frozen; a git failure propagates.

## Not Included

- The git observations themselves — **EOS-401** (the factory consumes the port).
- Building the `SessionContext` / invoking the `TriggerSource` — **EOS-407** (composition)
  and **EOS-408** (CLI). The factory receives a context; it does not source one.
- Sequencing the workflow — **EOS-406**.
- Persisting the session — nothing persists a `Session` in v1; provenance travels on the
  candidates and the report.

# Design Notes / Open Decisions (for the Planning Review)

- **Id generator (EOS-D3 left this to M5) — RECOMMENDATION: `randomUUID`.** Node's
  `node:crypto.randomUUID()` is built in, needs no dependency, and satisfies the frozen
  contract exactly ("non-empty, opaque, stable"). A ULID would add sortability and a
  dependency; nothing in v1 sorts sessions. **Recommendation: `randomUUID`, injected via an
  `id: () => string` dep so tests pin it.**
- **`Session.startedAt` semantics — GENUINE OPEN QUESTION, reviewer's call.** A manual
  `aj session end` observes only that the session *ended* now; when it *started* is not
  observable. Options:
  - **(a) `startedAt = endedAt = ` the trigger instant.** Honest ("v1 does not observe
    session start"), zero extra I/O, satisfies the contract. Cost: a zero-length session
    reads oddly to a human.
  - **(b) Derive `startedAt` from the range's base commit date** (one extra git read).
    More meaningful, but adds a git read and a semantic ("the session began at the base
    commit") the spec never claims.
  - **(c) `startedAt` = workflow-run start, `endedAt` = run end.** Rejected on sight: it
    conflates the *session* with the *run* — the run's timing is the `SessionReport`'s job
    (EOS-405), and EOS-D3 makes these distinct concepts.
  - **Recommendation: (a)** for v1, documented explicitly in the factory's docstring, with
    (b) recorded as a future enhancement. It keeps the stage free of speculative git reads
    and keeps the *run* timing where it belongs (the report).
- **Default range = `HEAD` — confirm the session semantics.** `git diff --name-status -M
  HEAD` reports working-tree **and** staged changes against HEAD, matching the
  "uncommitted + staged" default proposed in EOS-002/M2. `--since <ref>` ⇒ `<ref>..HEAD`
  covers a committed range. The reviewer should confirm this pair is the intended v1
  semantics, since it is the first place the workflow *decides* what "the session" means.
- **Where does `trigger` come from?** The factory takes the `TriggerKind` as an input
  (stamped by composition from `TriggerSource.trigger`, per EOS-D3) rather than depending
  on the `TriggerSource` itself — the factory should not be able to *invoke* a trigger.

# Worklog Summary

Delivered the Session stage in a new `src/end-of-session/session/` module (the *stage* that
produces a session; the `Session` *contract* stays in `contracts/session/`):

- **`SessionFactory`** (`SessionFactory.ts`) — the interface plus `SessionFactoryOptions`
  (`trigger`, optional `since`). The trigger arrives as **data**, not as a `TriggerSource`:
  the factory records which trigger fired and must not be able to invoke one (EOS-D3/D9).
- **`createSessionFactory({ gitPort, now?, id? })`** (`createSessionFactory.ts`) — frozen
  handle; missing port throws. `create` mints the id, observes state, constructs the range,
  and builds the `Session` through `parseSession` (the single validator, EOS-002). The three
  git reads run under one `Promise.all` — they are independent, and any rejection is fatal to
  session creation (SPEC-003 §15). `now()` is read **once**, so `startedAt === endedAt` — the
  ratified v1 policy: a manual trigger observes only that the session *ended*.
- **The Branch Policy** — `branch ?? detached@<short-head>`, reusing the already-observed
  `head` (no extra git read). Nullable branch handling stops at this stage.

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/session-factory.test.ts` (20 tests, stub port + pinned clock/id) covers
construction guards, observed state (**including that the observation beats a disagreeing
request**), fatal propagation, range construction (default / `--since` / blank / unresolvable
ref), the **Branch Policy** (synthesis, not fatal, non-empty, short head from the session's
own head), identity (distinct per run; opaque UUID), the `startedAt === endedAt` policy, and
contract conformance (validated, deep-frozen, request untouched, deterministic once pinned).
Public surface **+1** (`createSessionFactory`) — EOS-007 drift-guard manifest updated to
**21** operations. End-of-Session suite **19 files**; full platform suite **568 tests / 51
files**, green.

**Code review** (high effort, 8 angles) — five findings; the two mechanical ones fixed
(an oversized nine-line comment justifying the unused parameter, trimmed to the one constraint
the code cannot show, with the rationale moved to the interface docstring; and two trivially-true
assertions in the identity test that *looked* like they pinned EOS-D3's non-derivation property

## Raised for the reviewer (not fixed here)

1. **`SessionContext`'s request fields reach no consumer in v1 — `--notes` will be silently
   dropped.** A repo-wide grep finds no consumer of `sessionNotes`, `taskId`,
   `contextPackageRef`, or `commitMessage` outside the contract that defines them.
   `buildExtractionPrompt(changeSet)` takes only the `ChangeSet` (frozen in M3), and this
   stage derives no `Session` field from the request — so **EOS-408's planned
   `--notes <text>` flag would collect the engineer's notes and discard them.** That is the
   highest-signal input SPEC-003 §7 lists and the one thing a diff cannot convey. This is
   also the *root cause* of this task's unused `_context` parameter: the parameter is honest;
   the gap is in the plan. It lands on **EOS-406/EOS-408** (and touches M3's frozen extractor
   if notes are ever to reach the model). Flagged rather than absorbed — widening the
   extraction prompt is a Frozen Plan Change Proposal, not a session-factory decision.
2. **The shared git seam still lives under `analyzers/`.** EOS-D7 held that "the seam's
   identity is read-only git access to one repository, not any one consumer's reads", yet the
   file sits at `src/end-of-session/analyzers/git/GitPort.ts`, so this stage imports its git
   access from `../analyzers/git/index.js`. The path now contradicts the decision, and the
   next reader may reasonably read session state reads as trespassing on analyzer-owned code.
   A move to `src/end-of-session/git/` would touch M2-frozen file paths and every importer —
   out of EOS-402's scope, and worth a deliberate decision rather than a drive-by refactor.

# Lessons Learned

Writing the factory surfaced a gap no contract test could: **`SessionContext` is almost
inert in v1**. Its `branch` and `commitHash` are exactly the fields EOS-D7 decided *not* to
trust (a request's claim is not provenance), and its remaining fields reach nothing at all.
The unused parameter is the honest signal — the instinct to "use the context for something"
so the signature looks busy would have quietly reintroduced the very claims-over-facts
problem EOS-401 had just removed. An unused parameter that points at a real gap is better
than a used one that launders bad data.

The Branch Policy landed cleanly because the reviewer separated the two concerns the port had
conflated: *observing* that there is no branch (`null`, EOS-401) and *deciding* what a
branchless session records (`detached@<short-head>`, here). That split is why nullable
handling stops at one stage instead of leaking into candidates, projection, report, and
SPEC-004 — the shape of the fix, not just its content, is what contained the blast radius.

The review's most useful catch was a test that could not fail: asserting a UUID "does not
contain" a branch name or a timestamp reads like it pins EOS-D3's non-derivation property,
but it is true by construction and would pass against an id derived from either. A test that
restates a guarantee it cannot break is worse than no test — it makes the property *look*
covered. The distinct-id-per-run assertion is the one that actually holds the line.
