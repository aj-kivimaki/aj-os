# Purpose

Establish the two remaining extensibility seams: how a session is *triggered* and
how completion is *announced*.

> **Why does this task exist?** SPEC-003 lists many triggers (manual, git hook,
> scheduled, IDE) and a Notification agent. v1 implements only the manual trigger
> and a no-op notifier, but the *seams* must exist now so later triggers/notifiers
> plug in without changing orchestration.

---

# Objective

Define:

- **`TriggerSource`** port — produces a `SessionContext` and its `TriggerKind`.
  Provide the **manual** implementation (constructs a `SessionContext` from
  explicit inputs / CLI args).
- **`NotificationPort`** — announces a completed `SessionReport`. Provide the
  **no-op** implementation.

---

# Scope

## Included

- `TriggerSource` port + `ManualTriggerSource` implementation

- Public exports

## Not Included

# Implementation Notes

- Keep both ports **minimal** — a single method each. The manual trigger builds a
  `SessionContext`; the no-op notifier accepts a `SessionReport` and does nothing
  (returns cleanly). No stubs for future kinds.
- The no-op notifier is a real, useful default (the "explicitly configured
  otherwise" path in AJS-005 §6 is about *knowledge promotion*, not notifications —
  a silent default notifier is fine and keeps v1 free of side channels).
- Do not read config or environment here beyond what the manual trigger needs to
  assemble a `SessionContext`; wiring belongs to M5.
- These ports are the seams that let future triggers/notifiers be **added, not
  woven in** — assert that adding one would not require touching `run()`.

---

# Validation

- `ManualTriggerSource` produces a valid `SessionContext` with `trigger: manual`.
- `NoopNotificationPort` accepts a `SessionReport` and performs no observable
  side effect.
- Ports are small, immutable handles.
