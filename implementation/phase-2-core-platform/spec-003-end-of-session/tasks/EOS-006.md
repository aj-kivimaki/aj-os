# Purpose

Establish the two remaining extensibility seams: how a session is *triggered* and
how completion is *announced*.

> **Why does this task exist?** SPEC-003 lists many triggers (manual, git hook,
> scheduled, IDE) and a Notification agent. v1 implements only the manual trigger
> and a no-op notifier, but the *seams* must exist now so later triggers/notifiers
> plug in without changing orchestration.

---

# Objective

Define:

- **`TriggerSource`** port — produces a `SessionContext` and its `TriggerKind`.
  Provide the **manual** implementation (constructs a `SessionContext` from
  explicit inputs / CLI args).
- **`NotificationPort`** — announces a completed `SessionReport`. Provide the
  **no-op** implementation.

# Implementation Notes

- Keep both ports **minimal** — a single method each. The manual trigger builds a
  `SessionContext`; the no-op notifier accepts a `SessionReport` and does nothing
  (returns cleanly). No stubs for future kinds.
- The no-op notifier is a real, useful default (the "explicitly configured
  otherwise" path in AJS-005 §6 is about *knowledge promotion*, not notifications —
  a silent default notifier is fine and keeps v1 free of side channels).
- Do not read config or environment here beyond what the manual trigger needs to
  assemble a `SessionContext`; wiring belongs to M5.
- These ports are the seams that let future triggers/notifiers be **added, not
  woven in** — assert that adding one would not require touching `run()`.

# Worklog Summary

Established the two remaining extensibility seams as minimal, single-method ports
with their v1 implementations:

- **`TriggerSource`** (`src/end-of-session/trigger/`) — `{ trigger: TriggerKind;
  createContext(): Promise<SessionContext> }`. **`ManualTriggerSource`**
  (`createManualTriggerSource(input)`) advertises `trigger: "manual"` and, on
  `createContext()`, validates the explicit (CLI) input into an immutable
  `SessionContext` via the EOS-002 `parseSessionContext`. Reads no config/env beyond
  its input (wiring is M5).
- **`NotificationPort`** (`src/end-of-session/notification/`) — `{ notify(report:
  SessionReport): Promise<void> }`. **`NoopNotificationPort`**
  (`createNoopNotificationPort()`) accepts the completed report and resolves with no
  side effect — a real, useful default that keeps v1 free of external side channels.

Both ports are frozen handles that own **no** orchestration, policy, retry, routing,
or lifecycle. No future-trigger/notifier stubs were introduced.

Design decisions:

- **Both methods are `async`.** This is the seam-stability choice (the CB-004
  `provide`/`analyze` precedent): a future I/O-backed trigger (git-hook reading
  files) or notifier (Slack/email) plugs in behind the same port without a signature
  change — so adding one never touches `run()`. Each test includes a
  structural-conformance check (an alternate implementation flowing through a
  port-only consumer) proving additivity.
- **Construct-with-input, produce-arg-free.** `createContext()` takes no arguments;
  the manual source captures its input at construction (mirroring how the registry
  is constructed with its analyzers), so different future triggers gather input
  differently at construction while the port method stays uniform.

**Review finding (fixed before commit).** `createContext()` initially wrapped
`parseSessionContext(input)` in `Promise.resolve(...)`, which threw **synchronously**
on invalid input despite the `Promise` return type — a caller that only `.catch()`es
would miss it. Made the method `async` so a `ZodError` surfaces as a rejection.
Caught by the "rejects invalid inputs" test.

**Validation.** `npm run typecheck`, `npm run build` green; full suite
435 tests / 41 files (10 new). Code review clean after the async fix.

---

# Lessons Learned

A method typed to return a `Promise` must not throw synchronously — validation that
can throw belongs inside an `async` body (or an explicit `.then` chain) so failures
reach `.catch`, not a synchronous try/catch the caller never wrote. A single-method
port with its variance pushed to *construction* (input captured when the source is
built, not passed to the produce call) keeps the port signature uniform across
future implementations — the property that lets triggers/notifiers be added behind
the seam without touching orchestration.
