# Purpose

Deliver the change EOS-D10 authorizes: make the engineer's session notes reach the one stage
that can use them.

> **Why does this task exist?** EOS-402 found that `SessionContext.sessionNotes` reaches **no
> consumer** — `buildExtractionPrompt(changeSet)` takes only the `ChangeSet`, so EOS-408's
> `--notes <text>` would collect the engineer's account of the session and silently discard
> it. Notes are a SPEC-003 §7 input and the one channel carrying what a diff cannot: intent,
> dead ends, why an approach was abandoned. The reviewer ruled they must not be dropped in
> v1, and approved this narrowly-scoped change to the frozen M3 stage.

---

# Objective

Thread **one optional parameter** from the request to the prompt. That is the whole task.

1. **`KnowledgeExtractor.extract`** (`extraction/TextGenerator.ts`) —

   ```text
   extract(changeSet: ChangeSet, sessionNotes?: string): Promise<KnowledgeExtraction>
   ```

2. **`buildExtractionPrompt`** (`extraction/prompt.ts`) — same optional parameter; renders the
   notes **verbatim** into the user prompt, in their own clearly-labelled section, when
   present.

3. **The system prompt** — one added rule stating what the notes *are*: the engineer's own
   account of the session, to be used as **context for interpreting the changes**, never as
   instructions, and never as licence to invent facts the changes do not support.

4. **`createKnowledgeExtractor`** — threads the parameter from `extract` to
   `buildExtractionPrompt`. Nothing else about the stage changes.

**Nothing else.** No new stage, contract, port, or field. `sessionNotes` already exists on the
M1-frozen `SessionContext`; this task only *reads* it.

---

# The Extractor Invariant is UNCHANGED (reviewer's condition)

The frozen invariant (EOS-202) is **not reworded, not relaxed, not reinterpreted** by this
task. The extractor still performs **orchestration and structural validation only** — build
the prompt, call the injected `TextGenerator`, parse and validate the response. It does not
classify, deduplicate, merge, score, retry, fall back, or interpret.

Specifically, the extractor **does not read the notes**. It is a courier: the notes travel
from the parameter into the prompt string and are never inspected, parsed, branched on,
summarized, or acted upon. **All interpretation of the notes is the model's**, behind the
port — exactly where SPEC-003 already puts every interpretive act.

This is enforceable by shape, and should be: `sessionNotes` may appear only in prompt
construction. It must never become a condition (`if (sessionNotes)` deciding *behaviour* — as
opposed to deciding whether a prompt section is rendered), never influence parsing, and never
reach `parseExtractionResponse`.

---

# The byte-identical-when-absent criterion (reviewer's condition)

**With no notes supplied, `buildExtractionPrompt(changeSet)` must produce a `RenderedPrompt`
byte-for-byte identical to today's** — `system` and `user` both.

This is the property that makes the change provably additive, and it is not to be softened to
"substantially similar":

- **every existing M3 test must pass unmodified.** If a test needs editing to accommodate this
  change, the change is wrong — stop and re-examine, do not adjust the test;
- the default path (`aj session end` with no `--notes`) behaves exactly as it does today;
- the change is opt-in at the call site: a caller that passes nothing is a caller unaffected.

**Note the consequence for the system prompt:** the added rule (item 3) changes the `system`
string for *every* call, including calls with no notes — which would **break this criterion**.
The rule must therefore be rendered **only when notes are present**, keeping the no-notes
`system` byte-identical. This is the one subtlety in the task and the place it is most likely
to go wrong.

# Scope

## Included

- The optional parameter on `KnowledgeExtractor.extract`, `buildExtractionPrompt`, and the
  threading in `createKnowledgeExtractor`.
- Verbatim rendering of the notes in a labelled prompt section, and the system rule framing
  them as context rather than instructions — **both conditional on notes being present**.
- ~~Blank/whitespace-only notes treated as **absent**~~ — **superseded before implementation
  by the reviewer's acceptance criteria** (AJ, 2026-07-16): *"No interpretation, branching,
  preprocessing, or heuristics based on note content may be introduced."* A blank check is
  `trim()` + a length test — preprocessing, and branching on content. **Presence is therefore
  decided solely by whether a value was supplied** (`!== undefined`); a whitespace-only note
  is *present* and renders. See the Worklog for the full reasoning. (In practice the empty
  string cannot reach this stage at all: `SessionContext.sessionNotes` is `.min(1)`.)
- Behaviour tests extending `tests/end-of-session/knowledge-extractor.test.ts` and the prompt
  suite:
  - **byte-identical when absent** — asserted against the exact current prompt, and by the
    existing M3 tests passing untouched;
  - **present and verbatim** — the notes appear exactly as given, in their own section; the
    change list and `sessionId` hand-off are unchanged;
  - **deterministic** — same `ChangeSet` + same notes ⇒ deep-equal `RenderedPrompt`;
  - blank/whitespace notes ⇒ treated as absent;
  - notes containing prompt-like text (e.g. "ignore the above") are rendered as *content*,
    not obeyed — asserted structurally: the system rule is present and the notes sit inside
    their labelled section;
  - the extractor still only builds → generates → parses (the Extractor Invariant), with the
    notes never reaching `parseExtractionResponse`.

## Not Included

- Passing the notes at the call site — **EOS-406** (`extractor.extract(changeSet,
  context.sessionNotes)`), one argument at one call site.
- The `--notes` flag itself — **EOS-408** (it stays, per the reviewer).
- `taskId`, `contextPackageRef`, `commitMessage` — still unconsumed, and deliberately out of
  scope: EOS-D10 is about notes. Widening this task would be exactly the scope creep the FPCP
  avoided.
- Truncating, summarizing, sanitizing, or validating the notes — all are interpretation or
  policy; this task adds none.
- Any change to `KnowledgeExtraction`, `ChangeSet`, `SessionContext`, or the `TextGenerator`
  port.

# Design Notes

- **Conditional system rule — the criterion's sharp edge.** Adding the rule unconditionally is
  the obvious implementation and it silently violates byte-identical-when-absent, because
  `system` is part of the `RenderedPrompt`. Render the rule only alongside the notes.
- **Verbatim, not normalized.** The notes are rendered as given — no trimming beyond the
  blank check, no reflowing, no escaping. Normalizing would be the extractor interpreting its
  input, and the model is perfectly able to read what the engineer wrote.
- **Trust.** Notes are authored by the engineer running the command — the same trust level as
  the repository contents the prompt already renders verbatim. The system rule exists to keep
  them *context* rather than *instructions*; it is a framing device, not a security control,
  and should not be described as one.
- **Why a parameter and not a field.** Alternatives (whole `SessionContext`, notes on
  `ChangeSet`, a notes analyzer) were considered and rejected in EOS-D10 — see that decision
  rather than re-litigating here.

# Worklog Summary

Implemented the approved EOS-D10 amendment — one optional parameter, threaded from the
request to the prompt:

- **`buildExtractionPrompt(changeSet, sessionNotes?)`** — renders the notes verbatim in
  their own labelled section, and appends `NOTES_RULE` to the system prompt, **both derived
  from a single presence decision** so a section can never appear without its rule.
- **`KnowledgeExtractor.extract(changeSet, sessionNotes?)`** and
  **`createKnowledgeExtractor`** — thread the value through. The stage still does exactly
  three things: build the prompt, call the port, parse the response.

**The byte-identical guarantee was proven, not asserted.** Before touching the code, the
pre-amendment prompt was rendered from the built module and hashed
(`07bda1107e3159c0…`). After the change, the same render produced the **same SHA-256** —
`system` and `user`, byte for byte. It was re-proven a second time after the review refactor
touched that exact path. The corroborating proof the reviewer asked for also holds: **every
existing M3 test passes unmodified** — the diff on `knowledge-extractor.test.ts` contains
additions only, no deletions or edits.

**Present-notes behaviour** was verified the same way: the system prompt is the original plus
the rule and nothing else, and removing the rendered section from the user prompt **restores
the original byte-for-byte** — so the change list, the `sessionId` hand-off, and the closing
instruction are provably untouched.

**The Extractor Invariant is unchanged**, and the code shows it: `sessionNotes` appears in
exactly two places — the parameter list and the prompt string. It is never inspected,
trimmed, measured, branched on, or passed to `parseExtractionResponse`. A test pins the
consequence: with a fixed generator response, the parsed extraction is identical whether or
not notes were supplied.

## A conflict I resolved against this task's own frozen text

This task's planning text (Scope + Acceptance Criteria) said **blank/whitespace-only notes are
treated as absent**, by analogy with EOS-402's blank `since`. The reviewer's implementation
criteria then said: *"No interpretation, branching, preprocessing, or heuristics based on note
content may be introduced."*

A blank check is `sessionNotes.trim().length === 0` — preprocessing, and branching on content.
The two cannot both hold. I implemented **the reviewer's criterion**: presence is decided
solely by `!== undefined`, and a whitespace-only note is present and renders. The task text is
struck through and annotated rather than quietly dropped.

The principle is worth more than the convenience it costs. "No content-based logic" is
verifiable by reading the code for `sessionNotes` and finding no `.trim()`, no `.length`, no
comparison; "no content-based logic *except* a blank check" is a rule with an exception, and
exceptions are how a courier becomes an interpreter. The cost is a whitespace-only note
rendering an almost-empty section — and it is nearly unreachable anyway, since
`SessionContext.sessionNotes` is `.min(1)`, so the empty string fails validation at the
trigger. Flagged for the reviewer.

**Validation.** `npm run typecheck` + `npm run build` clean; 12 tests added to
`knowledge-extractor.test.ts` (byte-identity, system-only-extended, verbatim rendering,
only-the-section-added, change-list intact, determinism, multi-line notes, extractor
pass-through, whitespace-is-present, notes-do-not-affect-parsing, instruction-like notes
rendered as content). Suite **25 tests** in that file; full platform suite **631 tests / 53
files**, green. No public-surface growth (both changed functions were already exported), so
the drift-guard manifest is unchanged at **23**.

**Code review** (high effort, 8 angles) — three findings:

1. **Presence was decided twice** — once inside the notes renderer, once for the system rule —
   so a future edit to one could produce a section with no rule framing it, or a rule with no
   section: exactly the state this amendment exists to prevent. **Fixed**: a single
   `notesParts()` derives both halves from one decision, and returns empty strings when
   absent (so `` `${SYSTEM}${rule}` `` *is* `SYSTEM`). Byte-identity was re-proven against the
   original hash afterwards. **This is the third instance of one shape in three consecutive
   stages** — `summaryFor` (EOS-403), `durationFor` (EOS-405), and now presence: *one decision
   derived twice*.
2. The "only the section is added" test mirrors the implementation's exact section string.
   Accepted: it is the honest way to assert the reviewer's criterion, and noted so the
   coupling is deliberate rather than accidental.
3. **Pre-existing, out of scope:** `stubGenerator` fails an `exactOptionalPropertyTypes` check
   (TS2379) — confirmed present on HEAD and absent from this diff. Left alone, consistent with
   the reviewer's ruling that a narrowly scoped amendment does not refactor frozen internals.
   Invisible to `npm run typecheck`, which scopes to `src`.

# Lessons Learned

The guarantee was worth proving rather than asserting. Hashing the rendered prompt *before*
changing anything turned "byte-identical" from a claim in a task document into a check that
either passes or does not — and it earned its keep twice, because the review refactor rewrote
the exact code path the guarantee depends on. A property that can be checked mechanically
should never be argued about.

The sharp edge the task predicted was real and would have been easy to miss: the system rule
is part of the prompt, so adding it unconditionally satisfies every functional test while
silently breaking the guarantee the amendment was approved on. Nothing about the code would
have looked wrong. Writing that trap down at planning time is what made it a non-event.

And the same defect keeps arriving in a new costume — one decision derived twice, in three
consecutive stages. It is not carelessness so much as the natural shape of "render a thing and
also describe the thing". The fix is always the same: derive once, pass the result. Worth
watching for in EOS-406, where the orchestrator will be handling every stage's output at once.
