# Purpose

Prove the M2 Collection stage works **end to end**: the `GitChangeAnalyzer`,
registered in the `AnalyzerRegistry` and driven by the `collectChanges` execution
stage, produces a correct, deterministic `ChangeSet` from a real repository —
and remains partial in the face of a failing analyzer.

> **Why does this task exist?** EOS-101 (execution) and EOS-102 (analyzer) are
> each unit-tested in isolation with fixtures/stubs. This task supplies the two
> pieces that only make sense together: the **real git-backed `GitPort` adapter**
> (the thing that makes a run real) and the **integration/behaviour tests** that
> validate the whole Collection stage against a real fixture repository, closing
> M2's Definition of Done.

---

# Objective

- Implement the **real git-backed `GitPort` adapter** (`createGitPort`) that
  satisfies the EOS-102 `GitPort` interface by invoking **read-only** git
  commands over a resolved range. It never stages, commits, or mutates the repo.
- Wire `GitChangeAnalyzer` → `AnalyzerRegistry` → `collectChanges` and validate
  the assembled `ChangeSet` against **temp fixture repositories**.
- Validate **determinism** (same repo state ⇒ deep-equal `ChangeSet`) and
  **partial collection** (a failing analyzer alongside the git analyzer yields
  changes **and** an `AnalyzerError`, never an abort).

---

# Scope

## Included

- `createGitPort(...)` — the real read-only git adapter behind the EOS-102
  `GitPort` interface (e.g. `git diff --name-status <range>` and the working-
  tree/staged status, as needed to populate `GitFileChange`s).
- Integration tests that build a temp git repo fixture, register the git analyzer
  (via the real adapter), run `collectChanges`, and assert the resulting
  `ChangeSet`.
- A partial-collection behaviour test: a registry containing the git analyzer
  **and** a throwing fixture analyzer → `ChangeSet` carries the git changes
  **and** one `AnalyzerError`.
- A determinism test: the same fixture repo state produces a deep-equal
  `ChangeSet` across repeated runs.

## Not Included

- New contracts or new pipeline stages (M2 introduces none beyond EOS-101/102).
- Extraction, candidate generation, persistence, projection, CLI (M3–M5).
- Any git **write** (commit/stage/branch) — permanently out of the capture
  pipeline (ADR-002, AJS-005 §7).
- Non-git analyzers (deferred).

## Fixture-repo policy

- Real git is used **only** in these integration/behaviour tests, against
  disposable temp repositories — never in the EOS-101/EOS-102 unit tests, which
  stay on fixture analyzers and the stub `GitPort` (MILESTONES: "no real git in
  unit tests"). Temp repos are created and cleaned up per test; tests must be
  hermetic (no dependence on the host repository state).

# Implementation Notes

- **Read-only git only.** The adapter runs porcelain/plumbing **read** commands
  (`diff --name-status`, `status --porcelain`, `rev-parse`, as needed). It must
  not run any command that mutates the repository or its index. Invoke git via a
  child-process call with an explicit `cwd`; treat non-zero exits as adapter
  failures surfaced to the caller (the execution stage turns an analyzer-level
  failure into an `AnalyzerError`).
- **Range handling.** The adapter reads `session.gitState.range`. The default
  session range semantics (uncommitted+staged vs. `--since <ref>`) are finalized
  where the range string is constructed (M5/CLI); EOS-103 validates the adapter
  correctly reports changes for a **given** range against a fixture repo.
- **Hermetic fixtures.** Build each temp repo with a deterministic sequence of
  git operations inside the test, assert on the resulting `ChangeSet`, and remove
  the temp dir afterward. No test may read or write the host repository.
- **Determinism.** Assert deep equality of the `ChangeSet` across two runs over
  the same fixture state; path-sorted analyzer output (EOS-102) plus registry-
  order aggregation (EOS-101) make this hold.
- **Integration Check (PIPELINE-ARCHITECTURE).** Before proposing the M2 freeze,
  confirm: the new stage has one responsibility; it communicates only through
  immutable contracts; earlier stages are unmodified; the public `run` entry
  point is unchanged (still not introduced — that is M5); no git/wiki side effect
  entered the pipeline.

# Validation

- End-to-end: register `GitChangeAnalyzer` (real adapter) → `collectChanges` over
  a temp fixture repo returns a `ChangeSet` whose `changes` match the fixture's
  git operations (correct `changeType`, ids, order) with no `errors`.
- Partial collection: adding a throwing fixture analyzer yields a `ChangeSet`
  with the git changes **and** exactly one `AnalyzerError`; collection is not
  aborted.
- Determinism: the same fixture state produces a deep-equal `ChangeSet` across
  runs.
- The adapter performs no repository mutation (assert via fixture state / no
  commits or index changes introduced).
- Full platform suite `npm run typecheck`, `npm run build`, `npm test` green.
