# Purpose

Assemble the whole End-of-Session Workflow from configuration in **one place**, so entry
points can run it without knowing how it is built.

> **Why does this task exist?** EOS-406 sequences stages but constructs none of them: it
> takes every collaborator injected. Something must turn an `AjConfig` into the concrete
> pieces — resolve `<handbook>/<reviewPath>`, build the guarded store, wire the git port and
> analyzer registry, hand the `AIClient` to the extractor, mint the projector and notifier —
> and that job is a composition root, not an entry point's. This is the direct analog of
> `createKnowledgePipeline`, which MILESTONES names as the model.

---

# Objective

Implement, under `src/end-of-session/composition/`:

- **`createEndOfSessionWorkflow(config: AjConfig, deps?): Promise<EndOfSessionWorkflowPipeline>`**

It:

1. resolves the destination — `resolve(config.handbook.path, config.handbook.reviewPath)` —
   and ensures it exists (`mkdir -p`), the precondition the guarded store requires (exactly
   `createKnowledgePipeline`'s flow for `generatedWikiPath`);
2. builds the store (`createFilesystemReviewStore({ destination })`) and calls **`locate()`**
   as a fail-fast precondition (EOS-302 names the composition root as its caller) — so a
   mis-set `reviewPath` fails before any git or model work;
3. builds the git port (`createGitPort(repositoryPath)`) and the analyzer registry
   (`createAnalyzerRegistry([createGitChangeAnalyzer(gitPort)])`) — **v1 registers exactly
   one analyzer**;
4. builds the session factory (EOS-402), the extractor over the `AIClient` (or an injected
   `TextGenerator`), the candidate generator, the projector, and the no-op notifier;
5. builds the manual trigger source and resolves its context inputs — `branch` via the
   EOS-D7 git seam, `repository` from `repositoryPath`, `project` from its basename,
   `sessionNotes` from the CLI (EOS-D9);
6. returns the assembled handle — **`{ workflow, store, trigger }`** (EOS-D9).

Async, and for the same reason `createKnowledgePipeline` is: it owns the destination
precondition.

---

# Composition Root Invariant

Its single responsibility is **assembly**. It:

- **constructs; it never sequences** — no stage is invoked here except the `locate()`
  precondition. `run` belongs to EOS-406;
- **is the only module allowed to know every concrete piece at once** — the deliberate
  exception `createKnowledgePipeline` documents ("composition needs every concrete piece.
  The pieces themselves stay reusable and unaware of one another"). Nothing else may import
  across `store/` + `analyzers/` + `extraction/` + `platform/ai`;
- **owns defaults, not policy** — it defaults `generator`, `now`, and `repositoryPath` to
  real implementations and lets tests override each; it makes no decision a stage should
  make;
- **performs no git write and triggers no wiki generation** (ADR-002 §4, AJS-005 §7);
- **stays additive** — registering a second analyzer or swapping the notifier is an edit
  *here* and nowhere else. That is the seams' payoff.

# Scope

## Included

- `src/end-of-session/composition/` — `createEndOfSessionWorkflow.ts`, `index.ts`
  (mirrors `src/knowledge/composition/`).
- `EndOfSessionWorkflowDeps`: `generator?` (TextGenerator, default `AIClient`), `now?`
  (default wall clock), `repositoryPath?` (default `process.cwd()`), `since?` /
  `sessionNotes?` (the CLI's inputs).
- Destination resolution + `mkdir -p` + `locate()` precondition.
- The return shape — **`{ workflow, store, trigger }`** (EOS-D9; additive to the frozen
  plan's `{ workflow, store }`, and PIPELINE-ARCHITECTURE is updated to match).
- Module-barrel export (`createEndOfSessionWorkflow`); EOS-007 drift-guard manifest updated
  (+1 operation).
- Behaviour tests (temp-dir vault + stub generator; no network):
  - composes without a model call and returns a frozen handle exposing `run`;
  - resolves `<handbook>/<reviewPath>`, creates it, and `locate()` reports that root;
  - a `reviewPath` pointing at canonical space (`foundation`/`library`/`wiki`) **fails at
    composition** (the EOS-302 guard, proven at its real call site);
  - an injected stub `TextGenerator` and pinned clock make a composed run deterministic;
  - the registry holds exactly one analyzer in v1.

## Not Included

- Sequencing — **EOS-406**.
- CLI parsing/printing — **EOS-408** (which calls this and nothing else).
- Real end-to-end acceptance — **EOS-409**.
- Reading `aj.config.json` — the caller loads config via `ConfigService` and passes
  `AjConfig` (the `wikiBuildCommand` precedent), so this root stays testable with a literal
  config.
- Any second analyzer, trigger, or notifier — deferred post-v1; the seams make them
  additive.

# Design Notes / Open Decisions (for the Planning Review)

- **Who invokes the `TriggerSource`, and what does the root return? — RATIFIED: the root
  builds and exposes the trigger, returning `{ workflow, store, trigger }`
  ([EOS-D9](../decisions/EOS-D9-trigger-exposed-from-composition-root.md)).** This was the
  milestone's main open question; the reasoning is preserved below and canonically in EOS-D9.
  The frozen plan pulled two ways: `run(context)` takes a
  `SessionContext` (so something upstream must produce one), while *Pipeline Ownership*
  lists `TriggerSource` **inside** the workflow. Something must resolve `project`,
  `repository`, and `branch` — all **required, non-empty** in `SessionContext`.
  - **(a) The root builds the trigger and returns `{ workflow, store, trigger }`
    (recommended).** The CLI stays thin: `const context = await trigger.createContext();
    await workflow.run(context)`. The root resolves `branch` via the EOS-401 git read,
    `repository`/`project` from `repositoryPath`, and folds in the CLI's `--notes`. Honours
    "the workflow owns the trigger" without letting the engine invoke it.
    **Cost: extends the frozen `{ workflow, store }` return — an additive AJS-007 §7.2
    change needing ratification.**
  - **(b) The CLI builds the trigger itself.** No frozen-shape change. Cost: the CLI must
    resolve `branch` — i.e. **call git from the CLI**, breaking the thin-CLI pattern
    (`wikiBuildCommand` touches no git) and duplicating what EOS-401 already provides.
  - **(c) `run` invokes the trigger internally** (`run()` with no argument). Cost: changes
    the **frozen public entry point** `run(context)` — a far heavier change than (a), and it
    would make the workflow untestable with an arbitrary context.
  - **Recommendation: (a).** It is the smallest change that keeps the CLI thin, the entry
    point frozen, and git access behind one seam. If the reviewer prefers to hold
    `{ workflow, store }` exactly, (b) is workable at the cost of git in the CLI —
    **reviewer's call; recorded as EOS-D9 if (a) is accepted.**
- **`project` / `repository` resolution — no config key exists.** `AjConfig` has only
  `handbook`. **Recommendation:** `repository` = the resolved `repositoryPath`; `project` =
  its basename. Both are honest, need no new config, and match the manual trigger's example
  usage. Adding an `AjConfig.project` key would be speculative — defer until something
  actually needs it. *(Alternative: `--project` flag on the CLI. Flagged, not assumed.)*
- **`repositoryPath` default `process.cwd()`.** `aj session end` is run from the repo whose
  session ended, and repository discovery is explicitly not the adapter's job (EOS-103).
  Injectable so EOS-409 points it at a fixture repo. *Confirm.*
- **Does the root return `store`?** Yes — the frozen plan says so, and it mirrors
  `KnowledgePipeline` (which exposes the store for lifecycle management). v1 needs it for
  the `locate()` precondition and for acceptance tests to read the review area. No
  `resetGeneratedWiki` analog: the review area is an append-only capture record and the
  workflow owns no destructive lifecycle operation.
- **Naming.** `createEndOfSessionWorkflow` is fixed by the frozen plan; the returned type
  name (`EndOfSessionWorkflowPipeline`? `EndOfSessionPipeline`?) is not — reviewer's call.
  Note the mild collision: the *factory* is named for the workflow but returns a bundle
  containing it.

# Worklog Summary

Delivered the composition root in `src/end-of-session/composition/` — the analog of
`createKnowledgePipeline`, and the only module allowed to know every concrete piece at once:

- **`createEndOfSessionWorkflow(config, deps?)`** → **`{ workflow, store, trigger }`**
  (EOS-D9), frozen. Resolves `<handbook>/<reviewPath>`, ensures it exists (`mkdir -p`),
  builds the guarded store and calls **`locate()`** to fail fast, then constructs the git
  port, the single-analyzer registry, the session factory, the extractor over the platform
  `AIClient`, the candidate generator, the projector, and the no-op notifier — and hands them
  to `createSessionWorkflow`.
- **`deps`**: `generator` (default `AIClient`), `now` (default wall clock), `repositoryPath`
  (default `process.cwd()`), `since`, `sessionNotes` — each defaulted to the real thing and
  each overridable, so the whole workflow composes in a test without a network call.

**It constructs; it does not orchestrate.** Verified by grep: the only stage invocation in
the module is the `locate()` precondition the frozen invariant explicitly carves out, and the
only filesystem work is `mkdir -p` — the precondition the guarded store *requires* in order
to exist, and exactly what `createKnowledgePipeline` documents as "the composition root owns
that precondition". No sequencing, no policy, no stage logic, no path beneath the destination
(the store owns the layout — EOS-D6/EOS-D8).

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/composition-root.test.ts` (14 tests) over **real temp vaults and real
disposable git repos** — the adapters this task actually constructs — with the generator
always stubbed. Covers the exposed handles, composing without a model call, the trigger
producing a context whose branch/project/repository the *root* resolved (so the CLI needs no
git), destination resolution and `reviewPath` honouring, creating **only** the destination
(no `pending/` — that is the store's), canonical areas untouched, the **fail-fast refusal**
of a canonical `reviewPath` at its real call site, an unusable repository surfacing, the v1
single-analyzer wiring, the injected generator reaching the extraction seam, and the report
landing under the resolved destination. Public surface **+1**; drift-guard **25**. Full
platform suite **665 tests / 55 files**, green.

**The stub-drift check earned its keep immediately.** The new suite imported `type
AIResponse` from the End-of-Session barrel — which does not export it. Vitest strips types
and `npm run typecheck` scopes to `src`, so **the suite passed and the project's own
typecheck passed** while the import was invalid. Only the explicit `tsc` run caught it
(TS2305). This is the third distinct defect that tooling gap has hidden (EOS-401 stub
breakage, EOS-405 `exactOptionalPropertyTypes`, and now an invalid import).

**Code review** (high effort, 8 angles) — five findings; one fixed, four recorded:

1. **Fixed:** `sessionRequest` returned `Record<string, unknown>`, discarding the shape it had
   just built — a mistyped key (`sessionNote`) would have compiled and surfaced only as a
   runtime `ZodError`. Given a named `SessionRequest` type.
2. **Raised to EOS-408 (recorded in that task):** a **missing API key never reaches the CLI's
   friendly-error path**. `new AIClient()` does not throw (the key resolves lazily), so
   composition succeeds; the `AIError` then fires *inside extraction*, where EOS-406 converts
   it into a **persisted `failed` report**. So the most common first-run mistake writes a
   session directory into the vault recording a configuration problem, and EOS-408's planned
   `AIError` catch is dead for it. Options and a recommendation are now in EOS-408's Design
   Notes — it owns presentation, and this is a real behaviour change to decide before the CLI
   is written.
3. **Accepted, flagged:** `mkdir -p` necessarily runs *before* the store's guard (the guard
   requires the destination to exist), so a mis-set `reviewPath: "wiki"` creates an empty
   canonical-named directory and *then* refuses. Pre-checking here would duplicate the
   store's guard — policy in the root, which this task forbids.
4. **Accepted, flagged:** `NO_BRANCH = "detached"` is a *second* representation of a detached
   HEAD, distinct from the factory's `detached@<short-head>` — see below.
5. **Accepted:** `branch()` is observed twice per run (root for the request, factory for the
   Session). Inherent to EOS-D9 + EOS-402: the root must resolve the request's branch, and
   the Session's branch must be *observed* rather than passed, because a request's claim is
   not provenance.

## The `SessionContext.branch` problem (for the reviewer)

`SessionContext.branch` is **required and non-empty**, and `GitPort.branch()` returns `null`
when HEAD is detached — so the root must supply *something*. Every option is unattractive:

- **reproduce `detached@<short-head>`** → duplicates the frozen Branch Policy, whose whole
  point is that it "lives here, and only here" (EOS-402), and needs an extra `head()` read;
- **export the policy from the session module** → spreads it across two modules for a field
  EOS-402 proved **nothing reads**;
- **make `SessionContext.branch` optional** → an FPCP against an M1-frozen contract;
- **state plainly that there was no branch** (chosen) → `"detached"`, at the cost of two
  strings describing one fact.

Chosen the last: the request records what git said; the *authoritative* branch is
`Session.branch`, which carries the policy and is what provenance, the projection, and
SPEC-004 actually use. Recorded rather than decided quietly, because it is the first place
the request and the session disagree.

# Lessons Learned

The invariant was easy to honour and hard to verify by intuition, which is why the greps
mattered: *what does this module invoke, and what does it touch?* One `locate()` and one
`mkdir -p`, both construction preconditions with an explicit precedent. Anything else would
have shown up in two lines of output. The reviewer's phrase — "no filesystem behavior beyond
constructing the approved adapters" — turned out to be exactly the right test, because the
guarded store *cannot exist* without its destination, so creating it is construction rather
than behaviour.

The composition root is where the design's unstated assumptions finally collide. EOS-D9 said
the root resolves the request's branch; EOS-402 later made branch nullable and put the
detached policy in the factory; EOS-402 also proved the request is never read. Individually
each is right. Together they force the root to invent a value for a required field that has no
consumer — a seam that only appears when the last piece is wired in. That is an argument for
building the vertical slice early, not last.

And the tooling gap has now hidden three different defects across the milestone. Every one was
invisible to `npm run typecheck` and to a green suite, and every one took the same 15-second
explicit `tsc` run to surface. That is a strong argument for widening the compiler's scope to
`tests/` once SPEC-003 closes.
