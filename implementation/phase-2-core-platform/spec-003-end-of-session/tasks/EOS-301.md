# Purpose

Deliver the **Candidate Generation** stage — the deterministic transform that maps a
validated `KnowledgeExtraction` into the workflow's **canonical output**,
`CandidateKnowledge[]` (the SPEC-003 → SPEC-004 boundary; EOS-D1/D4). Each of the
extraction's `findings` becomes one immutable, provenance-complete candidate in the
AJS-006 `candidate` governance state. This is where the model's advisory findings
acquire authoritative identity, provenance, and kind.

> **Why does this task exist?** M3 produced a `KnowledgeExtraction` whose findings
> carry a *soft* kind hint, verbatim source linkage, and no identity or provenance —
> the extractor deliberately "does not classify, deduplicate, merge, score, or
> enrich" (the frozen Extractor Invariant). This task supplies exactly the
> complementary responsibility the invariant deferred to M4: authoritative kind
> assignment, stable id assignment, and provenance attachment — as a **pure,
> deterministic** function, so the whole pipeline from `ChangeSet` onward is
> deterministic once the one LLM seam (M3) is stubbed.

---

# Objective

Implement a deterministic Candidate Generator:

- **`CandidateGenerator`** — a behavioral interface with
  `generate(extraction: KnowledgeExtraction, session: Session): CandidateKnowledge[]`.
- **`createCandidateGenerator({ now }): CandidateGenerator`** — a factory taking an
  **injected clock** (`now: () => Date`, the Context Builder / `createKnowledgePipeline`
  precedent) so `provenance.generatedAt` is deterministic in tests. Returns a frozen
  handle (the module's factory convention).

For each finding (in the extraction's given order), the generator builds a
`CandidateKnowledge` and validates it through the existing
`parseCandidateKnowledge()` (EOS-003), which enforces the schema and deep-freezes.
The mapping is:

| Candidate field | Source | Notes |
| --- | --- | --- |
| `id` | `session:<session.id>:<n>` | `n` = 1-based position of the finding in `extraction.findings`; deterministic and stable within the session (the scheme documented in the EOS-003 contract). |
| `kind` | `finding.kind` | **Authoritative** assignment: the finding's soft hint is resolved through `candidateKindSchema` into the closed `CandidateKind` (unknown ⇒ `handbook-entry` fallback, already in the contract). v1 does **not** re-derive kind from content/paths. |
| `title` / `body` / `rationale` | finding, verbatim | Carried unchanged (the model's content). |
| `tags` | `finding.tags`, verbatim | May be empty. |
| `confidence` | `finding.confidence`, verbatim | Optional; carried, never derived (parity with the Extractor Invariant). |
| `related` | `[]` in v1 | Inter-candidate relationships are SPEC-004 grouping; not inferred here (no speculative relationship derivation). |
| `governanceState` | `"candidate"` literal | SPEC-003 only ever proposes (AJS-006 §3). |
| `provenance.sessionId` | `session.id` | See the session-consistency guard below. |
| `provenance.sourceChangeIds` | `finding.relatedChangeIds`, verbatim | Advisory linkage from M3; may be empty. |
| `provenance.sourcePaths` | `finding.relatedPaths`, verbatim | May be empty. |
| `provenance.commitHash` | `session.gitState.head` | The commit the session ended on; optional in the contract. |
| `provenance.generatedAt` | `now()` (ISO-8601) | The one injected value; deterministic in tests. |
| `provenance.generator` | a module constant | A stable producer id (e.g. `"end-of-session/candidate-generator"`); records *what* generated the candidate for audit. |

An **empty `findings` array yields an empty candidate array** — a valid outcome (a
session can produce no reusable knowledge). The extraction's `summary` is **not** a
candidate: it is session-level output that flows to the `SessionReport` / `ReviewPackage`
in M5, not a unit of proposed knowledge here.

# Candidate Generation Invariant (deterministic one-to-one structural mapping)

**Candidate generation is a deterministic one-to-one structural mapping.** Each
`KnowledgeExtraction` finding produces **exactly one** `CandidateKnowledge`, and every
candidate comes from exactly one finding. The generator **must not merge, split,
reorder, invent, or remove findings**: the output array has the same length as
`extraction.findings`, in the same order, with candidate `n` derived solely from
finding `n`. Consequently `candidates.length === extraction.findings.length` always
(and an empty `findings` array yields an empty candidate array). This 1:1, order-
preserving property is what makes the `session:<id>:<n>` identity scheme stable and the
stage fully deterministic — and it is asserted directly in this task's tests.

Beyond that mapping, the generator performs **deterministic structural mapping and
provenance attachment only**. Given the same `(extraction, session)` and a fixed clock
it returns a **deep-equal** `CandidateKnowledge[]`. It **does not**:

- **call the model / any port** — there is no non-determinism here; the LLM seam was M3;
- **deduplicate or compare against the Handbook** — that is SPEC-004's responsibility
  (and needs the `IdentityResolver`); the generator never reads canonical knowledge;
- **merge, split, reorder, invent, or remove findings** — one finding maps to exactly
  one candidate, in order (the one-to-one property above);
- **score / rank** — a model `confidence` is carried verbatim, never computed or used to sort;
- **re-interpret content** — it does not rewrite `title`/`body`/`rationale`, infer new
  tags, or derive kind from anything other than the finding's declared hint;
- **persist, project, approve, publish, or commit** — persistence is EOS-302, projection
  and orchestration are M5, approval/publication are SPEC-004.

The only fields the generator *introduces* are identity (`id`), the governance literal,
and provenance (`sessionId`, `sourceChangeIds`, `sourcePaths`, `commitHash`,
`generatedAt`, `generator`) — everything else is the model's content, unchanged.

# Scope

## Included

- `src/end-of-session/generation/` — `CandidateGenerator.ts` (interface),
  `createCandidateGenerator.ts` (factory + mapping), `index.ts` (mirrors the
  `analyzers/git/` and `extraction/` layout).
- The finding→candidate mapping table above, each candidate validated + frozen via
  `parseCandidateKnowledge()` (reuse the EOS-003 parser — no second validator).
- A **session-consistency guard**: `generate` requires `extraction.sessionId === session.id`;
  a mismatch is a programming/orchestration error and throws (fail-fast, never silent
  bad provenance) — parity with the "fail safe" reasoning behind the M3 sessionId echo.
- Injected clock; frozen factory handle; public exports (`createCandidateGenerator`,
  `CandidateGenerator`) from the module barrel; EOS-007 drift-guard manifest updated.
- **Behaviour & determinism tests** (pure, no filesystem, no network):
  - a fixed extraction + session + fixed clock ⇒ the expected `CandidateKnowledge[]`;
  - **one-to-one mapping**: `candidates.length === findings.length`, order-preserving,
    candidate `n` derived from finding `n` (no merge/split/reorder/invent/remove);
  - repeated runs ⇒ **deep-equal** output (determinism);
  - id scheme `session:<id>:<n>` assigned in finding order; ids stable and unique;
  - provenance is complete and correct (source linkage carried verbatim; `commitHash`
    from `gitState.head`; `generatedAt` from the injected clock; `generator` set);
  - kind: a recognized hint maps 1:1; an out-of-domain hint (via a lenient finding)
    resolves to `handbook-entry`; `confidence`/`tags` carried verbatim; `related` empty;
  - empty `findings` ⇒ empty array;
  - `governanceState` is always `"candidate"`; every candidate is deep-frozen;
  - a `sessionId` mismatch throws.

# Design Notes / Open Decisions (for the Planning Review)

- **Kind classification is a validated pass-through in v1 (recommended).** `EXTRACTION_KINDS`
  and `CANDIDATE_KINDS` are the same six values by construction, and the candidate schema
  already resolves unknowns to `handbook-entry`. So "M4 owns the authoritative mapping"
  reduces to *resolving the soft hint through the closed contract* — deterministic and
  simple. Content/path-based re-classification is deliberately **out of scope** (it would
  be interpretation and non-obvious to keep deterministic); a future analyzer can add it
  behind this same seam without changing the boundary. **Flagged for the reviewer** to
  confirm pass-through is the intended v1 behavior.
- **`related` is empty in v1 (recommended).** Deterministic inter-candidate relationships
  are not derivable without semantic comparison, which is SPEC-004's job. Leaving `related`
  empty keeps the generator honest (no speculative inference); the field exists in the
  contract for SPEC-004 to populate during grouping.
- **`generator` id is a module constant.** A stable string identifying the producer for
  audit/provenance (not a version-coupled value that would churn tests). Exact value to be
  fixed in implementation.
- **No `EOS-D#` decision is proposed** for this task: the mapping follows the frozen
  contract and the injected-clock precedent, with no genuine architectural choice beyond
  the pass-through confirmation above. The reviewer decides whether any of the flagged
  points warrants a recorded decision (the M3 precedent recorded none).

# Worklog Summary

Delivered the Candidate Generation stage as a pure, deterministic transform in a new
`src/end-of-session/generation/` module (mirroring the `extraction/` layout):

- **`CandidateGenerator`** (`CandidateGenerator.ts`) — the stage interface,
  `generate(extraction, session): CandidateKnowledge[]`.
- **`createCandidateGenerator({ now? })`** (`createCandidateGenerator.ts`) — the factory
  over an injected clock (defaults to the wall clock, the `createKnowledgePipeline`
  precedent), returning a **frozen** handle. `generate` fails fast when
  `extraction.sessionId !== session.id` (a mismatch would silently corrupt provenance —
  an orchestration bug, so it throws), stamps **one** `generatedAt` for the whole run, and
  maps each finding to a candidate via a private `toCandidate` that builds the object and
  validates + deep-freezes it through the existing **`parseCandidateKnowledge`** (EOS-003 —
  no second validator). Mapping: id `session:<id>:<n>` (1-based, finding order); `kind` =
  the finding's hint resolved authoritatively through `candidateKindSchema`;
  `title`/`body`/`rationale`/`tags`/`confidence` verbatim (`confidence` omitted, not sent as
  `undefined`, when absent); `related` empty; provenance from the finding's advisory
  linkage + `session.gitState.head` + the clock + a constant `generator` id;
  `governanceState` the `candidate` literal.

The **one-to-one, order-preserving** property is a property of the shape — a single
`findings.map` with no branch that could merge/split/drop — so the Candidate Generation
Invariant enforces itself. The returned array is `Object.freeze`d (code-review finding)
so the canonical output is immutable as a whole, matching the module's deep-freeze
convention.

Exported `createCandidateGenerator` (+ the `CandidateGenerator`/`CandidateGeneratorConfig`
types) from the module barrel and added `createCandidateGenerator` to the EOS-007
drift-guard manifest.

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/candidate-generator.test.ts` (13 tests, no stub/clock injected for
determinism) covers the one-to-one mapping + ordering + ids, verbatim content, complete
provenance, `confidence` omission + empty-list defaults, one shared `generatedAt`, empty
findings ⇒ empty array, determinism (deep-equal across runs), deep-freeze of every
candidate **and** the returned array, the frozen handle, the wall-clock default, and the
`sessionId`-mismatch guard. End-of-Session suite: **16 files / 178 tests**; full platform
suite green: **518 tests / 48 files**.

**Code review** (high effort, 8 angles): no correctness defects. One low-severity
consistency finding — the returned array was frozen per element but not as a whole —
addressed by `Object.freeze`ing the array (+ a test), bringing it in line with the
module's uniform output-immutability posture.

# Lessons Learned

Keeping the generator a pure `findings.map` made the one-to-one invariant structural
rather than a rule to police: there is no code path that could merge, split, or drop a
finding, so the property holds by construction and a single length+order test pins it.
The only real design choices were fail-fast on the `sessionId` mismatch (silent bad
provenance is worse than a throw, and M5 always wires one session through both stages) and
stamping a single `generatedAt` per run (so a run's candidates share provenance and stay
deep-equal under a fixed clock). Reusing `parseCandidateKnowledge` as the sole
construction path meant the mapping inherited validation + deep-freeze for free and cannot
drift from the boundary contract.
