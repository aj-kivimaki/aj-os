# Purpose

Define the human-readable projection and the execution log.

> **Why does this task exist?** SPEC-003 emits a review package and records logs.
> Per EOS-D4 the review package is a *projection* of the canonical
> `CandidateKnowledge[]`, not a primary artifact; and per SPEC-003 ¬ß16 the workflow
> records an execution log. This task fixes both shapes so M4/M5 can produce them.

---

# Objective

Define two immutable contracts:

- **`ReviewPackage`** ‚Äî a human-readable projection derived from
  `CandidateKnowledge[]` + `Session`: `{ sessionId, generatedAt, summary,
  candidateIds[], markdown }`. Explicitly typed and documented as a *projection*
  (non-canonical, regenerable).
- **`SessionReport`** ‚Äî the structured execution log (SPEC-003 ¬ß16):
  `{ sessionId, trigger, startedAt, endedAt, durationMs, analyzersRun[],
  filesAnalyzed, candidatesProduced (count + ids), errors[], result:
  completed|partial|failed, logEntry }`.

# Implementation Notes

- `ReviewPackage.markdown` is opaque rendered text; the package holds
  `candidateIds` (references), not embedded canonical candidate records ‚Äî the
  candidates live in the store as the source of truth (EOS-D4).
- `result` is a closed enum; `errors` reuse a shared error shape (align with the
  `AnalyzerError` defined in EOS-005 where practical).
- Keep both contracts free of a clock ‚Äî `generatedAt`/timestamps are injected at
  composition (CB `now` precedent), not read inside the contract.
- Do not compute anything here; these are shape-only contracts.

---

# Validation

- Valid packages/reports parse and deep-freeze; invalid `result` values and
  missing required fields are rejected; a `ReviewPackage` with `candidateIds` not
  matching provided candidates is out of scope for the contract (enforced at
  projection time, M5) ‚Äî the contract validates shape only.

# Worklog Summary

Defined the two workflow-output contracts, following the module's contract pattern
(Zod `.strict()` ‚Üí `deepFreeze` ‚Üí inferred `DeepReadonly`, public export via
`contracts/`, public-surface tests):

- **`ReviewPackage`** (`contracts/review-package/`) ‚Äî the human-readable
  **projection** (EOS-D4): `{ sessionId, generatedAt, summary, candidateIds[],
  markdown }`. Documented and typed as non-canonical and regenerable. It holds
  `candidateIds` (**references** into the review store), never embedded candidate
  records, so canonical data is not duplicated; `markdown` is opaque rendered text
  (the projector is M5).
- **`SessionReport`** (`contracts/session-report/`) ‚Äî the structured **execution
  log** (SPEC-003 ¬ß16): `{ sessionId, trigger, startedAt, endedAt, durationMs,
  analyzersRun[], filesAnalyzed, candidatesProduced {count, ids[]}, errors[],
  result, logEntry }`. Describes execution results/diagnostics, never workflow
  decisions.

Design decisions:

- **`result`** is a **closed** enum (`completed|partial|failed`) ‚Äî producer-
  controlled and exact, unlike the lenient trigger/kind enums.
- **`trigger`** **reuses** the EOS-002 `triggerKindSchema` (intra-module DRY) rather
  than forking a stricter variant.
- **`errors[]`** uses a **report-owned** `{ source, message, recoverable }` shape,
  structurally aligned with the future EOS-005 `AnalyzerError` but owned here: the
  log aggregates errors from any stage (not only analyzers), so EOS-004 does not
  depend on an unimplemented contract or invert the dependency. `recoverable` maps
  to the SPEC-003 ¬ß15 recoverable/fatal split.
- **Shape-only, no cross-checks** ‚Äî `count`/`durationMs` are denormalized ¬ß16
  observability fields kept consistent by the producer (M4/M5); the task's Validation
  note explicitly places cross-field consistency outside the contract. No clock is
  embedded ‚Äî timestamps are injected at composition (the Context Builder `now`
  precedent).

**Validation.** `npm run typecheck`, `npm run build` green; full suite
399 tests / 37 files (21 new). Reviewed against all 8 code-review angles ‚Äî no
correctness defects; three non-blocking observations (denormalized ¬ß16 fields,
reused lenient trigger, a trivial `stringListSchema` idiom repeated across contract
files) recorded and intentionally left as-is.

# Lessons Learned

An observability/log contract legitimately **denormalizes** derivable fields
(`count`, `durationMs`) for auditability; the discipline is to keep such contracts
*shape-only* (no `superRefine` cross-checks) and let the producer own consistency ‚Äî
the same rule the task applies to the projection's `candidateIds`. Owning a small
error shape at the boundary that *needs* it (the report), rather than reaching for a
not-yet-defined sibling contract (`AnalyzerError`, EOS-005), keeps the dependency
direction correct and avoids coupling to unimplemented milestones.
