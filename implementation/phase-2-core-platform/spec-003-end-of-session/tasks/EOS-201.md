# Purpose

Define the **immutable contract the model produces** — `KnowledgeExtraction` — and
the validating parser that turns a raw model response into a deep-frozen instance
of it. This is the *shape* of extraction, not the *act* of extracting.

> **Why does this task exist?** M3 introduces the pipeline's single
> non-deterministic seam (knowledge extraction via a text-generation model).
> Before wiring that seam (EOS-202), the workflow needs the immutable vocabulary
> the seam emits and a parser that makes the model's non-deterministic *content*
> safe by validating its *structure*. Splitting the contract from the extractor
> keeps the boundary (what a valid extraction is) reviewable independently of the
> behavior (how the model produces one), exactly as M1 split the analyzer output
> contracts (EOS-005) from the analyzer behavior (M2). This directly reuses the
> `src/knowledge/compiler/extraction.ts` pattern the milestone plan names.

---

# Objective

Implement, as a self-contained contract module:

- **`knowledgeExtractionSchema`** — a Zod schema describing the model's structured
  findings: a session `summary` plus a list of reusable-knowledge `findings`, each
  carrying enough for M4 to build a `CandidateKnowledge` deterministically (kind
  hint, title, body, rationale, source linkage, tags, optional confidence).
- **`KnowledgeExtraction`** — the `DeepReadonly` type inferred from the schema, so
  the runtime and compile-time contracts never drift (module convention).
- **`ExtractionError`** — a domain error raised when a model response cannot be
  parsed or validated (the `CompilerError` precedent).
- **`parseExtractionResponse(raw: string): KnowledgeExtraction`** — strip a
  ` ```json … ``` ` fence, `JSON.parse`, `safeParse` against the schema, and
  **deep-freeze** the result; on any failure throw `ExtractionError` with a
  human-readable message (never a stack trace).

The contract is **internal to the pipeline** (PIPELINE-ARCHITECTURE: `ChangeSet`,
`KnowledgeExtraction`, and `SessionContext` are internal contracts) — not the
SPEC-003 → SPEC-004 boundary. It is nonetheless immutable and provenance-carrying,
consistent with every other End-of-Session contract.

# Scope

## Included

- `src/end-of-session/contracts/knowledge-extraction/` — `schema.ts`, `types.ts`,
  `index.ts` (mirrors the `candidate/`, `change/`, `session/` contract layout).
- `knowledgeExtractionSchema` (`.strict()` throughout) with:
  - `sessionId` — non-empty; carries traceability from the extraction back to the
    `ChangeSet`/`Session` (the `ChangeSet.sessionId` precedent).
  - `summary` — `{ title: string.min(1), keyPoints: string[].min(1) }`, the
    SPEC-003 §8 "Session Summary" primary output.
  - `findings` — an array (**may be empty**: a session can yield no reusable
    knowledge) of `{ kind, title, body, rationale, relatedChangeIds, relatedPaths,
    tags, confidence? }`.
- `kind` as a **soft-hint** enum over the SPEC-003 §8 taxonomy values with a
  `.catch` fallback (the `changeKindSchema`/`entityTypeSchema` precedent): the
  model's classification is advisory; **M4 owns the authoritative mapping** to the
  closed `CandidateKind`. Downstream must not depend on a precise extraction kind.
- `parseExtractionResponse(raw: string)` — fence-strip + `JSON.parse` + `safeParse`
  + `deepFreeze`; `ExtractionError` on invalid JSON or schema mismatch.
- `ExtractionError extends Error` (`name = "ExtractionError"`).
- Public re-export from `contracts/index.ts` → `src/end-of-session/index.ts`;
  record the new public operations in the EOS-007 drift-guard manifest.
- Contract tests through the **public surface only**: valid response → frozen
  extraction; fenced/unfenced JSON; empty `findings`; unknown `kind` → fallback;
  missing/empty required fields, malformed JSON, and unknown keys → `ExtractionError`;
  deep immutability.

## Not Included

- The extractor stage, the prompt, or any `TextGenerator` call — that is EOS-202.
- Any real LLM invocation or network access.
- Candidate generation / the `KnowledgeExtraction → CandidateKnowledge[]` mapping —
  that is M4 (EOS-3xx). This task defines the *input* to that mapping only.
- Session-notes / commit-message enrichment of the extraction input — the extractor
  consumes the `ChangeSet` (M3's stated input); richer session context is deferred.

# Design Notes

- **Two proven idioms, reconciled.** The compiler folds fence-strip + validation
  into one `parseExtraction(raw: string)`; the End-of-Session module's convention
  is `parseX(input: unknown)` → **deep-frozen**. This task keeps the compiler's
  single raw-string entry point (the model response is the only producer of an
  extraction) **and** the module's deep-freeze, i.e.
  `parseExtractionResponse(raw: string): KnowledgeExtraction` that fence-strips,
  `JSON.parse`s, `safeParse`s, and `deepFreeze`s. A separate `parse(unknown)`
  variant is **intentionally not added** — there is no second entry point that
  hydrates an extraction (M4 consumes the in-memory instance), so it would be a
  speculative abstraction. _If the Planning Review prefers strict layering — the
  contract offering only `parse(unknown)` and the fence-strip living in EOS-202 —
  that is the primary alternative; recorded here for the review to ratify._
- **`kind` is a soft hint, not the candidate taxonomy.** Use `.catch` so an
  unrecognized/absent model kind never fails the whole extraction; the closed
  `CandidateKind` classification is M4's authoritative responsibility. This keeps
  the two taxonomies from coupling prematurely.
- **Source linkage for M4 provenance.** `findings[].relatedChangeIds` and
  `relatedPaths` are `string[].default([])`, giving M4 the `sourceChangeIds` /
  `sourcePaths` it needs for `CandidateProvenance` without M4 re-deriving them.
  They are advisory and may be empty.
- **`.strict()` everywhere** so a hallucinated extra key is rejected rather than
  silently carried, and `deepFreeze` so no stage can mutate a shared extraction.
- **Determinism of structure.** The schema is deterministic; only the *content*
  the model supplies is not. The parser never sorts, dedupes, or fills — it
  validates and freezes verbatim (ordering of `findings` is the model's, preserved).
- Suggested layout: `contracts/knowledge-extraction/{schema.ts,types.ts,index.ts}`;
  tests in `tests/end-of-session/knowledge-extraction-contract.test.ts`.

# Validation

- A well-formed JSON response (fenced and unfenced) parses to a deep-frozen
  `KnowledgeExtraction` with the expected fields and `findings` order preserved.
- An empty `findings` array is valid; a `summary` with at least one `keyPoint` is
  required.
- An unknown `kind` string falls back via `.catch`; a hallucinated extra key,
  missing/empty required field, or non-JSON body throws `ExtractionError` (with a
  readable message, never a stack trace).
- The returned value is deeply immutable (mutation attempts throw / are no-ops
  under the module's deep-freeze inspector).
- `npm run typecheck`, `npm run build`, `npm test` green.

# Worklog Summary

Delivered the immutable `KnowledgeExtraction` contract and its raw-response parser
as a self-contained contract module
(`src/end-of-session/contracts/knowledge-extraction/`), reusing the compiler
`extraction.ts` pattern under the End-of-Session module's deep-freeze convention:

- **`knowledgeExtractionSchema`** (`schema.ts`) — `.strict()` throughout:
  `sessionId` (traceability to the `ChangeSet`), a required `summary`
  (`title` + ≥1 `keyPoints` — the SPEC-003 §8 Session Summary), and `findings`
  (may be empty). Each finding carries a soft-hint `kind`, `title`/`body`/
  `rationale`, source linkage (`relatedChangeIds`, `relatedPaths`), `tags`
  (all `string[].default([])`), and an optional `confidence` (`[0,1]`) carried
  verbatim.
- **`extractionKindSchema`** — `z.enum(EXTRACTION_KINDS).catch("handbook-entry")`,
  a **decoupled** soft hint kept local to the extraction contract (not imported
  from `CANDIDATE_KINDS`) so the internal vocabulary and the cross-spec boundary do
  not couple; M4 owns the authoritative mapping to `CandidateKind`.
- **`parseExtractionResponse(raw: string)`** — single raw-string entry point:
  strip a ` ```json … ``` ` fence → `JSON.parse` → `safeParse` → `deepFreeze`;
  throws **`ExtractionError`** (readable message, never a stack trace) on invalid
  JSON or schema mismatch. No `parse(unknown)` variant (the model response is the
  only producer — a speculative second entry point was avoided).
- **Types** (`types.ts`) inferred + `DeepReadonly`-wrapped
  (`KnowledgeExtraction`, `KnowledgeFinding`, `ExtractionSummary`, `ExtractionKind`)
  so runtime and compile-time contracts cannot drift.

Re-exported the contract from `contracts/index.js` → `src/end-of-session/index.js`,
and recorded the two new public operations (`parseExtractionResponse`,
`ExtractionError`) in the EOS-007 drift-guard manifest — the manifest's comment was
extended to note that an exported error class (a function-typed constructor)
surfaces there and is pinned deliberately.

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/knowledge-extraction-contract.test.ts` (fenced/unfenced JSON,
whitespace tolerance, empty/omitted `findings`, list-field defaults, verbatim order,
every declared kind + unknown/absent-kind fallback, non-JSON / missing-field /
empty-keyPoints / unknown-key / out-of-range-confidence rejection, readable-message
guard, deep immutability, determinism). End-of-Session suite: **14 files / 154
tests**; full platform suite green: **494 tests / 46 files**.

**Code review** (high effort, 8 angles): no correctness defects. Two observations,
both deliberately at parity with the ratified precedent and left unchanged: the
duplicated `stripCodeFence`/`ExtractionError` follow the module's "parallel, not
shared" philosophy (EOS-005; cross-importing the compiler would introduce exactly
the coupling the module avoids), and the lowercase-only `json` fence label mirrors
the compiler idiom the task specified (degrades safely to `ExtractionError`).

# Lessons Learned

Reusing the compiler `extraction.ts` pattern made the contract fall out quickly, but
the two-module reconciliation mattered: the compiler folds fence-strip + validation
into one `parseExtraction(raw)`, while the End-of-Session convention is
`parseX → deep-frozen`. Keeping the single raw-string entry point **and** the module
deep-freeze (rather than adding a speculative `parse(unknown)`) honored both without
inventing surface. Keeping `EXTRACTION_KINDS` local — even though its values equal
`CANDIDATE_KINDS` — is the deliberate decoupling the plan called for: the soft hint
and the closed boundary taxonomy evolve independently, and M4 is the single place
that maps between them. The EOS-007 drift guard did its job again: exporting the
first error *class* in the module surfaced in the operations manifest (a class is a
function), a deliberate one-line pin rather than a surprise.

> **Engineering Rule**
>
> Structure is validated; content is not trusted. The model supplies
> non-deterministic content behind the port; the contract makes it safe by
> validating and freezing its shape — an invalid response is a typed
> `ExtractionError`, never a malformed object that flows downstream.
