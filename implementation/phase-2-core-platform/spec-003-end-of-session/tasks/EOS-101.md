# Purpose

Give the workflow the ability to **execute** its registered analyzers and
assemble their output into an immutable `ChangeSet`.

> **Why does this task exist?** M1 established the analyzer output contracts
> (`SessionChange`/`AnalyzerError`/`ChangeSet`), the `Analyzer` port, and a
> registry that **holds** analyzers but deliberately **does not execute** them
> (EOS-005). This task adds the missing behavior — the Collection stage's
> execution — as an **analyzer-agnostic** function. It knows nothing about git;
> it only runs whatever analyzers the registry carries and aggregates their
> results deterministically under the partial-collection model.

---

# Objective

Implement the Collection stage's execution:

- **`collectChanges(registry, session)`** — run every analyzer the
  `AnalyzerRegistry` holds against the identified `Session` and return an
  immutable `ChangeSet` (`{ sessionId, changes, errors, metadata }`).
- **Partial collection** — one analyzer throwing never aborts collection; it
  contributes exactly one `AnalyzerError` instead, and collection continues.
- **Deterministic ordering** — the registry's insertion order is authoritative;
  completion timing never influences the result.
- **Immutable output** — the `ChangeSet` is built through `parseChangeSet` so it
  is validated and deep-frozen.

This is the direct SPEC-003 analog of the SPEC-002 `collectKnowledge` execution
function (CB-010).

# Scope

## Included

- `collectChanges(registry: AnalyzerRegistry, session: Session): Promise<ChangeSet>`
- Deterministic failure mapping: a rejected analyzer → one `AnalyzerError`
  (`{ analyzer: <id>, message, recoverable: true }`), built via
  `parseAnalyzerError`; a helper deriving a human-readable message from an opaque
  rejection reason (the SPEC-002 `describeFailure` idiom — never a stack trace).
- Aggregation into a `ChangeSet` via `parseChangeSet` (validated + deep-frozen).
- Public export of `collectChanges` from the module barrel.
- Behaviour tests using **fixture analyzers** (one returns changes, one throws).

## Not Included

- Any concrete analyzer, including Git — that is EOS-102.
- The injectable `GitPort` or any git access — EOS-102/EOS-103.
- A `createCollectionEngine`-style frozen service handle — **intentionally
  deferred**. SPEC-002's engine handle earned its keep because it was injected
  into the pipeline assembly; here the composition root (M5) can call
  `collectChanges(registry, session)` directly, so a speculative service object
  is avoided (Implementation Playbook: *avoid unnecessary abstractions*). If M5
  composition later shows a handle is warranted, it is added then.
- Extraction, candidate generation, persistence, projection.

# Execution Invariant (determinism)

Collection is **deterministic with respect to registry order and analyzer
outputs**: for a fixed `AnalyzerRegistry` and `Session`, `collectChanges` always
produces a **deep-equal** `ChangeSet`, regardless of the wall-clock order in
which analyzers happen to settle. Formally:

- **Registry order is authoritative.** `changes` and `errors` are aggregated
  strictly in the registry's insertion index order; concurrent execution never
  leaks completion timing into the result.
- **Analyzer output is preserved verbatim.** Each analyzer's contributed
  `SessionChange[]` appears in the analyzer's returned order; the execution stage
  neither reorders, dedupes, nor filters within an analyzer's slice (intra-
  analyzer ordering is the analyzer's own responsibility — EOS-102 sorts).
- **Failures are positional data.** A rejected analyzer contributes exactly one
  `AnalyzerError` at its registry position; it never aborts or reorders the run.

This invariant is what makes the surrounding pipeline testable, and it is
asserted directly in this task's validation (same registry + session ⇒ deep-equal
`ChangeSet`).

---

# Implementation Notes

- **Concurrency without nondeterminism.** Run analyzers concurrently
  (`Promise.all` over `registry.analyzers`), each wrapped in its own try/catch so
  it can never reject the batch. `Promise.all` preserves registry index order
  regardless of completion timing; walk the settled outcomes back in that order.
- **Aggregation order.** `changes` are concatenated in registry order (analyzer
  0's changes, then analyzer 1's, …). Within a single analyzer, the analyzer's
  returned order is preserved verbatim — intra-analyzer ordering is the
  analyzer's own responsibility (EOS-102's GitChangeAnalyzer sorts
  deterministically). `errors` are likewise aggregated in registry order.
- **`recoverable` is always `true` for execution-caught failures.** Because
  execution never aborts, every rejection it catches is by definition
  recoverable (collection continued past it). A *fatal* (`recoverable: false`)
  error would be one that halts the workflow; execution produces none. The
  `Analyzer` port returns only `SessionChange[]`, so an analyzer cannot
  self-report a fatal error — that boundary is intentional and unchanged.
- **`sessionId`** on the `ChangeSet` is `session.id`.
- **`metadata`** defaults to `{}` (the schema default); this stage adds no
  metadata keys in v1.
- **No re-thrown failures once collection begins** — every failure is
  `AnalyzerError` data (mirrors CB-010).
- Suggested layout: `src/end-of-session/collection/collectChanges.ts` (+ an
  `index.ts` barrel), exported from `src/end-of-session/index.ts`. Tests in
  `tests/end-of-session/collection-execution.test.ts`.

# Validation

- A registry of resolving fixture analyzers produces a `ChangeSet` whose
  `changes` are in registry order; the result is deep-frozen.
- A registry containing a throwing fixture analyzer yields a `ChangeSet` that
  **still contains** the other analyzers' changes **and** one `AnalyzerError`
  for the failure — collection is not aborted (partial collection).
- The same registry + session always produces a deep-equal `ChangeSet`
  (determinism), independent of which analyzer resolves first.
- An empty registry yields a valid empty `ChangeSet` (`changes: []`,
  `errors: []`) for the session.
- `npm run typecheck`, `npm run build`, `npm test` green.

# Worklog Summary

Implemented the analyzer-agnostic Collection execution stage — the first M2
behavior — mirroring the SPEC-002 execution function (CB-010):

- **`collectChanges(registry, session): Promise<ChangeSet>`**
  (`src/end-of-session/collection/collectChanges.ts`) runs every analyzer the
  `AnalyzerRegistry` holds concurrently (`Promise.all`), each wrapped in its own
  try/catch so a rejection can never abort the batch. Fulfilled analyzers
  contribute their `SessionChange[]` verbatim; a rejected analyzer contributes
  exactly one `AnalyzerError` (`recoverable: true`) at its registry position.
  Outcomes are aggregated strictly in **registry index order**; the `ChangeSet`
  is stamped with `session.id` and built via `parseChangeSet` (validated +
  deep-frozen).
- Two private helpers mirror the SPEC-002 idiom: `describeFailure` (opaque
  rejection reason → non-empty, deterministic message; never a stack trace) and
  `toAnalyzerError` (→ validated, deep-frozen `AnalyzerError`).

  `src/end-of-session/index.ts`) and recorded it in the EOS-007 public-operations
  drift-guard manifest (`tests/end-of-session/foundation.test.ts`) — the guard
  fired on the new export, confirming it works; updated deliberately.

No `createCollectionEngine`-style handle was added (approved deferral): the M5
composition root can call `collectChanges` directly.

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/collection-execution.test.ts` — **16 tests** covering

# Lessons Learned

The frozen **execution determinism invariant** (deterministic w.r.t. registry
order and analyzer output) translated directly into tests: delayed fixture
analyzers that settle out of registry order pin the invariant far better than a
single happy-path assertion, and a whole-object `toEqual` across two runs proves
deep equality cheaply. Establishing the analyzer-agnostic stage first (no git)
kept EOS-101 a pure, fully-deterministic slice — git enters only at EOS-102/103,
exactly where the seam intended it. The EOS-007 public-surface drift guard did
its job: adding a public operation is a deliberate two-line manifest edit, not an
accident.
