# Purpose

Make the git observation match the range semantics EOS-402 already ratified, so a session
that creates a new file actually captures it.

> **Why does this task exist?** The workflow contradicts itself inside a single `Session`:
> `gitState.dirty` reports `true` (from `git status --porcelain`, which sees untracked files)
> while the `ChangeSet` omits them (from `git diff --name-status -M HEAD`, which does not). A
> session that creates a file and ends before `git add` captures **nothing** about it ‚Äî and
> new files are where new knowledge most often lives. Found by driving `aj session end` for
> real at EOS-408; invisible to every unit test, because they all supply their own `ChangeSet`.

---

# Objective

Extend **one adapter method**. In `analyzers/git/createGitPort.ts`:

`changes(range)` additionally runs, **only for working-tree ranges**:

```text
git ls-files --others --exclude-standard
```

mapping each returned path to an ordinary `GitFileChange`: `{ path, status: "A" }`.

**Working-tree vs. commit range** is decided by git's own vocabulary: a range containing
`..` is a commit-to-commit diff (`main..HEAD`) and carries no working-tree state; anything
else compares against the working tree (`HEAD`, `main`). This is **invocation strategy**,
which EOS-102 explicitly assigns to the adapter ‚Äî not domain policy.

---

# Scope

## Included

- The second read in `createGitPort.changes`, and the `..` test that gates it.
- Behaviour tests over disposable fixture repos:
  - an untracked file reaches the `ChangeSet` as `added`, beside modified and staged files;
  - **`.gitignore`d files never appear**;
  - **untracked files are excluded for a commit range** (`<ref>..HEAD`) and included for
    working-tree ranges (`HEAD`, and a bare `<ref>`);
  - a **staged-new** file appears exactly once, not twice (the reads are disjoint);
  - deterministic, path-sorted output across runs;
  - no git write is invoked.

## Not Included

- **The analyzer** ‚Äî `createGitChangeAnalyzer` is **not touched**. It already translates an
  `"A"`; it must not learn that untracked files exist.
- **`GitPort`** ‚Äî the interface is unchanged. No new method, no new port.
- **Any contract** ‚Äî `GitFileChange`, `SessionChange`, `ChangeSet`, `Session` all unchanged.
- Deduplication ‚Äî the two reads are disjoint (`git diff` reports tracked paths, `ls-files
  --others` untracked ones), so none is added.
- Truncation, filtering, or any judgement about *which* untracked files matter beyond
  `.gitignore` ‚Äî that would be policy, and this task adds none.

---

# Invariants preserved (the reviewer's conditions)

- **The analyzer stays a pure translator** ‚Äî literally unmodified. It cannot distinguish an
  untracked file from a staged one; both arrive as `status: "A"`.
- **Deterministic ordering** ‚Äî the analyzer already sorts by path (EOS-102), unchanged.
  `ls-files` output is itself sorted.
- **Read-only** ‚Äî `ls-files` is an observation. The EOS-401 read-only proof still holds.
- **The adapter stays minimal** ‚Äî invoke, parse, produce. It gains one command and one
  well-defined piece of git vocabulary, and no domain knowledge.

# Worklog Summary

Implemented the approved EOS-D11 amendment, exactly as proposed:

- **`createGitPort.changes`** now also runs `git ls-files --others --exclude-standard` for
  **working-tree ranges**, mapping each untracked path to an ordinary `{ path, status: "A" }`.
  `comparesWorkingTree(range)` decides by git's own vocabulary ‚Äî a range containing `..` is a
  commit diff and carries no working-tree state.

**Adapter-only, proven by diff.** One file changed. `createGitChangeAnalyzer` and `GitPort`
are **byte-untouched**, and **M2's 38 analyzer/collection tests pass unmodified** ‚Äî the proof
the change is confined to where EOS-D11 authorized it. The analyzer stays a pure translator:
it already translates an `"A"`, so it cannot tell an untracked file from a staged one, and
ordering stays deterministic because it already sorts by path. The two reads are disjoint
(`git diff` reports tracked paths, `ls-files --others` untracked ones), so no deduplication
exists to get wrong.

**8 new real-git tests**: untracked reported as `added`; tracked and untracked together;
**no duplication** of a staged-new file; **`.gitignore` respected**; **excluded for a commit
range** and included for a bare ref (`main`, which also compares the tree); deterministic
across reads; and no git write.

**Acceptance extended and re-validated.** The EOS-409 fixture session now ends with a file
never `git add`ed *and* an ignored one ‚Äî the shape of a real session. The new ¬ß19 scenario
asserts on **the prompt the model was actually shown**, captured from the stub, rather than
on a finding: a stubbed finding would only prove the stub chose to cite the path, not that
the pipeline ever saw it. `filesAnalyzed` is now **7** (6 tracked + 1 untracked; `build.log`
correctly excluded), and the workflow no longer contradicts its own `dirty` flag.

**Validation.** Typecheck, build, and the explicit test typecheck clean. **Full M5 acceptance
re-validation: 713 tests / 58 files, green, run three consecutive times.**

**Code review** (high effort, 8 angles) ‚Äî two findings, **neither this task's to fix**:

1. **Non-ASCII path quoting (pre-existing).** Git's `core.quotePath` defaults to true, so
   `caf√©.ts` is reported as the literal `"caf\303\251.ts"` ‚Äî by **both** `git diff
   --name-status` *and* `git ls-files --others` (verified directly). Neither parser unquotes,
   so such a path would carry quotes and octal escapes into `SessionChange.path`, a candidate
   id, and provenance. **EOS-411 did not introduce it**: M2's frozen `parseNameStatus` has had
   this exposure since EOS-103, and the new read inherits it *identically* ‚Äî the consistent
   outcome. Fixing it means changing the frozen parser, which is beyond this amendment and
   beyond M5. **Recorded under MILESTONES ‚Üí Future Hardening**, alongside the accepted
   `maxBuffer` deferral.
2. **Two sequential git spawns** for a working-tree range. Left alone deliberately:
   parallelising would require starting `ls-files` before the range check, spawning it
   needlessly for every `--since` run, and the current shape keeps a commit range at exactly
   one spawn.

# Lessons Learned

The change was three lines of logic and a paragraph of reasoning, and the reasoning is what
made it safe. Deciding *where* the untracked read belonged ‚Äî the adapter, because "which git
commands answer a range" is invocation strategy ‚Äî is what let the analyzer stay untouched.
Had the decision gone into the analyzer instead, it would have needed to inspect the range,
hold a policy, and stop being a pure translator; the fix would have been the same size and the
seam would have been worse.

The gap itself is the milestone's sharpest lesson. `dirty: true` beside a `ChangeSet` that
omitted the new files was a contradiction sitting in plain sight since M2, and every unit
test missed it because they all supply their own `ChangeSet` ‚Äî the fixture *was* the
assumption. Only running the real command over a real repository showed `Files analyzed : 1`
where two files had changed. Stubs prove that stages agree with their doubles; only the real
thing proves they agree with the world.
