# Purpose

Establish the extensibility seam for *what changed in a session*.

> **Why does this task exist?** The workflow must support additional analyzers
> (documentation, lessons-learned, IDE timeline) without changing orchestration.
> This task defines the analyzer output contracts and the port + registry seam —
> the boundary only, no analyzer behavior.

---

# Objective

Define:

- **`SessionChange`** — a normalized change record: `{ id (namespaced, e.g.
  `git:<path>`), kind (soft hint: source|documentation|config|test|other), path,
  changeType (added|modified|deleted|renamed), summary, metadata }`.
- **`AnalyzerError`** — `{ analyzer, message, recoverable }`.
- **`ChangeSet`** — `{ sessionId, changes: SessionChange[], errors:
  AnalyzerError[], metadata }` (items **and** errors, mirroring `CollectionResult`).
- **`Analyzer`** port — the interface an analyzer implements to contribute changes
  for a `Session`.
- **Analyzer registry** — holds registered analyzers in deterministic order; does
  not execute them (execution is M2).

---

# Scope

## Included

- Zod schemas + types for `SessionChange`, `AnalyzerError`, `ChangeSet`
- The `Analyzer` port interface
- A registry factory that accepts and holds analyzers (frozen, deterministic order)
- Public exports

## Not Included

- Any analyzer implementation, including Git (M2)
- Registry *execution* / partial-collection behavior (M2)
- Extraction, candidates, persistence

# Implementation Notes

- Model the registry on the SPEC-002 **provider registry** precedent (CB-005):
  accept and **hold** analyzers on a frozen handle, expose them for later
  execution, and validate construction inputs fail-fast — but add **no `run()`
  method** (that is M2 behavior; a boundary task earns its keep by being
  observable, per the CB-007 principle).
- `ChangeSet` carries **both** changes and errors so M2 can implement **partial
  collection** — one analyzer failing never aborts the workflow.
- `SessionChange.kind` is a soft hint (lenient enum) — analyzers may not classify
  perfectly; downstream must not depend on precise kinds.
- Namespaced ids (`<analyzer>:<local>`) mirror `SourceRecord`'s namespacing so ids
  are stable and origin-tagged.

# Validation

- Contracts validate/reject correctly and are immutable.

  execute them (enforced with a fixture analyzer that throws on execution).
