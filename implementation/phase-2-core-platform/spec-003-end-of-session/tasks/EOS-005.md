# Purpose

Establish the extensibility seam for *what changed in a session*.

> **Why does this task exist?** The workflow must support additional analyzers
> (documentation, lessons-learned, IDE timeline) without changing orchestration.
> This task defines the analyzer output contracts and the port + registry seam —
> the boundary only, no analyzer behavior.

---

# Objective

Define:

- **`SessionChange`** — a normalized change record: `{ id (namespaced, e.g.
  `git:<path>`), kind (soft hint: source|documentation|config|test|other), path,
  changeType (added|modified|deleted|renamed), summary, metadata }`.
- **`AnalyzerError`** — `{ analyzer, message, recoverable }`.
- **`ChangeSet`** — `{ sessionId, changes: SessionChange[], errors:
  AnalyzerError[], metadata }` (items **and** errors, mirroring `CollectionResult`).
- **`Analyzer`** port — the interface an analyzer implements to contribute changes
  for a `Session`.
- **Analyzer registry** — holds registered analyzers in deterministic order; does
  not execute them (execution is M2).

# Implementation Notes

- Model the registry on the SPEC-002 **provider registry** precedent (CB-005):
  accept and **hold** analyzers on a frozen handle, expose them for later
  execution, and validate construction inputs fail-fast — but add **no `run()`
  method** (that is M2 behavior; a boundary task earns its keep by being
  observable, per the CB-007 principle).
- `ChangeSet` carries **both** changes and errors so M2 can implement **partial
  collection** — one analyzer failing never aborts the workflow.
- `SessionChange.kind` is a soft hint (lenient enum) — analyzers may not classify
  perfectly; downstream must not depend on precise kinds.
- Namespaced ids (`<analyzer>:<local>`) mirror `SourceRecord`'s namespacing so ids
  are stable and origin-tagged.

# Worklog Summary

Established the analyzer (Collection stage) seam — its output contracts, the
`Analyzer` port, and a boundary-only registry — with no analyzer behavior.

Contracts (`src/end-of-session/contracts/change/`), following the module pattern
(Zod `.strict()` → `deepFreeze` → `DeepReadonly`, public export via `contracts/`):

- **`SessionChange`** — `{ id (namespaced `<analyzer>:<local>`), kind, path,
  changeType, summary, metadata }`. `kind` is a **soft hint** (lenient
  `z.enum(...).catch("other")` — analyzers may misclassify; downstream must not
  depend on it); `changeType` is a **closed** enum (`added|modified|deleted|
  renamed`) — a definite classification.
- **`AnalyzerError`** — `{ analyzer, message, recoverable }`, the canonical
  analyzer failure. Owned here; the EOS-004 `SessionReport` error shape aligns to
  it (correct producer-owned direction).
- **`ChangeSet`** — `{ sessionId, changes[], errors[], metadata }`, carrying changes
  **and** errors together so M2 can implement partial collection (mirrors the
  SPEC-002 `CollectionResult`).
- **`Analyzer`** port — a behavioural interface (`analyze(session): Promise<readonly
  SessionChange[]>`) mirroring the CB-004 `KnowledgeProvider`; extends
  `AnalyzerMetadata` (`id`/`name`/`description`).

Registry (`src/end-of-session/registry/createAnalyzerRegistry.ts`): modeled
**line-for-line on the SPEC-002 provider registry** (CB-005) with `Analyzer`
substituted. It owns **deterministic registration and lookup only** — deterministic
insertion order, fail-fast on empty/duplicate `id`, frozen handle + `analyzers`
array, `get(id)` lookup — and deliberately has **no** `run`/execute method,
orchestration, retry, reporting, or policy (execution + partial-collection error
handling are M2).

Design decisions:

- **`metadata`** is a bounded `z.record(string, string)` map (default `{}`) on
  `SessionChange`/`ChangeSet`. The task specifies a metadata field; constraining
  values to strings keeps it a small immutable annotation rather than the untyped
  dumping ground CB-004 forbids on `KnowledgeItem`. Downstream must not depend on
  specific keys.
- The `AnalyzerRegistry` intentionally **parallels** `ProviderRegistry` rather than
  sharing a generic factory — a shared abstraction would couple SPEC-003 to
  SPEC-002 across a module boundary (premature).

**Validation.** `npm run typecheck`, `npm run build` green; full suite
425 tests / 39 files (26 new), including a fixture analyzer whose `analyze` throws,
proving the registry never executes it. Reviewed against all 8 code-review angles —
no correctness defects; two non-blocking observations (task-specified `metadata`
maps, intentional registry parallelism) recorded and left as-is. Confirmed
empirically that zod's `.default({})` yields a fresh object per parse (no
shared-frozen-default aliasing).

# Lessons Learned

Two enums in one contract can warrant **different** strictness: a *soft hint*
(`kind`) earns a lenient `.catch`, while a *definite classification* (`changeType`)
stays a closed enum — leniency is a per-field judgment about who controls the value
and how reliably, not a module-wide default. Establishing the `Analyzer`/registry
seam as pure infrastructure (registration + lookup, no execution) keeps later
milestones additive: M2 adds the Git analyzer and the execution stage without
touching this boundary — the CB-005/CB-007 "observable boundary earns its keep"
principle.
