# Purpose

Deliver the **Persistence** stage — the Review Store — which writes the canonical
`CandidateKnowledge[]` and the `SessionReport` for a run to
`<vault>/knowledge-review/pending/<session-id>/` (EOS-D2). This is the SPEC-003 →
SPEC-004 **filesystem boundary**: the store is where captured knowledge waits for the
human review gate. It is **persistence-only** and **never performs version control**
(ADR-002 §4, AJS-005 §7); it writes only to the **non-canonical** review area.

> **Why does this task exist?** EOS-301 produces canonical candidates in memory; they
> must be durably written where SPEC-004 can read them, without ever touching canonical
> `foundation/`, `library/`, or the generator-owned `wiki/`, and without git. This task
> supplies that capability as a scoped, path-guarded store — the direct End-of-Session
> analog of `FilesystemWikiStore` (persistence-only, destination-scoped, `appendLog`),
> specialized to the SPEC-003 artifacts and layout.

---

# Objective

Implement, under `src/end-of-session/store/`:

- **`ReviewStore`** — the store interface. Persistence-only; all operations are confined
  to the configured review destination and rejected if they would escape it.
- **`createFilesystemReviewStore({ destination }): ReviewStore`** — the filesystem
  implementation. Resolves + validates the destination (realpath, must exist and be a
  directory), **validates it is non-canonical** (`∉ {foundation/, library/, wiki/}`,
  path-escape guarded), and writes the per-session layout. Returns a frozen handle.

## Store surface (domain-aware — recommended; see Open Decisions)

Persist the SPEC-003 artifacts for one session under a **single directory per session**:

```
<destination>/pending/<session-id>/
    candidates/<candidate-id>.json     one canonical CandidateKnowledge per file
    report.json                        the SessionReport for the run
    log.md                             append-only execution log (appendLog)
```

- **`saveCandidates(sessionId, candidates: CandidateKnowledge[]): Promise<void>`** — writes
  each candidate as canonical **JSON** (structured data — SPEC-004 consumes data, not
  markdown; EOS-D4), file-named by its stable `id`, under `pending/<session-id>/candidates/`.
- **`saveReport(sessionId, report: SessionReport): Promise<void>`** — writes `report.json`.
- **`appendLog(sessionId, entry: string): Promise<void>`** — appends a line to the session's
  `log.md` (the `WikiStore.appendLog` precedent; carries no version-control semantics).
- **`locate(): Promise<ReviewLocation>`** — resolves + validates the destination root
  (the `WikiStore.locate` precedent); the composition root (M5) calls it as a precondition.

The store **serializes and lays out**; it does **not** interpret candidate content, decide
*which* candidates exist, generate, project, or approve.

---

# Persistence Invariant (write-only, non-canonical, no git)

The Review Store's single responsibility is durable persistence to the review area. It:

- **never invokes git** — no `commit`, no staging, no VCS of any kind (ADR-002 §4,
  AJS-005 §7); committing the review area, if ever wanted, is orchestration's concern;
- **writes only inside the configured destination** — every path is guarded both
  lexically (no `..`, no absolute paths, no NUL) and against symlink escape (the
  `FilesystemWikiStore` guard); a `<session-id>` is likewise validated so it cannot
  escape `pending/`;
- **writes only to a non-canonical destination** — construction fails if the resolved
  destination is, or is inside, `foundation/`, `library/`, or `wiki/` (defense in depth
  against a mis-set `reviewPath`), satisfying SPEC-003 §17 "must never modify canonical
  handbook content";
- **does not interpret** the artifacts — it serializes `CandidateKnowledge` / `SessionReport`
  and appends log lines; it does not parse frontmatter, deduplicate, or understand review
  semantics (parity with the semantics-free `WikiStore`);
- **is deterministic in layout** — the same `sessionId` + artifacts produce the same file
  tree and paths (candidate files named by stable id; sorted/derived deterministically).

# Scope

## Included

- `src/end-of-session/store/` — `ReviewStore.ts` (interface + `ReviewLocation`),
  `FilesystemReviewStore.ts` (`createFilesystemReviewStore`, `ReviewStoreError`),
  `index.ts`. Mirrors `src/knowledge/wiki-store/` structure.
- Destination resolution + validation (realpath, exists, is-directory), **non-canonical
  guard**, and the lexical + symlink path guards (reuse the `FilesystemWikiStore` idioms;
  do not weaken them).
- Per-session layout writer (`saveCandidates` / `saveReport` / `appendLog` / `locate`),
  creating parent directories as needed (`mkdir -p`, the `write` precedent).
- Canonical **JSON** serialization of candidates + report (stable key output; pretty or
  compact fixed by implementation — deterministic either way).
- Frozen factory handle; public exports (`createFilesystemReviewStore`, `ReviewStore`,
  `ReviewLocation`, `ReviewStoreError`) from the module barrel; EOS-007 drift-guard
  manifest updated.
- **Behaviour tests using temporary directories** (no real vault; the `FilesystemWikiStore`
  test precedent):
  - `saveCandidates` writes one JSON file per candidate under
    `pending/<session-id>/candidates/`, round-trips to a deep-equal `CandidateKnowledge`
    (via `parseCandidateKnowledge`), and is deterministic in layout;
  - `saveReport` writes `report.json` round-tripping via `parseSessionReport`;
  - `appendLog` appends to the session `log.md` (append, not overwrite);
  - **the non-canonical guard**: constructing against a `foundation/`, `library/`, or
    `wiki/` destination throws — a candidate write can never land in canonical space;
  - path-escape attempts (`..`, absolute, NUL, a crafted `sessionId`) are rejected;
  - a non-existent / non-directory destination fails clearly at `locate`/construction;
  - empty candidates ⇒ an empty (or absent) `candidates/` dir with no error.

## Not Included

- Candidate generation — **EOS-301** (the store persists what it is given).
- The `reviewPath` **configuration** and the config→destination resolution — **EOS-303**
  (config) and **M5** (the composition root resolves `<handbook>/<reviewPath>` and passes
  `destination`, exactly as `createKnowledgePipeline` resolves `generatedWikiPath`).
- Building the `SessionReport` and the `ReviewPackage` markdown — **M5** (the projector
  writes `review-package.md` into the same session directory then).
- Wiring the store into the workflow / calling it from `run` — **M5**.
- Any git operation, approval, publication, or dedupe — out of scope permanently for the store.

# Design Notes / Open Decisions (for the Planning Review)

- **Domain-aware store (RATIFIED — [EOS-D6](../decisions/EOS-D6-review-store-domain-api.md)).**
  `WikiStore` is deliberately semantics-free (`write(path, content)`), pushing layout onto
  the caller. SPEC-003, however, defines the Persistence *stage* as "writes the canonical
  candidates and the `SessionReport`" — a domain responsibility. The reviewer **accepted a
  domain-aware `ReviewStore`** (`saveCandidates` / `saveReport` / `appendLog` / `locate`)
  that owns the per-session layout and JSON serialization: one consumer, one layout, so a
  generic store + a separate layout module would be an abstraction without a second client;
  the domain surface keeps the stage cohesive and gives M5 a clean call site, while
  preserving every `WikiStore` persistence guarantee (no git, path-guarded, scoped). This
  establishes the long-term SPEC-003 → SPEC-004 filesystem boundary shape — recorded as
  **EOS-D6**.
- **Candidates persisted as JSON, one file per candidate (RATIFIED).** Candidates are the
  *canonical structured* artifact (EOS-D4) — SPEC-004 consumes data, not prose — so JSON, not
  markdown. One file per candidate (named by stable `id`) over a single `candidates.json`
  array gives SPEC-004 stable, individually addressable units and clean diffs.
- **Non-canonical guard placement (RATIFIED — at construction).** The guard runs during
  store construction (fail-fast, like `FilesystemWikiStore`'s destination validation) so a
  mis-set `reviewPath` can never even build a store that points at canonical space — defense
  in depth beyond the M5 config default.
- **`report.json` in M4 vs. M5.** The store *capability* to persist a report is built and
  tested here (with a fixture report); the *real* report is assembled and written by M5
  orchestration at run end. `review-package.md` is written to the same session directory by
  the M5 projector — the store simply guards/hosts that directory.

# Validation

- `saveCandidates` / `saveReport` write the documented layout under
  `pending/<session-id>/`; artifacts round-trip to deep-equal contracts; layout is
  deterministic for the same inputs.
- `appendLog` appends (does not overwrite) the session `log.md`.
- Constructing against `foundation/`, `library/`, or `wiki/` throws; `..`/absolute/NUL/
  crafted-`sessionId` escapes are rejected; a missing/non-directory destination fails clearly.
- No git invoked anywhere; the factory handle is frozen; canonical content is provably untouched.
- `npm run typecheck`, `npm run build`, `npm test` green.
