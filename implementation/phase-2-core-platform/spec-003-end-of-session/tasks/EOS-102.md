# Purpose

Deliver the workflow's **first concrete analyzer** — the one that determines
*what changed in the session's git repository* — behind a small **injectable git
port** so it is deterministic and testable without touching a real repository.

> **Why does this task exist?** v1 registers exactly one analyzer
> (PIPELINE-ARCHITECTURE §Collection): the Git change analyzer. It must map git's
> view of the session's changes onto the normalized `SessionChange` contract. Git
> access is a side-effecting, environment-dependent dependency, so it is isolated
> behind a `GitPort` interface — the analyzer depends on the port, never on git
> directly, mirroring how extraction depends on the injected `TextGenerator`.

---

# Objective

Implement, behind a testability seam:

- **`GitPort`** — a minimal, injectable interface abstracting the git reads the
  analyzer needs. It exposes the changed files (path + status, plus the former
  path for renames) for a resolved range; it does **not** expose write/commit
  operations (ADR-002: version control is orchestration, not an engine concern).
- **`GitChangeAnalyzer`** (`createGitChangeAnalyzer(gitPort)`) — an `Analyzer`
  implementation that, given a `Session`, reads the port over
  `session.gitState.range` and maps each git file change onto a normalized,
  deterministically-ordered `SessionChange`.

The analyzer returns `readonly SessionChange[]` (the `Analyzer` port contract);
it does **not** build a `ChangeSet`, aggregate errors, or persist — that is the
EOS-101 execution stage.

# Scope

## Included

- The `GitPort` interface + its raw change type (e.g. `GitFileChange`:
  `{ path, status, oldPath? }`), owned in the analyzer's module.
- `createGitChangeAnalyzer(gitPort): Analyzer` with stable metadata
  (`id: "git"`, `name`, `description`).
- Mapping rules (below): git status → `changeType`; path heuristic → `kind`;
  namespaced id `git:<path>`; deterministic human-readable `summary`; rename's
  former path carried in `metadata`.
- Deterministic intra-analyzer ordering (sort by `path`, stable).
- Public exports (`createGitChangeAnalyzer`, `GitPort`, `GitFileChange`).
- Unit tests driving the analyzer through a **stub `GitPort`** — no real git.

## Not Included

- The **real git-backed `GitPort` adapter** (shells out to git) and its
  temp-repo integration test — that is **EOS-103** (where a real repo makes the
  test meaningful).
- Registry execution / `ChangeSet` aggregation — that is EOS-101.
- Deciding *what the range covers* (uncommitted+staged vs. `--since <ref>`): the
  analyzer consumes `session.gitState.range` as given; how that range string is
  constructed is a Session-creation/CLI concern (M5) and a git-adapter concern
  (EOS-103). It does not affect this task's contract.
- Extraction, candidates, persistence, projection.

# Implementation Notes

- **`changeType` mapping (closed enum).** git status → contract `changeType`:
  `A → added`, `M → modified`, `D → deleted`, `R → renamed`. Finer-grained
  statuses map onto the nearest of these (e.g. copied `C → added`, type-change
  `T → modified`). An unmapped/unknown status is a mapping bug — surface it, do
  not silently coerce (the contract's `changeType` is a definite classification,
  not a soft hint).
- **`kind` heuristic (soft hint).** Classify from the path only, best-effort:
  test files → `test`; `*.md`/docs paths → `documentation`; config
  (`*.json`, `*.yml`, dotfiles, etc.) → `config`; source under `src/` → `source`;
  otherwise `other`. Because `kind` is a lenient hint, downstream must not depend
  on it — imperfect classification is acceptable by contract.
- **Ids.** `git:<path>` (namespaced `<analyzer>:<local>`, the `SourceRecord`
  precedent). For a rename, key on the new `path`; carry the former path in
  `metadata` (e.g. `{ oldPath: "<former>" }`).
- **`summary`.** A deterministic, human-readable one-liner derived from
  `changeType` + `path` (e.g. `"modified src/foo.ts"`) — no timestamps, no
  nondeterministic content.
- **Deterministic ordering.** Sort the produced `SessionChange[]` by `path`
  (stable, e.g. localeCompare with a fixed locale or code-unit order) so the same
  repository state always yields the same order regardless of the port's return
  order.
- **Read-only port.** `GitPort` exposes only reads; it never stages, commits, or
  mutates the repo. Keeping the interface tiny keeps the EOS-103 real adapter and
  the unit stub trivially interchangeable.
- Suggested layout: `src/end-of-session/analyzers/git/` — `GitPort.ts`,
  `createGitChangeAnalyzer.ts`, `index.ts`. Tests in
  `tests/end-of-session/git-change-analyzer.test.ts`.

# Validation

- Given a stub `GitPort` returning a fixed set of file changes (add/modify/
  delete/rename), the analyzer returns the expected `SessionChange[]`:
  correct `changeType`, namespaced ids, rename's former path in `metadata`,
  path-sorted order.
- The same stub input always yields a deep-equal `SessionChange[]` (determinism),
  independent of the port's return order.
- A stub `GitPort` that rejects surfaces as a rejection from `analyze` (the
  EOS-101 execution stage converts it to an `AnalyzerError`; the analyzer itself
  does not swallow it).
- The analyzer never calls real git in tests (enforced by using only the stub).
- `npm run typecheck`, `npm run build`, `npm test` green.
