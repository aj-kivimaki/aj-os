# Purpose

Give the workflow the **read-only git observations a `Session` requires**: the head commit,
whether the working tree is dirty, and the current branch.

> **Why does this task exist?** This is a **gap between the frozen plan and the delivered
> code**, surfaced at M5 planning. The frozen `Session` contract requires
> `gitState.head` (non-empty), `gitState.dirty`, and `branch` (non-empty). EOS-002 recorded
> that the git access to populate `gitState` was **M2's** job — but M2 delivered a `GitPort`
> with exactly one read, `changes(range)`, because that was all the `GitChangeAnalyzer`
> needed. The result: **no `Session` can be constructed today**, and none is — a repo-wide
> check finds `parseSession` used only by the schema and by tests. M5 cannot build a real
> session without these reads. This task supplies them, and *only* them.

---

# Objective

Extend the existing read-only git seam — **not** introduce a second one:

- **`GitPort`** (`analyzers/git/GitPort.ts`) gains three reads:
  - `head(): Promise<string>` — the HEAD commit hash;
  - `dirty(): Promise<boolean>` — whether the working tree has uncommitted changes;
  - `branch(): Promise<string | null>` — the current branch name, or `null` when HEAD is
    detached (**refined during implementation — see the Worklog**).
- **`createGitPort(repositoryPath)`** (`analyzers/git/createGitPort.ts`) implements them
  with read-only invocations, in the adapter's existing *invoke → parse → produce* style:
  - `head` → `git rev-parse HEAD` (trimmed);
  - `dirty` → `git status --porcelain` (non-empty output ⇒ dirty);
  - `branch` → `git branch --show-current` (trimmed; empty ⇒ detached ⇒ `null`).

`changes(range)` is **unchanged**. No existing behaviour, signature, or guarantee moves.

---

# Read-Only Invariant (unchanged, and reaffirmed)

The seam remains exactly what EOS-102 froze — this task widens *what is observed*, never
*what is permitted*:

- **Reads only.** No command that stages, commits, checks out, fetches, or mutates the
  repository, in the port or the adapter (ADR-002 §4, AJS-005 §7). `rev-parse` and
  `status` are strictly observational.
- **No orchestration or policy.** The adapter still invokes, parses, and produces. It does
  not discover the repository (the path stays injected), construct ranges (EOS-402 does),
  retry, filter, or decide anything.
- **Failures propagate.** A non-zero git exit rejects; the adapter never swallows, retries,
  or substitutes a default. Callers decide what a failure means — for `changes`, the
  EOS-101 execution stage turns it into one `AnalyzerError`; for these three reads, the
  caller is EOS-402/EOS-406, and a failure is **fatal** (SPEC-003 §15 "repository
  unavailable"), because a session with no head or branch cannot be identified at all.

# Scope

## Included

- The three read methods on the `GitPort` interface, documented in the port's existing
  voice (what is observed, in git's vocabulary; not how it is used).
- Their implementation in `createGitPort`, reusing the adapter's `execFileAsync` idiom.
- Behaviour tests over a **disposable fixture repo** (the EOS-103 precedent — real git,
  temp dir, no fixture of the developer's own repo):
  - `head` returns the fixture's HEAD hash and matches `git rev-parse HEAD`;
  - `dirty` is `false` on a clean tree and `true` with an uncommitted modification (and
    with a staged-but-uncommitted change);
  - `branch` returns the fixture's branch, including after a checkout;
  - a broken repository path rejects for each read.
- No public-surface growth: `createGitPort` / `GitPort` are already exported, so the
  EOS-007 drift-guard manifest is **unchanged** (worth stating explicitly at review — the
  operation count stays at 20).

## Not Included

- Constructing the `range`, minting session identity, or assembling `gitState` — **EOS-402**
  (this task only *observes*; it does not shape a `Session`).
- Any write, stage, or commit capability — permanently out of scope for this seam.
- Repository discovery — the path stays injected at construction (**EOS-407** supplies it).
- Changing `changes(range)`, the `GitChangeAnalyzer`, or the collection stage — all frozen
  and untouched.
- `execFile` `maxBuffer` hardening — a reviewer-accepted M2 deferral, still deferred
  (MILESTONES *Deferred (post-v1)*).

# Design Notes / Open Decisions (for the Planning Review)

- **Extend `GitPort` vs. a separate port — RATIFIED: extend ([EOS-D7](../decisions/EOS-D7-git-port-extension.md)).**
  This was the one architectural choice in the task, and it changes an M2-frozen contract.
  The reviewer accepted the extension at the M5 Planning Review; the reasoning is preserved
  below and recorded canonically in EOS-D7.
  - **(a) Extend `GitPort` (recommended).** One seam for "read-only git access to this
    repository", one adapter, one injection point, one test double. The port's own
    docstring already frames it as *the* read-only git seam ("Keeping the interface tiny
    keeps the real adapter and the unit-test stub trivially interchangeable"); four reads
    is still tiny. Cost: the `GitChangeAnalyzer` depends on an interface with three methods
    it never calls — a mild interface-segregation smell, and honest to name.
  - **(b) A separate `GitStatePort` (head/dirty/branch), leaving `GitPort` frozen.**
    Keeps each consumer's interface minimal and leaves M2 literally untouched. Cost: two
    ports, two adapters (or one class implementing both), two injection points and two
    stubs — for one repository, read the same way, by one composition root. That is the
    "abstraction without a second client" this project has rejected twice already (EOS-D6
    on the store; the M3 decision to keep `TextGenerator` local).
  - **Recommendation: (a).** The seam's identity is "read-only git access", not "the
    analyzer's reads". Splitting it would fragment one concept across two ports to avoid an
    additive change. **If the reviewer agrees, record the extension of the frozen M2 port as
    EOS-D7** — the same treatment EOS-D6 gave the store's surface. If the reviewer prefers
    (b), only this task changes; EOS-402 injects a second port and nothing else in M5 moves.
- **Naming: `dirty()` vs `isDirty()`.** The contract field is `gitState.dirty`; `dirty()`
  keeps the port's vocabulary aligned with the contract it feeds. Minor — reviewer's call.
- **Why not derive `head` from `SessionContext.commitHash`?** The context's `commitHash` is
  *optional* and caller-supplied. The session's head must be **observed**, not asserted, or
  provenance records a claim rather than a fact. `branch` is likewise required and must not
  depend on the caller getting it right (see EOS-408's thin-CLI argument).

# Worklog Summary

Extended the read-only git seam per EOS-D7. `GitPort` gained `head()`, `dirty()`, and
`branch()`; `createGitPort` implements them with `git rev-parse HEAD`,
`git status --porcelain`, and `git branch --show-current`, all through a small local
`git(...args)` helper that funnels every invocation through one `execFileAsync` call with
the injected `cwd`. `changes(range)`, the `GitChangeAnalyzer`, and the collection stage are
untouched; the M2 suites pass unmodified, which is what proves the extension additive. No
public-surface growth (`GitPort`/`createGitPort` were already exported), so the EOS-007
drift-guard manifest is unchanged at 20 operations.

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/git-state-reads.test.ts` (14 tests, disposable fixture repos, real
git) covers `head` (agrees with `git rev-parse HEAD`, bare/trimmed, tracks a new commit),
`dirty` (clean / unstaged / **staged-but-uncommitted** / untracked / back to clean),
`branch` (checked-out branch, tracks a checkout, **detached ⇒ `null`**), fatal failure
propagation for all three reads, construction guard, the exact four-read frozen surface, and
an explicit **read-only proof** (HEAD, index, and working tree byte-identical after every
read — the ADR-002 promise). End-of-Session suite **18 files**; full platform suite
**548 tests / 50 files**, all green.

**Code review** (high effort, 8 angles) — five findings, two of them real defects, both
fixed:

1. **The extension silently broke two M2 test stubs.** `git-change-analyzer.test.ts` declares
   `const failing: GitPort = {…}` and a `stubGitPort(): GitPort & {…}` implementing only
   `changes` — both became type errors (TS2739/TS2322: "missing … head, dirty, branch")
   the moment the port grew. **Nothing caught it**: `tsconfig.json` scopes typechecking to
   `include: ["src"]`, and Vitest strips types, so the suite stayed green while the code
   lied about its type. Confirmed by running `tsc` directly over the file. **Fixed** by
   giving the stubs the three reads as functions that *throw* — turning EOS-D7's accepted
   interface-segregation cost into an enforced guarantee: if the analyzer ever reaches for
   session state, the tests fail loudly instead of passing on a plausible dummy.
2. **`branch()` returned the literal `"HEAD"` when HEAD is detached** (rebase, bisect, tag
   checkout — and `aj session end` may well be run mid-rebase). That string is non-empty, so
   it would satisfy `Session.branch`'s `.min(1)` and flow into the review package and every
   candidate's provenance as a branch that does not exist — and it is indistinguishable from
   a branch genuinely named `HEAD`. **Fixed**: switched to `git branch --show-current`
   (empty when detached) and refined the signature to `Promise<string | null>` so the port
   reports the fact rather than laundering it. See the deviation note below.

Also raised, not fixed: **tests are outside the typecheck scope** (the systemic condition
that hid finding 1 — it will recur for every M5 stub); a **hard-coded 40-hex hash
assertion** (relaxed to `/^[0-9a-f]+$/`, since a SHA-256 repo yields 64); and **three git
subprocess spawns** per session for the state reads (accepted — each read stays minimal and
independently stubbable, matching the adapter's philosophy).

## Deviation from the ratified signature (for the reviewer)

EOS-D7 records `branch(): Promise<string>`. The implementation returns
**`Promise<string | null>`**. The decision's *substance* — extend the one read-only seam
rather than add a second git abstraction — is unchanged; this is a fidelity refinement found
at code review, and the alternative is knowingly recording a non-existent branch. It hands
**EOS-402 a decision**: `Session.branch` is required and `.min(1)`, so the Session factory
must choose what a detached session records (a rendering such as `detached@<short-head>`; or
treating detached HEAD as fatal; or a contract change). Flagged rather than pre-empted —
EOS-402's scope, not this task's.

# Lessons Learned

The gap this task closed was invisible for three milestones because **nothing constructed a
`Session`** — the contract was frozen in M1, its git access was assigned to M2, and M2
delivered only what its one consumer needed. A frozen contract with no production caller is
not validated; it is only *typed*. The M5 acceptance suite (EOS-409) is the first thing that
would have caught it, which is an argument for building the vertical slice earlier rather
than later.

The review's sharpest finding came from asking "who else implements this interface?" rather
than from reading the diff — and the answer was hidden by the test tooling: `tsconfig`
scopes to `src`, so test doubles can drift from their interfaces with no failing check. The
guard against that is not a stricter compiler flag but the shape of the stub: making the
unused reads *throw* means a future analyzer that quietly starts reading session state gets a
loud failure instead of a plausible value. The same trick converts EOS-D7's one accepted
trade-off into an enforced invariant.

Finally, `git rev-parse --abbrev-ref HEAD` is the idiom everyone reaches for and it is
subtly wrong for provenance: it answers `"HEAD"` for a detached repository — a non-empty
string that every downstream validator accepts. When a contract's job is to record facts,
the honest primitive (`branch --show-current`, empty when there is no branch) beats the
familiar one.
