# Purpose

Deliver the **Projection** stage: render the human-readable `ReviewPackage` markdown
deterministically **from** the canonical `CandidateKnowledge[]` + `Session`.

> **Why does this task exist?** The candidates are structured data built for SPEC-004 to
> consume; a human reviewer needs prose. EOS-D4 settled that the package is a *derived view*
> of the canonical candidates, not a second source of truth — so the spec's "Review Package
> Builder" agent is, precisely, a **projector**. This task is the one place the workflow
> knows markdown, exactly as `createWikiRenderer` is the one place the Knowledge Platform
> knows the page format.

---

# Objective

Implement, under `src/end-of-session/projection/`:

- **`createReviewPackageProjector(): ReviewPackageProjector`** — with
  `project(candidates, session, generatedAt): ReviewPackage`.

It builds the frozen `ReviewPackage` shape:

| Field | Derivation |
| --- | --- |
| `sessionId` | `session.id` |
| `generatedAt` | The injected ISO-8601 instant (never an ambient clock read) |
| `summary` | Derived from candidates + session only — see the Projection Invariant |
| `candidateIds` | `candidates.map(c => c.id)`, in order — **references, not records** |
| `markdown` | The rendered body (non-empty) |

Built through `parseReviewPackage` — the single validator (EOS-004) — so the result is
validated and deep-frozen.

---

# Projection Invariant (derived, never authoritative)

The projector's single responsibility is **rendering a view of canonical data**. It:

- **derives everything it shows from `candidates` + `session`** — nothing else. In
  particular it must **not** read `KnowledgeExtraction`: the extraction is an *internal,
  unpersisted* contract (M3), so any field sourced from it could not be reproduced from the
  review store, breaking EOS-D4's "regenerable at any time from the candidates". This
  applies to `summary` above all — the tempting shortcut is `extraction.summary`, and it is
  the wrong one;
- **is a pure, deterministic function** — same candidates + session + `generatedAt` ⇒
  deep-equal package, every time. No clock, no I/O, no randomness, no ordering by anything
  but the canonical candidate order;
- **adds no information and drops none silently** — every candidate appears in
  `candidateIds` and in the markdown, in canonical order; the projector does not filter,
  rank, deduplicate, merge, or summarize *away* a candidate. Rendering is presentation, not
  editorial;
- **never persists** — writing `review-package.md` is the store's job (EOS-404), called by
  orchestration (EOS-406). The projector returns a value;
- **is never read back** — the markdown is opaque output. Nothing in AJ-OS parses it, and
  nothing may come to depend on its layout as a data format (EOS-D4).

# Scope

## Included

- `src/end-of-session/projection/` — `ReviewPackageProjector.ts` (the interface),
  `createReviewPackageProjector.ts`, `index.ts`.
- **Markdown rendering** of a session's review, deterministic and human-first. Proposed
  body (reviewer's call — see Design Notes):
  - a heading + session context line (branch, head commit, trigger, `generatedAt`);
  - a one-line synopsis (the `summary`);
  - one section per candidate, in canonical order: title, kind, rationale, body, tags,
    confidence when present, and its provenance (source paths / change ids) so a reviewer
    can trace a claim back to the diff;
  - the empty case: a valid package stating plainly that the session produced no
    candidates.
- Frozen handle; module-barrel export (`createReviewPackageProjector`); EOS-007 drift-guard
  manifest updated (+1 operation).
- Behaviour tests (pure — no filesystem, no stubs beyond fixtures built through the public
  contracts):
  - **determinism**: repeated `project` over identical inputs deep-equal;
  - **completeness**: `candidateIds` equals the candidates' ids **in order**; every
    candidate's title appears in the markdown;
  - **canonical-order preservation** and no filtering (N candidates in ⇒ N rendered);
  - **contract conformance**: `parseReviewPackage` accepts every produced package; the
    result is deep-frozen;
  - **empty candidates** ⇒ a valid package with empty `candidateIds` and non-empty
    `markdown` (the schema requires `markdown.min(1)`);
  - **immutability by divergence**: the input candidates are untouched.

## Not Included

- Persisting the package — **EOS-404** (the store operation) + **EOS-406** (the call).
- Building the `SessionReport` — **EOS-405** (a different artifact with a different
  audience: the report is machine/audit, the package is human).
- Deciding *which* candidates exist — **EOS-301** (frozen).
- Any HTML/JSON/alternate rendering, or a review UI — future projections of the same
  canonical candidates (EOS-D4 *Future Review*), not v1.
- Parsing markdown back into data — permanently out of scope (EOS-D4).

# Design Notes / Open Decisions (for the Planning Review)

- **`summary` must derive from candidates + session — CONSEQUENCE OF EOS-D4, needs
  confirming.** The `ReviewPackage` schema calls `summary` an "at-a-glance human synopsis",
  and `KnowledgeExtraction` carries a model-written `summary` that would read better. Using
  it would be a **boundary violation**: the extraction is internal and never persisted, so a
  package regenerated from the store could not reproduce that field — contradicting EOS-D4
  ("regenerable at any time from the candidates"). **Recommendation: derive `summary`
  structurally** (e.g. "N candidates from <branch> at <short-head>: <kind breakdown>").
  Cost: a duller synopsis than the model's prose. If the reviewer wants the model's summary
  in the package, the correct fix is to **carry it onto the canonical candidates or the
  report** — not to reach into the extraction from the projector. Flagged because it is the
  one place M5 could quietly undo EOS-D4.
- **`generatedAt` injected, not read.** The schema's docstring already fixes this ("No clock
  is embedded: `generatedAt` is injected at composition"). The projector therefore takes it
  as a parameter rather than a `now` dep — the stage stays a pure function of explicit
  inputs (the CB-021/Decision B precedent), and EOS-406 owns the single clock read per run.
- **Markdown layout is not a contract.** The body's exact shape is an implementation detail
  and should be reviewed as *readability*, not as an interface — nothing parses it. Tests
  must assert on *presence and order* (every candidate rendered, canonical order), never on
  exact prose, or they will ossify the layout into a de-facto contract.
- **Provenance in the body — RECOMMENDATION: include.** Rendering each candidate's
  `sourcePaths` lets a reviewer trace a proposal back to the diff, which is the whole point
  of the review gate (AJS-006 traceability). Cheap, derived from canonical data, no boundary
  cost.

# Worklog Summary

Delivered the Projection stage in a new `src/end-of-session/projection/` module — the
workflow's one markdown-aware stage, the direct analog of `createWikiRenderer`:

- **`ReviewPackageProjector`** (`ReviewPackageProjector.ts`) — the interface;
  `project(candidates, session, generatedAt): ReviewPackage`.
- **`createReviewPackageProjector()`** (`createReviewPackageProjector.ts`) — frozen handle,
  **no dependencies at all**. Renders: a session-context block (session id, branch, short
  head + clean/dirty, range analyzed, trigger, `generatedAt`); the derived synopsis; then one
  section per candidate in canonical order — title, kind (human-labelled), candidate id, tags
  and confidence *when present*, rationale, body, and the source paths it traces back to. The
  candidate list is introduced by a line stating candidates are **proposed, never approved**
  (AJS-006), so the artifact says what it is. The empty session renders a valid package that
  tells the reviewer plainly there is nothing to review — a successful capture of nothing, not
  an error.

**`summary` derives from candidates + session only** — the reviewer's explicit ratification.
The model's prose summary lives on the unpersisted `KnowledgeExtraction`; using it would have
made the package unreproducible from the review store and broken EOS-D4. The structural
synopsis (`"2 candidates proposed from this session on <branch> (1 handbook entry, 1 lesson
learned)."`) is duller and always true. A test asserts the regenerability property directly:
a package rebuilt from re-parsed candidates deep-equals the original.

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/review-package-projector.test.ts` (22 tests) covers contract conformance
(validated + deep-frozen), purity/determinism (repeat projections deep-equal; the instant
round-trips, proving no clock is read; inputs untouched), completeness and canonical order (ids
in order; every title/body/rationale rendered; order preserved *in the body*, not just the ids;
look-alike candidates not deduplicated), traceability, optional fields (rendered when present,
no `undefined` when absent), the empty session, and the summary's derivation. Public surface
**+1** (`createReviewPackageProjector`) — drift-guard manifest updated to **22**. Full platform
suite **590 tests / 52 files**, green.

Beyond the tests, the rendered package was **read as a human artifact** (populated and empty
cases rendered from the built module). Tests assert presence and order, which cannot tell you
whether the output is any good; the thing exists to be read, so it was read.

**Code review** (high effort, 8 angles) — six findings; four fixed:

1. **`summaryFor` was computed twice** — once for the `summary` field, once inside the body —
   so the field and the synopsis the reviewer reads agreed only by coincidence. Now derived
   once and passed in: they are the same sentence by construction.
2. **A newline in a candidate `title` broke its `###` heading.** `title` is model output and
   the contract permits any non-empty string, so a multi-line title would split the heading
   and spill into the body. Titles are now flattened to one line *in the heading only* —
   presentation, not interpretation; the canonical candidate keeps the original.
3. **A vacuous test assertion**: `expect(markdown.toLowerCase()).toContain("no")` for the
   empty case — "no" hides inside "knowledge", so it passed regardless. Replaced with the
   durable properties: the body contains its derived `summary`, and no `## Candidates`
   scaffolding is rendered. *Note: the first replacement pinned the exact sentence, which
   contradicts this task's own rule that prose must not be pinned — corrected before commit.*
4. **A confusing no-op** in a test (`{ provenance: { ...candidate(1).provenance } }` spread
   over an identical candidate) — removed.

Two accepted with notes: `SHORT_HEAD_LENGTH = 7` is now defined independently here and in the
Session factory (same meaning, different stages; hoisting one integer into a shared module
would buy coupling for very little), and the **factory-vs-function question** — this stage is
dependency-free yet a factory, while EOS-405's equally dependency-free `buildSessionReport` is
a plain function. The distinction is real but was undocumented: the projector is *injected* as
a replaceable stage (EOS-D4 anticipates further projections of the same candidates), while the
report builder is called directly like `collectChanges`. Now stated in the factory's docstring.

# Lessons Learned

The `summary` field was the milestone's one live opportunity to quietly undo EOS-D4, exactly
as this task's planning predicted. The model's prose summary is *right there* on the
extraction and reads far better than a structural count — and using it would have been
invisible until someone tried to regenerate a package from the review store and found the
field unreproducible. Naming the temptation in the task doc before implementing is what made
it a non-event; the acceptance criterion did the work the good intentions would not have.

Rendering someone else's text into a structural format is where a projector earns its
purity. The heading bug is small and would never have surfaced in a test that asserts
`toContain(title)` — the title *is* present, just not where it belongs. It took reading the
output to see it. Tests assert that a thing is there; only a human can tell whether it is
readable, which is why an artifact built for people should be looked at before it is called
done.

Two of the six findings were in the tests rather than the code, and one of my own fixes had to
be corrected for breaking this task's stated rule against pinning prose. Test code drifts from
its own principles as easily as product code, and "the suite is green" says nothing about
whether the assertions could ever fail.
