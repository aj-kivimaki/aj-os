# Purpose

Deliver the **Observability** stage: assemble the structured `SessionReport` — the workflow's
execution log — from the facts of one run.

> **Why does this task exist?** SPEC-003 §16 requires every run to record trigger, duration,
> files analyzed, candidates produced, errors, and result; §19 requires "Logs recorded"; and
> `run(context)` **returns** a `SessionReport`, so this is the workflow's return value, not a
> side artifact. The report contract is deliberately **shape-only** — its docstring states it
> "does not compute or cross-validate its fields: denormalized values (`durationMs`,
> `candidatesProduced.count`) are produced and kept consistent by the workflow (M4/M5), not
> derived or checked here". This task is the "M5" that docstring names: the one place those
> values are computed and the run's outcome is judged. Keeping it out of the orchestrator
> (EOS-406) keeps the composition root a sequencer rather than a place where policy accretes.

---

# Objective

Implement, under `src/end-of-session/report/`:

- **`buildSessionReport(facts: SessionRunFacts): SessionReport`** — a pure function.

Field derivation:

| Field | Derivation |
| --- | --- |
| `sessionId` | `session.id` |
| `trigger` | `session.trigger` |
| `startedAt` / `endedAt` | The **run's** window (injected instants — see Design Notes) |
| `durationMs` | `endedAt − startedAt` (computed here; the contract will not check it) |
| `analyzersRun` | The registry's analyzer ids |
| `filesAnalyzed` | Count of the `ChangeSet`'s changes |
| `candidatesProduced` | `{ count: candidates.length, ids: candidates.map(c => c.id) }` — kept consistent here |
| `errors` | `ChangeSet` `AnalyzerError`s + any stage error, mapped to `{ source, message, recoverable }` |
| `result` | The outcome policy below |
| `logEntry` | A deterministic one-line human summary |

Built through `parseSessionReport` — the single validator (EOS-004) — so the result is
validated and deep-frozen.

# Outcome Policy (the task's one real decision)

`result` is a **closed** enum — "producer-controlled and must be exact, so an unrecognized
value is a bug" (EOS-004). The mapping, aligned to SPEC-003 §15's recoverable/fatal split:

- **`completed`** — the run finished and `errors` is empty.
- **`partial`** — the run finished and produced candidates, but at least one **recoverable**
  error was recorded (the partial-collection model: an analyzer failed, the rest continued).
- **`failed`** — the run could not complete: a **fatal** error stopped it before candidates
  could be produced (SPEC-003 §15 — repository unavailable, invalid project, corrupted
  configuration; and a failed extraction, per EOS-406).

A run with zero candidates and no errors is **`completed`**, not `failed`: a session that
taught us nothing is a successful capture of nothing. This is worth stating because it is
the most likely place for a wrong assumption to harden.

# Report Builder Invariant (a pure projection over existing pipeline outputs)

> **Strengthened at the reviewer's requirement (AJ, 2026-07-16)** before implementation, and
> frozen. It joins the module's other frozen invariants — the Extractor Invariant (EOS-202),
> the Candidate Generation Invariant (EOS-301), the Persistence Invariant (EOS-302), and the
> Orchestrator Invariant (EOS-406).

**The builder is a pure projection over existing pipeline outputs.**

**It must not:**

- **perform new observations** — it learns nothing the pipeline did not already produce;
- **consult git** — no `GitPort`, no commands, no repository access of any kind;
- **inspect the filesystem** — no store, no reads, no writes, no paths;
- **regenerate information** — it never recomputes what a stage computed, re-runs a stage, or
  re-derives a value it was handed;
- **duplicate logic already owned by earlier stages** — collection counted the changes,
  generation produced the candidates, the factory identified the session. The builder repeats
  none of that work and holds no second opinion about any of it.

**It only assembles a `SessionReport` from the canonical data already available to it.**

The properties that follow:

- **Records execution results and diagnostics, never workflow decisions.** Approving,
  rejecting, ranking, or deduplicating candidates is SPEC-004's, and "never appears here"
  (EOS-004).
- **Keeps the denormalized fields consistent.** `count === ids.length` and `durationMs`
  matching the run window are *this function's* guarantees, because the contract explicitly
  declines to check them ("produced and kept consistent by the workflow (M4/M5), not derived
  or checked here"). This is arithmetic over given values, not observation.
- **Pure and deterministic** — same facts ⇒ deep-equal report, `logEntry` included. No clock:
  the run window is an input.
- **Never throws on a bad run** — a `failed` run still produces a valid report. The report is
  how a failure becomes observable; a builder that threw on failure would erase the only
  record of it.

The invariant is enforceable **by shape**: the builder's inputs are plain data, it constructs
nothing, imports no port, adapter, or store, and its only import beyond types is
`parseSessionReport`. If it ever needs a dependency, it has stopped being a projection.

# Scope

## Included

- `src/end-of-session/report/` — `SessionRunFacts.ts` (the input type), `buildSessionReport.ts`,
  `index.ts`.
- The `SessionRunFacts` input: `session`, `startedAt`/`endedAt` (run window), `analyzersRun`,
  `changeSet`, `candidates`, and any `stageErrors` (fatal or recoverable) the orchestrator
  collected.
- Error mapping: `AnalyzerError { analyzer, message, recoverable }` → `SessionReportError
  { source, message, recoverable }` — `source` generalizes `analyzer` (EOS-004's stated
  intent), so a stage error records the stage name.
- The outcome policy and a deterministic `logEntry` (the `GenerationReport.logEntry`
  precedent — one line, human-readable, suitable for `appendLog`).
- Module-barrel export (`buildSessionReport`); EOS-007 drift-guard manifest updated
  (+1 operation).
- Behaviour tests (pure — fixtures built through the public contracts):
  - each `result` case: clean ⇒ `completed`; recoverable analyzer error ⇒ `partial`; fatal
    stage error ⇒ `failed`; **zero candidates, no errors ⇒ `completed`**;
  - `count === ids.length`; `durationMs` matches the injected window; `filesAnalyzed`
    matches the change count; `analyzersRun` matches the registry;
  - determinism (same facts ⇒ deep-equal, `logEntry` included);
  - `parseSessionReport` accepts every produced report; output deep-frozen.

## Not Included

- Sequencing the run, catching stage errors, or deciding what is fatal at the *call site* —
  **EOS-406** (which supplies the facts; this function applies the policy to them).
- Persisting the report (**EOS-302** `saveReport`) or appending the log (**EOS-302**
  `appendLog`) — **EOS-406** makes both calls.
- Printing the report — **EOS-408** (CLI presentation).
- Notifying — **EOS-006**'s no-op port, called by EOS-406.
- Any review decision, threshold, or scoring — permanently SPEC-004's (EOS-004).

# Design Notes / Open Decisions (for the Planning Review)

- **The report times the *run*, not the *session* — CONFIRM.** `Session.startedAt`/`endedAt`
  describe the coding session (EOS-402); `SessionReport.startedAt`/`endedAt` describe the
  **workflow execution**, because §16's "Duration" is the run's duration and `durationMs` is
  a run metric. These are different windows on purpose, and the two contracts use the same
  field names — a genuine trap for a future reader. **Recommendation: confirm the split and
  state it explicitly in both docstrings.**
- **Timestamps injected, not read.** The builder takes the window as input; EOS-406 reads the
  clock (once at run start, once at run end). Keeps the stage pure and the report
  deterministic under a pinned clock (the CB-021 / `createCandidateGenerator` precedent).
- **A function, not a factory — RECOMMENDATION: function.** Every other stage is a
  `create*` factory because each owns injected dependencies; this one owns **none** (all
  facts are parameters). `buildSessionReport(facts)` is the honest shape — a factory would
  be ceremony around a pure function. `parseExtractionResponse` and `buildExtractionPrompt`
  set the precedent for bare functions in this module. *Reviewer's call; noted because it
  breaks the surface's `create*` rhythm.*
- **`filesAnalyzed` counts changes, not distinct paths.** The git analyzer emits one
  `SessionChange` per file, so the two coincide today. If a future analyzer emits several
  changes per file they would diverge — `changes.length` is the honest count of *what was
  analyzed*, and §16 asks for "files analyzed". **Recommendation: count changes now, revisit
  when a second analyzer exists** (noting it rather than pre-solving it).
- **Is `partial` reachable with a fatal error?** No: a fatal error means the run stopped, so
  the outcome is `failed`. `partial` requires the run to have finished. Stated so the policy
  is unambiguous at implementation.

# Worklog Summary

Delivered the Observability stage in a new `src/end-of-session/report/` module:

- **`SessionRunFacts`** (`SessionRunFacts.ts`) — the builder's entire input: the identified
  `session`, the run's `startedAt`/`endedAt` window, `analyzersRun`, and the optional
  `changeSet`, `candidates`, and `fatalError`. The optionals are what make a **failed** run
  reportable — a run that died in extraction has a `changeSet` but no `candidates`; one that
  died in session creation has neither. The shape admits the partial truth rather than
  demanding a complete one.
- **`buildSessionReport(facts)`** (`buildSessionReport.ts`) — a plain function, not a
  factory: it owns no dependency, and the run window is an input rather than a clock read.
  Assembles every §16 field and validates through `parseSessionReport`.

**The Report Builder Invariant holds by shape.** The module's only non-type import is
`parseSessionReport`; it imports no port, adapter, or store, constructs nothing, and reads no
clock. A grep confirms nothing else in the codebase computes `durationMs`, `filesAnalyzed`, or
`candidatesProduced` — no logic is duplicated from an earlier stage. The builder counts what
collection collected, names the candidates generation produced, and states an outcome. That
is all it does.

**The outcome policy**, as ratified: fatal ⇒ `failed`; otherwise any recorded error ⇒
`partial`; otherwise `completed`. **Zero candidates with no errors is `completed`** — a
session that taught us nothing is a successful capture of nothing.

*One clarification worth recording:* this task's planning text described `partial` as "the run
finished **and produced candidates**, but at least one recoverable error". Taken literally
that leaves a run with errors and *zero* candidates undefined. The **contract** (EOS-004,
M1-frozen and the higher authority) defines it without reference to candidates — "the run
finished, but one or more analyzers/stages contributed errors" — so the implementation
discriminates on **errors alone**, and a test pins the zero-candidates-with-an-error case as
`partial`. The task text was descriptive of the typical case, not a condition.

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/session-report-builder.test.ts` (23 tests) covers every outcome path
(including fatal-beats-recoverable and the two zero-candidate cases), the failed-run-still-
reports property, error mapping and ordering, the denormalized fields, the run-vs-session
window distinction, `analyzersRun`, purity/immutability, and `logEntry`. Public surface **+1**
(`buildSessionReport`) — drift-guard manifest updated to **23**. Full platform suite **619
tests / 53 files**, green. The report and its log line were also rendered and read as an
operator would see them.

**Code review** (high effort, 8 angles) — three findings, all fixed:

1. **`durationFor` was computed twice** — once for `durationMs`, once inside `logEntryFor` —
   so the structured field and the log line agreed only by coincidence. This is *the same
   defect EOS-403's review found* with `summaryFor`, one stage later. Fixed structurally
   rather than locally: `logEntryFor` now derives from the **assembled report fields**, so
   the summary is a projection of the record and cannot describe the run differently. A test
   asserts the log line agrees with every field it cites.
2. **`analyzersRun` had no direct assertion** — a §16 field pinned only incidentally, via the
   `analyzers=` count in `logEntry`. Two tests added.
3. A test helper typed its errors as `unknown[]`; given a named fixture type.

Separately, the **explicit test typecheck caught a real type error the project's own
`npm run typecheck` cannot see**: `facts({ candidates: undefined })` violates
`exactOptionalPropertyTypes` — under that flag, "absent" and "explicitly undefined" are
different types. Replaced with a `fatalFacts` helper that *omits* the key, which is also what
a failed run actually looks like. `tsconfig.json` scopes typechecking to `src`, so without
running `tsc` over the test file by hand this would have shipped as a latent error.

# Lessons Learned

The same defect surfaced in consecutive stages: a value derived twice — `summaryFor` in the
projector, `durationFor` here — so that a record and its human-readable summary agreed only
because two computations happened to match. The pattern is specific and worth naming: when a
stage emits both structured data *and* prose about that data, the prose must be derived **from
the assembled structure**, never recomputed from the same upstream inputs. Deriving it from
the fields makes divergence unrepresentable instead of merely absent.

The reviewer's invariant did real work before a line was written. "A pure projection over
existing pipeline outputs" is checkable by shape — no port, no adapter, no store, no clock —
so the honest test is the import list, and it can be verified in seconds by anyone. Invariants
that reduce to a grep are worth more than invariants that reduce to vigilance.

The planning text and the contract disagreed about `partial`, and the contract won. A task doc
describes intent at planning time; a frozen contract is the thing SPEC-004 will actually read.
When they conflict on a detail neither anticipated — a run with errors and no candidates —
the right move is to follow the contract, pin the case with a test, and record why, rather
than implement the looser prose and leave a gap.
