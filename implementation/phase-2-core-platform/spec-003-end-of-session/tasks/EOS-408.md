# Purpose

Expose the workflow to a human: `aj session end` — the v1 **manual trigger** (SPEC-003 §6),
and the only way anyone runs the capture pipeline.

> **Why does this task exist?** Everything upstream is a library. This is the product
> surface: the engineer finishes a session, types one command, and gets candidates in the
> review area plus a summary on stdout. It is modeled on `aj wiki build` — the platform's
> proven thin-CLI pattern.

---

# Objective

Implement:

- `src/cli/commands/session.ts` — **`sessionEndCommand(options)`**;
- registration in `src/cli/index.ts` — `aj session end`, with `--since <ref>` and
  `--notes <text>`.

The command's whole job:

1. load configuration (`new ConfigService().load()`);
2. ask the composition root for the assembled workflow (EOS-407);
3. obtain the `SessionContext` and call `workflow.run(context)`;
4. print the `SessionReport` summary;
5. map known, user-facing failures (`ConfigError`, `AIError`, `ReviewStoreError`) to a
   friendly message; let anything unexpected surface loudly.

---

# Thin CLI Invariant

The command's single responsibility is **entry and presentation**. It:

- **contains no workflow logic** — no wiring, no path composition, no stage construction, no
  git, no filesystem. All wiring lives in the composition root; all logic lives in the
  platform (the `wikiBuildCommand` docstring, verbatim in spirit);
- **performs no git** — "This command performs no git — orchestration and commits are
  handled elsewhere" applies identically here, and doubly so: SPEC-003 v1 commits nothing
  (ADR-002 §4, AJS-005 §7);
- **interprets nothing it prints** — it renders the report's fields; it does not recompute
  a count, re-derive a result, or summarize candidates. The report already says what
  happened (EOS-405);
- **owns option parsing only** — `--since` / `--notes` are passed through to the composition
  root untouched; the CLI does not resolve refs, validate them against git, or default them
  cleverly;
- **stays unchanged when the pipeline grows** — a new analyzer or notifier must not touch
  this file.

# Scope

## Included

- `sessionEndCommand(options: SessionEndOptions)` with
  `{ since?: string; notes?: string }`.
- Command registration under an `aj session` parent (`aj session end`), matching the
  `aj wiki build` nesting.
- A concise, human-readable report summary (the `printReport` precedent), showing:
  trigger, branch, result, duration, files analyzed, candidates produced, errors (listed
  when present), the review-package location, and `report.logEntry`. **Where the output
  landed matters most** — the command's entire point is to send a human to the review area.
- Known-error handling: `ConfigError` (no config), `AIError` (no API key / model failure),
  `ReviewStoreError` (bad `reviewPath`) ⇒ friendly message, no stack trace.
- A **non-`completed` result is reported honestly**: a `partial` run lists its recoverable
  errors; a `failed` run says so and points at the persisted report (EOS-406 returns a
  report rather than throwing).
- A "this may take a minute — the session is analyzed by the model" progress line before the
  run (the `wikiBuildCommand` precedent for a slow model-backed command).
- Tests: option pass-through and report rendering against a stubbed workflow — **no network,
  no vault**. (End-to-end is EOS-409.)

## Not Included

- Composition — **EOS-407** (the command calls it and nothing else).
- Sequencing, session creation, projection, reporting — **EOS-402..406**.
- Any git commit, push, or wiki build — permanently out of v1 scope.
- Additional triggers (git hook, scheduled, IDE, n8n) — deferred post-v1; they are
  `TriggerSource` implementations, not CLI flags.
- Interactive review, approval, or editing of candidates — **SPEC-004**. This command
  captures; it never promotes.
- `--project`, `--rebuild`-style flags, or output-format options — not in the frozen M5
  deliverable (`--since`, `--notes`).

# Design Notes / Open Decisions (for the Planning Review)

- **`aj session end` vs. a flat `aj session-end`.** The frozen plan says `aj session end`;
  `wiki build` establishes the nested shape. No decision needed — noted for symmetry.
- **`--notes <text>` vs. `--notes` opening an editor.** **Recommendation: `<text>`.** It maps
  straight onto the optional `SessionContext.sessionNotes`, keeps the command
  non-interactive (so it stays usable from a future git hook or n8n trigger unchanged), and
  an editor prompt is a feature nobody asked for. *Confirm.*
- **`--notes` reaches the model — resolved.** EOS-402 found that `sessionNotes` had no
  consumer, which would have made this flag collect the engineer's notes and silently discard
  them. **[EOS-D10](../decisions/EOS-D10-session-notes-into-extraction.md) is approved** and
  **EOS-410** routes the notes into the extraction prompt, so **`--notes` stays** and is
  meaningful. The CLI's job is unchanged: pass the text through untouched (it does not read,
  trim, or interpret it).
- **Print the review-package path — needs EOS-407's return.** The most useful line of output
  is where to go review. `store.locate()` gives the root; the session directory is
  `pending/<session-id>/`. **Recommendation: print
  `<root>/pending/<session-id>/review-package.md`**, composed from `locate()` + the report's
  `sessionId`. *Mild tension: the store owns the layout (EOS-D6), so the CLI composing this
  path duplicates layout knowledge. Alternative: a `locate(sessionId)` overload returning the
  session directory — a second EOS-D6 surface touch on top of EOS-404. **Recommendation:
  compose it in the CLI for v1** (display-only, never read back) and revisit if SPEC-004
  needs directory resolution. Reviewer's call.*
- **⚠️ `AIError` will not reach the `catch` — raised by EOS-407, decide here.** The planned
  known-error handling assumes a missing API key surfaces as an `AIError` the command can
  catch and print kindly (the `wikiBuildCommand` precedent). **It cannot.** `new AIClient()`
  never throws (the key resolves lazily), so composition succeeds; the `AIError` then fires
  inside extraction, where EOS-406's orchestrator catches it and converts it into a
  **persisted `failed` `SessionReport`**. The error never escapes `run`. Consequences: the
  friendly-message path is dead for the most common first-run mistake, and every keyless
  `aj session end` leaves a session directory in the vault recording a *configuration*
  problem as though it were a captured session. Options: (a) accept — the report is honest,
  `result: failed` + `exit 1` tells the user something is wrong, and the CLI can special-case
  the *reported* error's message; (b) have the CLI detect an extraction-sourced failure and
  print the friendly hint from `report.errors`; (c) pre-flight the key at composition — but
  that puts policy in the root, which EOS-407 forbids. **Recommendation: (b)** — the CLI owns
  presentation, and a `failed` report is exactly the data it needs to present well. Reviewer's
  call.
- **Exit code on `failed`.** A `failed` run returns a report rather than throwing (EOS-406).
  Should the CLI exit non-zero? **Recommendation: yes — `process.exitCode = 1` on
  `result === "failed"`**, so a future git hook or CI wrapper can detect it. `partial` stays
  zero (it succeeded; some analyzers didn't). Flagged: `wikiBuildCommand` sets no exit code,
  so this is a small, deliberate divergence. *Reviewer's call.*
- **Does the command print candidate titles?** **Recommendation: no** — counts plus the
  package path. The package *is* the human-readable projection (EOS-D4); reprinting its
  content in the terminal would make the CLI a second, undocumented projection with no
  owner.

# Worklog Summary

Delivered `aj session end` — the v1 manual trigger, and the first way a human can run the
capture pipeline:

- **`src/cli/commands/session.ts`** — `sessionEndCommand({ since?, notes? })`: load config →
  compose → `trigger.createContext()` → `workflow.run(context)` → print. Plus two **pure,
  exported** presentation functions, `formatSessionReport` and `aiConfigurationHelp`.
- **`src/cli/index.ts`** — `aj session end --since <ref> --notes <text>`, nested like
  `aj wiki build`.

**The Thin CLI Invariant holds**, verified by grep: no git, no filesystem (only
`node:path.join` to display a directory), no stage construction. The command imports the
composition root, `ConfigService`, and two error types — nothing else.

**AI configuration guidance, per the reviewer's ruling.** No pre-flight key check exists
anywhere: the workflow runs, extraction fails, and the failure is persisted in a
`SessionReport` like any other. The CLI then *reads the finished report* and, if an
**extraction-stage** error mentions `ANTHROPIC_API_KEY`, prints guidance — what to set, how
to re-run, and that nothing was lost. Detection is deliberately narrow on both axes: a
transport failure ("The AI request failed: 503") gets no key advice, and a `persistence`
error that merely quotes the variable is ignored. Both are pinned by tests.

**Exit codes.** `failed` ⇒ `exitCode = 1`; `partial` ⇒ `0` (the run succeeded, some analyzer
did not). Known errors (`ConfigError`, `ReviewStoreError`) print friendly and also exit `1` —
a small, deliberate extension of this task's text: the command did not complete, and a git
hook or CI wrapper should be able to tell.

**Testing shape (a deviation worth naming).** This task planned tests "against a stubbed
workflow". There is **no CLI test precedent** in the repo, and the command constructs its own
`ConfigService` and composition root, with no seam to inject a generator — so any test of the
command itself would have had to either use a real vault and a **real API key** (a network
call in the suite — unacceptable, and it would have written to the developer's own Handbook)
or bolt a test-only DI seam onto a CLI. Instead the command's *only real logic* — the
presentation decisions — was extracted into two pure functions and tested directly (13 tests,
no vault, no repo, no model), leaving the command as glue that EOS-409 exercises end to end.

**Validation.** `npm run typecheck` + `npm run build` clean; explicit test typecheck clean;
full platform suite **678 tests / 55 files**, green. Beyond the tests, the command was
**driven for real**: `aj session end --help` and `aj session --help` render correctly, and
both output paths (a successful capture and a keyless run) were rendered from a **real
composed workflow** over a temp vault and a fixture repo with a stubbed generator — never
against the developer's vault or the live API.

**Code review** (high effort, 8 angles) — three findings, two fixed:

1. **The `AIError` branch in the catch was unreachable**, and the comment beside it said so —
   code contradicting its own docstring. Removed (with the import), and the comment now
   explains *why* `AIError` is absent rather than defending against it. Same shape as
   EOS-406's unreachable branch.
2. **Two assertions pinned exact column alignment** (`"Files analyzed  : 2"`). Loosened to
   `/Files analyzed\s*: 2/`: the label and its value are the claim; the padding is
   presentation this stage may change freely.
3. **Accepted, re-recorded:** the CLI composes `pending/<session-id>` itself, duplicating
   layout the store owns (EOS-D6/EOS-D8). Pre-flagged in this task's Design Notes: the path
   is display-only and never read back, and the alternative is a second EOS-D6 surface touch.

## Raised: untracked files never reach capture (not this task's scope)

Driving the real command surfaced a **v1 capture gap**. The Session's `dirty` flag comes from
`git status --porcelain`, which sees untracked files — but the analyzer reads
`git diff --name-status -M HEAD`, which **does not**. Verified directly:

```
git status --porcelain     git diff --name-status -M HEAD
 M a.ts                    M   a.ts
A  staged.ts               A   staged.ts
?? untracked.md            (absent)
```

So a session that creates a brand-new file and ends before `git add` reports
`dirty: true` while the file is invisible to extraction — no finding can cite it, and no
candidate can come from it. New files are where new knowledge most often lives, which makes
this worth a decision rather than a footnote. It belongs to EOS-402's ratified range
semantics (`HEAD` = "uncommitted + staged" — an untracked file *is* uncommitted) and the
M2-frozen analyzer, so it is out of the CLI's remit. Flagged for the reviewer before EOS-409
pins acceptance behaviour around it.

---

# Lessons Learned

Driving the command taught what the tests could not. Thirteen green assertions said the
formatter worked; running a real composed workflow over a fixture repo showed
`Files analyzed : 1` when the session had touched two files — and that discrepancy is the
untracked-file gap, invisible to every unit test in the milestone because they all supply
their own `ChangeSet`. The gap had been there since M2. It took looking at real output over a
real repository to see it.

The reviewer's split — platform records, CLI explains — turned out to be what made the
missing-key case tractable at all. There is no exception to catch, because the workflow
correctly treats a failed extraction as data. Once that is accepted, the CLI's job is
obvious: read the report. The alternative (pre-flighting the key at composition) would have
been easier to write and would have put policy in the one place forbidden to hold it.

Finally, a test plan can be wrong about its own subject. This task asked for a stubbed
workflow, but the command has no seam for one and the repo has no CLI test precedent —
and forcing it would have meant either a network call in the suite or DI bolted onto an entry
point. Extracting the two pure decisions and testing those covers the same risk honestly.
When a plan's test strategy does not survive contact with the code, the strategy is what
should bend.
