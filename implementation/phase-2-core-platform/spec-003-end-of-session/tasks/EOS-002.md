# Purpose

Define the input request and the identified run.

> **Why does this task exist?** Every artifact the workflow produces is attributed
> to a session. Per EOS-D3, a session is a first-class concept with a stable
> identity, not an id derived from branch/timestamp. This task establishes that
> identity model before anything references it.

---

# Objective

Define two immutable, validated contracts:

- **`SessionContext`** — the workflow input (SPEC-003 §7): required `project`,
  `repository`, `branch`; optional `commitHash`, `commitMessage`, `sessionNotes`,
  `taskId`, `contextPackageRef`.
- **`Session`** — the identified run: `id` (stable, opaque), `startedAt`,
  `endedAt`, `trigger`, `gitState` (head commit, dirty flag, resolved range),
  `branch`.

---

# Scope

## Included

- Zod schemas + inferred types for `SessionContext` and `Session`
- A `TriggerKind` enum (lenient), with `manual` as the v1 value
- Deep-frozen, immutable structures and public exports via `contracts/`
- Documentation of the id-generation contract (stable, opaque, not derived)

## Not Included

- Session *creation* logic / id generation implementation (M5 composition;
  the contract only specifies the shape and invariants)
- Git access to populate `gitState` (M2)
- `ChangeSet`, candidate, package, or report contracts (later tasks)

# Implementation Notes

- Model `trigger` as a lenient enum (`.catch` to a safe default) so future
  trigger kinds don't break validation — the same tolerance the compiler's
  `entityTypeSchema` uses.
- Keep `SessionContext` (input) and `Session` (identified run) **separate** — do
  not conflate the request with the identified session (EOS-D3).
- Timestamps are ISO-8601 strings; do not embed a clock in the contract (clocks
  are injected at composition — the CB `now` precedent).
- The `id` field's contract: non-empty, opaque, stable for the life of the
  session; the contract does not prescribe the generator (UUID/ULID chosen at
  implementation), only the invariant.

---

# Validation

- Valid contexts/sessions parse; missing required fields and malformed timestamps
  are rejected; unknown `trigger` values fall back rather than throw.
- Structures are deep-frozen; mutation attempts fail.

# Worklog Summary

Defined the two session contracts in `src/end-of-session/contracts/session/`,
following the established contract pattern (Zod schema → `deepFreeze` → inferred
`DeepReadonly` types, `.strict()`, public export via the `contracts/` barrel):

- **`SessionContext`** (input request, §7): required `project`, `repository`,
  `branch`; optional `commitHash`, `commitMessage`, `sessionNotes`, `taskId`,
  `contextPackageRef`. Kept deliberately separate from `Session` (EOS-D3).
- **`Session`** (identified run, EOS-D3): stable **opaque** `id` (invariant
  documented: non-empty, opaque, not derived from branch/timestamp; generator
  chosen at M5), ISO-8601 `startedAt`/`endedAt`, lenient `trigger`, `gitState`
  (`head`, `dirty`, `range`), and `branch`.
- **`triggerKindSchema`** — lenient enum (`z.enum(TRIGGER_KINDS).catch("manual")`,
  the compiler `entityTypeSchema` precedent) so a not-yet-implemented trigger
  parses (forward-compat) rather than throwing.
- **`immutable.ts`** — module-internal `deepFreeze` + `DeepReadonly` shared by the
  module's contracts (not reused from context-builder, which would couple SPEC-003
  to SPEC-002's internal `package/types.ts` across an architecture boundary).

Contracts are re-exported through `contracts/index.ts` and the module entry point
`index.ts`; the EOS-001 scaffold test was repurposed into a public-surface test and
joined by `session-contracts.test.ts` (valid/invalid/immutable/lenient-trigger).

**Decisions.** `TRIGGER_KINDS` holds **only** `"manual"` (the v1 value): the
lenient `.catch` already tolerates future kinds, so pre-listing git-hook/scheduled/
ide/n8n would anticipate later milestones (implementation playbook: "do not
anticipate future milestones"). Anticipated kinds are documented in prose only and
added to the domain by the task that implements each `TriggerSource`. (This was a
self-review finding, corrected before commit.)

**Validation.** `npm run typecheck`, `npm run build` green; full suite
360 tests / 34 files (18 new). Code review clean after the `TRIGGER_KINDS` fix.

---

# Lessons Learned

A lenient enum (`.catch`) is the right tool for forward-compatibility *and* keeps
the contract honest about what v1 actually implements — listing unimplemented enum
members is speculative scope, not documentation. Note: `.catch` also makes a
*missing* `trigger` fall back to `manual` rather than erroring; that is an
intentional consequence of the requested leniency, recorded here so a later task
can revisit it if strict presence is ever wanted.
