# Purpose

Prove the **v1 vertical slice** end to end, and prove the SPEC-003 **acceptance criteria**
(§18/§19) against a real run over a fixture repo and a fixture vault.

> **Why does this task exist?** Every stage is tested in isolation, and every M5 task tests
> its own unit with stubs. None of that proves the *pipeline*. M4's planning deferred exactly
> this — "end-to-end integration is M5's acceptance test (the two M4 stages are wired
> together only in M5 composition)" — and M5's deliverables name it. This is also the only
> place the workflow's central safety claim is **proven rather than asserted**: canonical
> knowledge is untouched.

# Objective

Author permanent suites under `tests/end-of-session/`:

- **`session-workflow-integration.test.ts`** — a real composed run: real `createGitPort`
  against a **disposable fixture repo**, real `FilesystemReviewStore` against a **fixture
  vault** (temp dir), a **stub `TextGenerator`** (the one non-deterministic seam, pinned),
  and a pinned clock.
- **`session-acceptance.test.ts`** — the SPEC-003 §19 criteria, asserted one by one.

Validated **only through the public surface** (AJS-007 §6.1 Public-Surface Validation): the
composition root, `run`, and the contracts barrel. No internal file is imported.

# The Canonical-Unchanged Proof

SPEC-003's permanent promise — "must never modify canonical handbook content" (§17) — is
proven, not assumed:

- the fixture vault is seeded with real content in `foundation/`, `library/`, and `wiki/`;
- a **recursive snapshot** (path + content hash) is taken before the run and compared after;
- the assertion is **byte-identical trees**, not "no error was thrown". The store's
  construction guard (EOS-302) proves a canonical *destination* is refused; this proves a
  correctly-configured run *also* never reaches canonical space.

This is the strongest test in the milestone and the one a future contributor must not be
able to weaken accidentally.

# Scope

## Included

- **Fixture repo** (disposable, temp dir, real `git init` + commits — the EOS-103 precedent):
  a session with source, test, doc, and config changes so the change-kind classification and
  `filesAnalyzed` are exercised for real.
- **Fixture vault** (temp dir): `foundation/`, `library/`, `wiki/`, and a `knowledge-review/`
  review area configured via a literal `AjConfig`.
- **Stub `TextGenerator`**: returns a fixed, valid extraction response (including a fenced-
  JSON variant, since `parseExtractionResponse` strips fences) — the seam stays pinned so the
  run is deterministic.
- **Integration assertions**:
  - a full `run(context)` writes the complete session layout:
    `pending/<id>/{candidates/<id>.json, report.json, review-package.md, log.md}`;
  - the returned report matches what was persisted (deep-equal against `report.json`);
  - `candidatesProduced.ids` match the candidate filenames — the store, generator, and
    report agree on identity;
  - **the Candidate Generation Invariant survives composition**: `candidates.length ===`
    the stubbed extraction's findings count, order preserved;
  - **provenance is complete and real**: `provenance.sessionId === report.sessionId`;
    `commitHash` equals the fixture repo's actual HEAD (proving EOS-401's read is wired);
  - **determinism**: two runs with the same stub, clock, and repo produce deep-equal
    artifacts **except** the session id (EOS-D3: identity is not a pure function of inputs) —
    the one place that property is proven end-to-end;
  - **`--since <ref>`**: a run over a commit range analyzes exactly the range's files, and
    `gitState.range` records it;
  - **partial collection end-to-end**: a failing co-analyzer yields `result: "partial"`, its
    `AnalyzerError` in `report.errors`, and **candidates still produced** — the model that
    matters most and is currently only proven at the collection stage;
  - **no git write**: the fixture repo's HEAD, index, and working tree are unchanged after
    the run (the ADR-002 promise, proven);
  - **no wiki generation**: the vault's `wiki/` is untouched (covered by the canonical
    snapshot, called out separately because it is a distinct claim).
- **Acceptance suite**: the §19 table above, each criterion a named test.

## Not Included

- Unit-level behaviour of any stage — owned by EOS-401..408 and the frozen M1–M4 suites.
- Any **real model call** — the `TextGenerator` is stubbed. The suite must run offline, with
  no API key, in CI.
- The user's real vault or the real `aj-os` repo — every fixture is disposable and temp.
- SPEC-004 behaviour (reading/approving the store) — out of scope; this suite proves only
  that the store is *consumable*: the layout exists and every artifact round-trips through
  the published parsers.
- Performance/load testing — not a v1 concern.

# Design Notes / Open Decisions (for the Planning Review)

- **"Candidate wiki publications prepared" (§19) — RECOMMENDATION: assert pass-through, not
  production.** v1 treats `kind` as a **validated pass-through** (an M4 ratified decision):
  the model proposes the kind; the generator does not classify. So the workflow produces a
  `wiki-publication` candidate exactly when the extraction yields one. The honest test:
  stub an extraction containing a `wiki-publication` finding and assert it reaches the store
  as a `wiki-publication` candidate. Asserting the workflow *decides* to prepare wiki
  publications would test a capability v1 deliberately does not have. **Flagged because it is
  the §19 criterion most likely to be over-read** at the Freeze Review.
- **Determinism across runs, minus identity.** Two runs cannot be deep-equal: `Session.id` is
  freshly minted by design (EOS-D3), which cascades into candidate ids
  (`session:<id>:<n>`), paths, and provenance. **Recommendation: assert deep-equality after
  normalizing the session id**, and assert *separately* that two runs' ids differ. Both
  properties are real and neither is expressible without the other.
- **Suite split — RECOMMENDATION: two files.** Integration (mechanics: layout, wiring,
  ranges, partial collection) and acceptance (the §19 criteria, in the spec's own language)
  serve different readers. The acceptance suite should be legible to someone holding SPEC-003
  who has never read the code. *Reviewer's call.*
- **Does this task write product code?** **No** — tests only, mirroring CB-024 (SPEC-002's
  permanent behaviour-test task). If the suite reveals a defect, the fix lands in the owning
  task under the AJS-007 cycle, not here (§7.5 Regression Re-homing).
- **Runtime.** Real `git init` per fixture makes this the slowest suite in the repo. EOS-103
  already established the cost is acceptable; keep fixtures minimal (a handful of files,
  two commits) so the suite stays usable in the inner loop.

# Worklog Summary

Delivered the v1 vertical-slice proof — **three files, no product code**:

- **`session-fixtures.ts`** — shared test infrastructure: a disposable vault **seeded with
  real canonical content** (an empty vault would make "canonical unchanged" a vacuous
  claim), a disposable git repo whose session touches source, test, doc, and config files,
  the pinned stub generator, and the content-hash snapshot helpers.
- **`session-workflow-integration.test.ts`** (16 tests) — the mechanics.
- **`session-acceptance.test.ts`** (8 tests) — one test per SPEC-003 §19 criterion, named in
  the specification's own words.

Every stage is real — the actual git port against a real repo, the actual store against a
real vault, assembled by the actual composition root. Only the `TextGenerator` is stubbed,
and the stub returns its extraction **inside a fenced code block**, because that is what a
real model emits and `parseExtractionResponse` strips fences: the slice is proven against the
shape reality produces, not a tidied one. The suites run offline, with no API key.

**The canonical-unchanged proof is real, and I verified that it can fail.** A vacuous
guarantee is worse than none, so before trusting it I confirmed the snapshot detects a
**modified**, an **added**, and a **deleted** canonical file, and that the seeded vault
contains actual content (3 files). Only then is "byte-identical before and after" worth
anything.

**Also proven end to end:** the complete session layout; the returned report is byte-equal to
the persisted `report.json`; the store, generator, and report agree on candidate identity;
the Candidate Generation Invariant survives composition (2 findings ⇒ 2 candidates, in
order); provenance carries the fixture's **actual HEAD** (proof EOS-401's read reaches a
persisted candidate); `--since` measures a commit range; **partial collection** — a failing
co-analyzer yields `partial` with candidates still produced; **no git write** (HEAD, index,
and working tree byte-identical); no wiki generation; and every write lands under
`knowledge-review/`.

**Validation.** `npm run typecheck`, `npm run build`, explicit test typecheck all clean. Full
platform suite **702 tests / 58 files**, green — run **three consecutive times** to confirm
stability (see below).

**Code review** (high effort, 8 angles) — four findings, three fixed:

1. **A circular acceptance test.** "Session summary generated" built a `ReviewPackage` out of
   the markdown it had just read and then asserted on that package's markdown — which was the
   string it passed in. It also invented a placeholder `summary: "checked below"`. It could
   not fail for the reason it claimed. Rewritten to assert what the *run* produced: the
   report's log entry, and the package's synopsis derived from the candidates.
2. **An `as never` cast** past the `Session` contract in the direct-collection test —
   silenced the compiler in a suite whose entire point is proving the pipeline works on real
   data. Now built through `parseSession`, with the fixture's real HEAD.
3. **A vacuous assertion**: `expect(Object.keys(candidate)).not.toContain("decision")` — the
   contract is `.strict()`, so the key cannot exist on a parsed candidate under any change to
   this code. Removed, with a comment explaining why it is *not* there.
4. **Accepted:** the stub parses the session id out of the prompt (`/Session id: (\S+)/`),
   coupling it to `buildExtractionPrompt`'s wording. Unavoidable — the id is minted at
   runtime — and self-announcing: `createCandidateGenerator` throws on a session-id mismatch,
   so a reworded prompt fails loudly rather than silently.

## The flaky test I wrote (worth recording)

The determinism tests passed in isolation and **failed in the full suite**. My fault, not the
code's: each run built a *fresh* fixture repo, and git commit hashes incorporate the commit
timestamp — so two repos created in the same second produced identical HEADs, and two created
a second apart did not. The review package renders the short head, so the "byte-identical"
comparison broke.

The tests were comparing **two different repositories** and calling it determinism. Fixed by
running both over **one** repo and vault, with the reason recorded in the test. I then ran
the full suite three times to confirm the flake was gone rather than hiding.

## Acceptance behaviour left open, then closed (EOS-D11)

While EOS-D11 was under review this suite deliberately pinned **no** untracked-file
behaviour: the fixture staged everything, so nothing here depended on the reviewer's pending
decision. That held — when the FPCP was **approved**, EOS-411 extended the fixture (a file
never `git add`ed, plus an ignored one) and added the §19 scenario, and **no existing
assertion needed rewriting**; only the file counts moved (5 → 7). Writing the suite to be
indifferent to an open question is what made the amendment cheap.

# Lessons Learned

The best test in this suite is the one I checked could fail. "Canonical knowledge unchanged"
is SPEC-003's permanent promise, and a snapshot comparison would have passed just as happily
over two empty directories. Seeding the vault and then *deliberately tampering with it* to
watch the assertion fire is what turned it from a comforting green tick into a proof. A test
guarding an important claim deserves that treatment; the rest of the milestone's reviews kept
finding assertions that could not fail, and this one would have joined them.

My own determinism test was flaky for a reason worth remembering: it varied something it
meant to hold constant. Two fresh fixture repos are not "the same inputs" — git bakes the
clock into a commit hash — so the test only passed when both happened to land in the same
second. It survived the isolated run and died in the full suite, which is the classic
signature. Determinism tests have to be paranoid about what they hold fixed, because the
thing that varies is rarely the thing under test.

And the slice held. Ten tasks, four frozen-plan gaps, two FPCPs, and every stage built
against stubs — then the first real end-to-end run produced the right artifacts, with real
provenance, leaving canonical knowledge untouched. That is the argument for contract-first
work with the seams stubbed: composition surfaced ordering questions and one genuine capture
gap, but no stage had to be redesigned to meet another.
