# Purpose

Add the single configuration value the review area needs: **`AjConfig.handbook.reviewPath`**
(default `"knowledge-review"`), resolved relative to the handbook path — the exact
convention already used for `generatedWikiPath` (EOS-D2). This is what lets M5's composition
root resolve the Review Store destination to `<handbook>/<reviewPath>/pending/<session-id>/`
without any consumer re-defaulting it.

> **Why does this task exist?** EOS-D2 fixed the review location *and its config knob*:
> "add `AjConfig.handbook.reviewPath` (default `"knowledge-review"`), resolved relative to
> the handbook path — the same convention as `generatedWikiPath`." This is a **platform
> configuration** concern (`src/platform/config/`), distinct from the End-of-Session store
> and independently reviewable, so it is its own small task rather than being smuggled into
> the store (which takes a resolved `destination`, not config).

---

# Objective

Extend the configuration contract and loader, mirroring `generatedWikiPath` exactly:

- **`AjConfig.handbook.reviewPath: string`** — added to `src/platform/config/types.ts`
  with a doc-comment matching the `generatedWikiPath` style (relative to `handbook.path`;
  optional in the file, always resolved here so consumers never re-default it).
- **`ConfigService`** — a `DEFAULT_REVIEW_PATH = "knowledge-review"` constant and a
  `resolveReviewPath(parsed)` step (the mirror image of `resolveGeneratedWikiPath`):
  absent ⇒ default; present-but-not-a-non-empty-string ⇒ a clear `ConfigError` (a silent
  default would hide a typo). Threaded into the returned `handbook` block from `validateShape`.

No path *resolution to an absolute directory* and no `pending/` suffixing happen here —
that is M5 composition (`resolve(handbookPath, reviewPath)`), exactly as
`createKnowledgePipeline` resolves the wiki destination. This task only makes `reviewPath`
a first-class, defaulted, validated config field.

# Scope

## Included

- `src/platform/config/types.ts` — the `reviewPath` field + doc-comment.
- `src/platform/config/ConfigService.ts` — `DEFAULT_REVIEW_PATH`, `resolveReviewPath`,
  and its inclusion in the validated `handbook` block.
- **Config tests** (extend the existing config suite, the `generatedWikiPath` precedent):
  - absent ⇒ defaults to `"knowledge-review"`;
  - a valid string ⇒ carried through;
  - present-but-empty / non-string ⇒ `ConfigError`;
  - `reviewPath` does not disturb `path` / `generatedWikiPath` resolution.
- Update the config example / `aj.config.json` documentation only if a default currently
  surfaces there (otherwise no doc change — the field is optional).

## Not Included

- The Review Store or destination resolution — **EOS-302** (store takes `destination`) and
  **M5** (composition resolves `<handbook>/<reviewPath>` + `pending/` + `mkdir -p`).
- Candidate generation — **EOS-301**.
- Any new top-level config section or unrelated config fields (add only what M4 needs —
  the `types.ts` "added by the milestone that needs them" rule).

# Design Notes

- **Strictly parallel to `generatedWikiPath`.** Same optionality, same default-when-absent,
  same "present-but-invalid is a `ConfigError`, not a silent default" rule, same relative-to-
  `handbook.path` semantics. This keeps configuration coherent and the diff small and obvious.
- **No resolution/side effects here.** The loader returns a validated string; turning it into
  an absolute, existing directory (and appending `pending/`) is composition's job (M5), which
  already owns `mkdir -p` for such destinations.
- **No decision record expected.** This implements the already-Accepted EOS-D2 verbatim; there
  is no new architectural choice. (Noted for the Planning Review only for completeness.)
