# Purpose

Deliver the workflow's **single public entry point** ÔÇö `run(context)` ÔÇö by sequencing the
stages M1ÔÇôM5 built into one end-to-end capture run.

> **Why does this task exist?** Every stage exists and none of them are connected: there is
> no code path today from a `SessionContext` to a `SessionReport`. This task is that path.
> It is also where the module's promised entry point finally appears ÔÇö the module barrel has
> stated since EOS-001 that "this module's single public entry point will be `run(context)`
> ÔÇŽ wired at the composition root in Milestone M5".

# Orchestrator Invariant

> **Frozen at the M5 Planning Review (reviewer: AJ, 2026-07-16)**, at the reviewer's
> requirement, as a condition of the M5 Planning Freeze. It joins the module's other frozen
> invariants ÔÇö the Extractor Invariant (EOS-202), the Candidate Generation Invariant
> (EOS-301), and the Persistence Invariant (EOS-302) ÔÇö and is binding on implementation and
> review.

**The orchestrator owns sequencing only.**

**It may:**

- **invoke stages** ÔÇö call each injected stage's operation, once, in the documented order;
- **propagate results** ÔÇö pass a stage's output to the next stage **unmodified**;
- **coordinate execution** ÔÇö await stages, read the injected clock to open and close the run
  window, catch a stage failure and route it to the outcome path (EOS-405 decides what the
  outcome *is*), and hand the assembled facts to the report builder.

**It must not:**

- **perform transformations** ÔÇö it never maps, reshapes, enriches, filters, reorders,
  re-times, or serializes any contract in flight. What a stage returns is what the next stage
  receives (the `createContextBuilder` precedent: "never inspects, modifies, filters,
  reorders, deduplicates, enriches, or re-timestamps");
- **duplicate stage logic** ÔÇö it never recomputes what a stage computed, re-counts what
  EOS-405 counts, re-derives a result, re-renders a package, or re-validates a contract a
  stage already validated. Exactly one component owns each computation;
- **introduce business rules** ÔÇö no classification, deduplication, ranking, thresholding,
  scoring, retry policy, or fallback. If a rule wants to live here, **a stage is missing** ÔÇö
  stop and raise it (AJS-007 *Frozen Plan Change Proposal*) rather than absorbing it;
- **bypass the existing adapters** ÔÇö every effect goes through an injected port or stage. No
  direct filesystem write, no path composition, no git call, no config read, no ambient clock,
  no reaching around the store into the review directory. It imports **no** concrete
  implementation; EOS-407 supplies them all.

**Consequences that follow, and are testable:**

- **Stateless** ÔÇö `run` is self-contained; two runs share nothing.
- **No side effects of its own** ÔÇö **no git write, no wiki generation** ever enters the
  pipeline (ADR-002 ┬ž4, AJS-005 ┬ž7).
- **Seam-additive** ÔÇö registering a second analyzer, trigger, or notifier requires **zero**
  edits here. That property is the point of the seams, and this task is where it is proven or
  lost.

The invariant is enforced **by shape wherever possible** (the orchestrator holds no
collaborator it did not receive, and imports no concrete module) and by review otherwise ÔÇö
the same approach that made the Extractor and Persistence Invariants need no enforcement
code.

# Error Handling (SPEC-003 ┬ž15)

The orchestrator is the **only** place stage failures become outcomes:

- **Recoverable** ÔÇö analyzer failures already arrive as `AnalyzerError` data inside the
  `ChangeSet` (EOS-101's partial-collection model). The run continues; they flow to the
  report and yield `partial`.
- **Fatal** ÔÇö a failure that stops the run: session creation (repository unavailable ÔÇö the
  git reads of EOS-401), extraction failure (`ExtractionError` or a transport rejection),
  or candidate generation. The orchestrator catches it, records it as a fatal
  `stageError`, and **still produces, persists, and returns a `failed` `SessionReport`**
  (see Design Notes).
- **Infrastructure** ÔÇö if the store itself cannot be written, `run` rejects: there is
  nowhere to record the failure, so the caller (EOS-408) surfaces it.

# Scope

## Included

- `src/end-of-session/workflow/` ÔÇö `EndOfSessionWorkflow.ts`, `createSessionWorkflow.ts`,
  `index.ts`.
- The full sequence above; the run window (`now()` at open and close) and the single
  `generatedAt` read for the projection.
- Assembling `SessionRunFacts` for EOS-405 (`analyzersRun` from the registry;
  `changeSet`/`candidates`/`stageErrors` from the run).
- Fatal-error capture Ôćĺ `failed` report Ôćĺ persist Ôćĺ return.
- Frozen handle; module-barrel export (`createSessionWorkflow`, `EndOfSessionWorkflow`);
  EOS-007 drift-guard manifest updated (+1 operation).
- Behaviour tests with **all stages stubbed** (no filesystem, no git, no model ÔÇö the unit
  level; the real wiring is EOS-409):
  - the happy path calls every stage **once, in the documented order**, and returns the
    report (spy-recorded call order ÔÇö the one place order is a guarantee);
  - candidates are saved **before** the package is rendered (EOS-D4 ordering);
  - a recoverable analyzer error yields `partial` and the run completes;
  - a fatal extraction failure yields a persisted `failed` report, **not** a throw;
  - a store failure rejects;
  - `notify` receives the same report `run` returns;
  - **seam-additivity**: adding a second analyzer to the registry changes the output and
    requires no change here;
  - with every stage stubbed and the clock pinned, two runs are deep-equal.

- Passing `context.sessionNotes` to `extract` (**EOS-D10**, approved; implemented by
  **EOS-410**, which runs before this task). This is *propagation of an input*, not a
  transformation ÔÇö the Orchestrator Invariant is untouched: the orchestrator hands the notes
  over unread, exactly as it hands every other value between stages. It is also the only use
  this task makes of the `context` beyond passing it to the Session factory.

## Not Included

- Building concrete stages, reading `AjConfig`, resolving the destination, or constructing
  the store/git port/`AIClient` ÔÇö **EOS-407** (the composition root). This task injects
  everything and knows no concrete implementation.
- Rendering the notes into the prompt ÔÇö **EOS-410** (the orchestrator passes a value; the
  extraction stage decides what a prompt looks like).
- The stages' internals ÔÇö all frozen or owned by EOS-401..405.
- CLI parsing/printing ÔÇö **EOS-408**.
- Real end-to-end integration ÔÇö **EOS-409**.
- Git commit / wiki generation ÔÇö permanently out of the capture pipeline (v1).

# Design Notes / Open Decisions (for the Planning Review)

- **A fatal stage failure produces a `failed` report rather than throwing ÔÇö RECOMMENDATION,
  needs ratification.** The extractor's docstring explicitly defers this: "how a failed
  extraction maps to the `SessionReport` is orchestration's concern (M5)". Options:
  - **(a) Catch, build a `failed` report, persist it, return it (recommended).** SPEC-003
    ┬ž19 requires "Logs recorded", and a failure is exactly when a log earns its keep. `run`'s
    signature already promises a `SessionReport`, and the contract has a `failed` result for
    this. The CLI prints the failure from the report (EOS-408).
    Cost: `run` rarely rejects, so callers must read `result` ÔÇö stated plainly in the
    docstring and asserted in tests.
  - **(b) Let it propagate; no report on failure.** Simpler, and a rejected promise is hard
    to ignore. Cost: the run that most needs a record leaves none, and `failed` becomes dead
    code in a frozen contract.
  - **Recommendation: (a)**, with rejection reserved for the infrastructure case (the store
    is unwritable ÔÇö nowhere to record). Flagged because it defines what `run` promises.
- **Orchestrator separate from the composition root ÔÇö RECOMMENDATION: separate (this task
  vs. EOS-407).** MILESTONES names one factory, `createEndOfSessionWorkflow(config, deps)` Ôćĺ
  `{ workflow, store }`, which implies a *workflow* being composed. Splitting sequencing
  (here, fully injected, unit-testable with stubs and no I/O) from assembly (EOS-407: config,
  paths, real adapters) mirrors `createWikiGenerator` + `createKnowledgePipeline` exactly ÔÇö
  the precedent MILESTONES itself cites. The alternative (sequence inside the composition
  root, the `createContextBuilder` shape) would make the orchestrator untestable without a
  vault and a repo. *Names the frozen plan does not fix: `createSessionWorkflow` for the
  engine ÔÇö reviewer's call.*
- **`run` takes a `SessionContext`; the `TriggerSource` sits upstream.** The frozen entry
  point is `run(context)`, yet *Pipeline Ownership* lists `TriggerSource` under the workflow.
  Resolution: the workflow **owns the trigger's `kind`** (stamped onto the `Session` via
  EOS-402) but never invokes it; producing the context is upstream. **Who invokes the
  trigger is EOS-407's open decision** ÔÇö see that task.
- **Two clock reads or one?** The run window needs open and close instants; the projection
  needs `generatedAt`. **Recommendation: `now()` at run start, `now()` for the projection,
  `now()` at run end** ÔÇö three reads, each recording a distinct real moment, all through the
  injected clock so a pinned clock still makes runs deep-equal. (Reusing one instant for all
  three would make `durationMs` a constant zero.)
