# Purpose

Deliver the workflow's **single public entry point** — `run(context)` — by sequencing the
stages M1–M5 built into one end-to-end capture run.

> **Why does this task exist?** Every stage exists and none of them are connected: there is
> no code path today from a `SessionContext` to a `SessionReport`. This task is that path.
> It is also where the module's promised entry point finally appears — the module barrel has
> stated since EOS-001 that "this module's single public entry point will be `run(context)`
> … wired at the composition root in Milestone M5".

---

# Objective

Implement, under `src/end-of-session/workflow/`:

- **`EndOfSessionWorkflow`** — the interface: `run(context: SessionContext): Promise<SessionReport>`.
- **`createSessionWorkflow(deps): EndOfSessionWorkflow`** — the orchestrator over **injected
  stages only**: `sessionFactory`, `registry`, `extractor`, `candidateGenerator`, `store`,
  `projector`, `notifier`, `now`.

The sequence, exactly the frozen pipeline:

```text
run(context)
  │  now() ─────────────────────────────────► run window opens
  ├─ sessionFactory.create(context, …)       → Session          (EOS-402)
  ├─ collectChanges(registry, session)       → ChangeSet         (EOS-101)
  ├─ extractor.extract(changeSet, context.sessionNotes)
  │                                          → KnowledgeExtraction (EOS-202/EOS-410) ◄ non-deterministic seam
  ├─ candidateGenerator.generate(extraction, session) → CandidateKnowledge[] (EOS-301)
  ├─ store.saveCandidates(session.id, candidates)      (EOS-302)  ◄ canonical persisted first
  ├─ projector.project(candidates, session, now())     → ReviewPackage (EOS-403)
  ├─ store.saveReviewPackage(session.id, pkg)          (EOS-404)
  │  now() ─────────────────────────────────► run window closes
  ├─ buildSessionReport(facts)               → SessionReport     (EOS-405)
  ├─ store.saveReport(session.id, report)              (EOS-302)
  ├─ store.appendLog(session.id, report.logEntry)      (EOS-302)
  └─ notifier.notify(report)                           (EOS-006)
  → report
```

Canonical candidates are persisted **before** the projection is rendered — the ordering
EOS-D4 requires ("candidate generation (M4) produces and persists the canonical candidates
*before* the projection (M5) renders the package").

# Orchestrator Invariant

> **Frozen at the M5 Planning Review (reviewer: AJ, 2026-07-16)**, at the reviewer's
> requirement, as a condition of the M5 Planning Freeze. It joins the module's other frozen
> invariants — the Extractor Invariant (EOS-202), the Candidate Generation Invariant
> (EOS-301), and the Persistence Invariant (EOS-302) — and is binding on implementation and
> review.

**The orchestrator owns sequencing only.**

**It may:**

- **invoke stages** — call each injected stage's operation, once, in the documented order;
- **propagate results** — pass a stage's output to the next stage **unmodified**;
- **coordinate execution** — await stages, read the injected clock to open and close the run
  window, catch a stage failure and route it to the outcome path (EOS-405 decides what the
  outcome *is*), and hand the assembled facts to the report builder.

**It must not:**

- **perform transformations** — it never maps, reshapes, enriches, filters, reorders,
  re-times, or serializes any contract in flight. What a stage returns is what the next stage
  receives (the `createContextBuilder` precedent: "never inspects, modifies, filters,
  reorders, deduplicates, enriches, or re-timestamps");
- **duplicate stage logic** — it never recomputes what a stage computed, re-counts what
  EOS-405 counts, re-derives a result, re-renders a package, or re-validates a contract a
  stage already validated. Exactly one component owns each computation;
- **introduce business rules** — no classification, deduplication, ranking, thresholding,
  scoring, retry policy, or fallback. If a rule wants to live here, **a stage is missing** —
  stop and raise it (AJS-007 *Frozen Plan Change Proposal*) rather than absorbing it;
- **bypass the existing adapters** — every effect goes through an injected port or stage. No
  direct filesystem write, no path composition, no git call, no config read, no ambient clock,
  no reaching around the store into the review directory. It imports **no** concrete
  implementation; EOS-407 supplies them all.

**Consequences that follow, and are testable:**

- **Stateless** — `run` is self-contained; two runs share nothing.
- **No side effects of its own** — **no git write, no wiki generation** ever enters the
  pipeline (ADR-002 §4, AJS-005 §7).
- **Seam-additive** — registering a second analyzer, trigger, or notifier requires **zero**
  edits here. That property is the point of the seams, and this task is where it is proven or
  lost.

The invariant is enforced **by shape wherever possible** (the orchestrator holds no
collaborator it did not receive, and imports no concrete module) and by review otherwise —
the same approach that made the Extractor and Persistence Invariants need no enforcement
code.

# Error Handling (SPEC-003 §15)

The orchestrator is the **only** place stage failures become outcomes:

- **Recoverable** — analyzer failures already arrive as `AnalyzerError` data inside the
  `ChangeSet` (EOS-101's partial-collection model). The run continues; they flow to the
  report and yield `partial`.
- **Fatal** — a failure that stops the run: session creation (repository unavailable — the
  git reads of EOS-401), extraction failure (`ExtractionError` or a transport rejection),
  or candidate generation. The orchestrator catches it, records it as a fatal
  `stageError`, and **still produces, persists, and returns a `failed` `SessionReport`**
  (see Design Notes).
- **Infrastructure** — if the store itself cannot be written, `run` rejects: there is
  nowhere to record the failure, so the caller (EOS-408) surfaces it.

# Scope

## Included

- `src/end-of-session/workflow/` — `EndOfSessionWorkflow.ts`, `createSessionWorkflow.ts`,
  `index.ts`.
- The full sequence above; the run window (`now()` at open and close) and the single
  `generatedAt` read for the projection.
- Assembling `SessionRunFacts` for EOS-405 (`analyzersRun` from the registry;
  `changeSet`/`candidates`/`stageErrors` from the run).
- Fatal-error capture → `failed` report → persist → return.
- Frozen handle; module-barrel export (`createSessionWorkflow`, `EndOfSessionWorkflow`);
  EOS-007 drift-guard manifest updated (+1 operation).
- Behaviour tests with **all stages stubbed** (no filesystem, no git, no model — the unit
  level; the real wiring is EOS-409):
  - the happy path calls every stage **once, in the documented order**, and returns the
    report (spy-recorded call order — the one place order is a guarantee);
  - candidates are saved **before** the package is rendered (EOS-D4 ordering);
  - a recoverable analyzer error yields `partial` and the run completes;
  - a fatal extraction failure yields a persisted `failed` report, **not** a throw;
  - a store failure rejects;
  - `notify` receives the same report `run` returns;
  - **seam-additivity**: adding a second analyzer to the registry changes the output and
    requires no change here;
  - with every stage stubbed and the clock pinned, two runs are deep-equal.

- Passing `context.sessionNotes` to `extract` (**EOS-D10**, approved; implemented by
  **EOS-410**, which runs before this task). This is *propagation of an input*, not a
  transformation — the Orchestrator Invariant is untouched: the orchestrator hands the notes
  over unread, exactly as it hands every other value between stages. It is also the only use
  this task makes of the `context` beyond passing it to the Session factory.

## Not Included

- Building concrete stages, reading `AjConfig`, resolving the destination, or constructing
  the store/git port/`AIClient` — **EOS-407** (the composition root). This task injects
  everything and knows no concrete implementation.
- Rendering the notes into the prompt — **EOS-410** (the orchestrator passes a value; the
  extraction stage decides what a prompt looks like).
- The stages' internals — all frozen or owned by EOS-401..405.
- CLI parsing/printing — **EOS-408**.
- Real end-to-end integration — **EOS-409**.
- Git commit / wiki generation — permanently out of the capture pipeline (v1).

# Design Notes / Open Decisions (for the Planning Review)

- **A fatal stage failure produces a `failed` report rather than throwing — RECOMMENDATION,
  needs ratification.** The extractor's docstring explicitly defers this: "how a failed
  extraction maps to the `SessionReport` is orchestration's concern (M5)". Options:
  - **(a) Catch, build a `failed` report, persist it, return it (recommended).** SPEC-003
    §19 requires "Logs recorded", and a failure is exactly when a log earns its keep. `run`'s
    signature already promises a `SessionReport`, and the contract has a `failed` result for
    this. The CLI prints the failure from the report (EOS-408).
    Cost: `run` rarely rejects, so callers must read `result` — stated plainly in the
    docstring and asserted in tests.
  - **(b) Let it propagate; no report on failure.** Simpler, and a rejected promise is hard
    to ignore. Cost: the run that most needs a record leaves none, and `failed` becomes dead
    code in a frozen contract.
  - **Recommendation: (a)**, with rejection reserved for the infrastructure case (the store
    is unwritable — nowhere to record). Flagged because it defines what `run` promises.
- **Orchestrator separate from the composition root — RECOMMENDATION: separate (this task
  vs. EOS-407).** MILESTONES names one factory, `createEndOfSessionWorkflow(config, deps)` →
  `{ workflow, store }`, which implies a *workflow* being composed. Splitting sequencing
  (here, fully injected, unit-testable with stubs and no I/O) from assembly (EOS-407: config,
  paths, real adapters) mirrors `createWikiGenerator` + `createKnowledgePipeline` exactly —
  the precedent MILESTONES itself cites. The alternative (sequence inside the composition
  root, the `createContextBuilder` shape) would make the orchestrator untestable without a
  vault and a repo. *Names the frozen plan does not fix: `createSessionWorkflow` for the
  engine — reviewer's call.*
- **`run` takes a `SessionContext`; the `TriggerSource` sits upstream.** The frozen entry
  point is `run(context)`, yet *Pipeline Ownership* lists `TriggerSource` under the workflow.
  Resolution: the workflow **owns the trigger's `kind`** (stamped onto the `Session` via
  EOS-402) but never invokes it; producing the context is upstream. **Who invokes the
  trigger is EOS-407's open decision** — see that task.
- **Two clock reads or one?** The run window needs open and close instants; the projection
  needs `generatedAt`. **Recommendation: `now()` at run start, `now()` for the projection,
  `now()` at run end** — three reads, each recording a distinct real moment, all through the
  injected clock so a pinned clock still makes runs deep-equal. (Reusing one instant for all
  three would make `durationMs` a constant zero.)

# Worklog Summary

Delivered the orchestrator in a new `src/end-of-session/workflow/` module — the module's
promised public entry point, first named in EOS-001 and unimplemented until now:

- **`EndOfSessionWorkflow`** — `run(context): Promise<SessionReport>`.
- **`createSessionWorkflow(deps)`** — frozen handle, stateless across runs, every
  collaborator injected.

**The Orchestrator Invariant holds by shape.** The module has exactly **two** value imports —
`collectChanges` and `buildSessionReport`, both pure stage functions — and everything else is
a *type*. No port, no adapter, no store, no `AIClient`, no `node:` module, no `process`, no
config, and no clock beyond the injected `now`. With comments stripped, `run` is a flat
sequence of delegating calls: session → collection → extraction → generation →
saveCandidates → projection → saveReviewPackage → report → saveReport → appendLog → notify.
Nothing sits between the stages except the plumbing that carries one stage's output to the
next, unmodified.

**Failure attribution without policy.** A small `inStage(source, op)` helper tags any stage
rejection with the stage's name, because `SessionReportError.source` needs it and the
orchestrator is the only component that knows the sequence. It adds attribution, not
behaviour — no retry, no fallback, no judgement about what a failure *means* (EOS-405 owns
that).

**Validation.** `npm run typecheck` + `npm run build` clean; new suite
`tests/end-of-session/session-workflow.test.ts` (20 tests, every stage stubbed — no
filesystem, git, or model) covers the frozen call order via a shared trace, the EOS-D4
ordering, propagation without transformation, notes pass-through (present and absent), all
three outcome paths, per-stage attribution, the two rejection cases, notifier-failure
surfacing, determinism, statelessness, and seam-additivity. Public surface **+1**; drift-guard
manifest **24**. Full platform suite **651 tests / 54 files**, green.

**Reviewer-requested stub-drift check** (the EOS-401 tooling gap — `tsconfig` scopes to
`src`, so `npm run typecheck` never sees `tests/`). Ran `tsc` explicitly, strict +
`exactOptionalPropertyTypes`, over **every** End-of-Session suite:

- **EOS-406's orchestration doubles are clean** — all seven stubs (`SessionFactory`,
  `KnowledgeExtractor`, `CandidateGenerator`, `ReviewStore`, `ReviewPackageProjector`,
  `NotificationPort`, `Analyzer`) typecheck. This was the first task to stub `ReviewStore`,
  and adding `saveReviewPackage` in EOS-404 caused no drift.
- **Two pre-existing errors surfaced, both confirmed on HEAD** by stashing this task's work
  and re-running: `collection-execution.test.ts` (TS2345, indexed access) and
  `knowledge-extractor.test.ts:88` (TS2379, `stubGenerator`'s `options`). Neither is touched
  by this task; both are invisible to the project's own typecheck. Raised, not fixed —
  consistent with the reviewer's rulings on frozen-milestone internals.

**Code review** (high effort, 8 angles) — four findings, three fixed:

1. **The notifier-failure behaviour was documented but untested.** The code commits to a real
   decision (a notifier failure surfaces rather than being swallowed, because swallowing is a
   retry policy the orchestrator must not own) — and nothing verified it. Someone wrapping
   `notify` in a try/catch "to be safe" would have passed every test while the orchestrator
   silently acquired the policy the invariant forbids. Test added, asserting both the
   rejection *and* that the report was durable first.
2. **An unreachable branch**: `toFatalError`'s non-`StageFailure` fallback could never run,
   since every statement in the try is wrapped by `inStage`. Replaced with an explicit
   rethrow, which keeps the attribution honest if a future edit adds an unwrapped statement
   rather than mislabelling it `"workflow"`.
3. A test used `CONTEXT.sessionNotes ?? ""`, whose fallback was dead and which obscured what
   was being asserted — replaced with a named `NOTES` constant.
4. `describeFailure` duplicates the identical private helper in `collectChanges` (four
   lines). **Not fixed**: sharing it would widen a frozen M2 module's surface, which the
   reviewer has twice ruled out for this milestone. Recorded as bounded duplication.

## Two deviations from this task's frozen text (for the reviewer)

1. **Session-creation failure rejects; it does not produce a `failed` report.** This task's
   Error Handling section lists session creation among the fatals that "still produce,
   persist, and return a `failed` `SessionReport`". That is **impossible**: without a
   `Session` there is no `sessionId`, and both `SessionRunFacts.session` and
   `SessionReport.sessionId` (`.min(1)`) are required — there is also no
   `pending/<session-id>/` to persist it under. A failure to *identify* the run can only be
   surfaced, exactly like an unwritable store (which this task already routes to a
   rejection). Implemented as a rejection, documented on the entry point, and covered by a
   test.
2. **"Persistence" appears at two points, not one.** The reviewer's composition lists
   Persistence once, after Session Report. The implementation persists the **canonical
   candidates immediately after generation** — before the projector renders a view of them —
   because **EOS-D4** requires it ("candidate generation … produces and persists the
   canonical candidates *before* the projection … renders the package") and this task's
   frozen sequence diagram encodes it. The durability argument is the point: if the projector
   fails, the artifact SPEC-004 consumes is already safe, and the run degrades to a `failed`
   report *with its candidates intact* (a test pins exactly this). Read as a composition of
   stages the order is unchanged; only the store is called at two moments rather than one.

# Lessons Learned

The invariant did its work by being checkable rather than aspirational. "Owns sequencing
only" reduces to two mechanical questions — what does this module import, and what sits
between the stage calls? — and both are answerable in seconds. The import list is two pure
functions and a wall of types; the `run` body with comments stripped is the pipeline and
nothing else. An invariant that can be verified by reading an import list is worth more than
one that has to be argued.

The most valuable finding was a *missing test for a decision the comments made*. Prose in a
docstring is not a constraint; it is an intention that the next contributor can undo without
any signal. Letting a notifier failure surface is a real architectural choice — swallowing it
would import a retry policy into the one component forbidden to hold policy — and until it was
pinned by a test it was one well-meaning try/catch away from evaporating. Where a comment
claims a behaviour, a test should make the claim expensive to break.

Finally, the frozen plan asked for something impossible, and the contracts said so. Session
creation cannot produce a report because there is no identity to report under — a fact
visible only once `SessionRunFacts.session` was a required field staring back. Plans written
before the types exist will occasionally promise what the types forbid; the types win, and
the deviation gets written down rather than quietly resolved.
