# Purpose

Give the Review Store the one operation it is missing: writing a session's rendered
`ReviewPackage` to `pending/<session-id>/review-package.md`.

> **Why does this task exist?** EOS-302 *anticipated* the file ‚Äî "`review-package.md` is
> written to the same session directory by the M5 projector ‚Äî the store simply guards/hosts
> that directory" ‚Äî but EOS-D6 froze the store's surface at four operations
> (`saveCandidates` / `saveReport` / `appendLog` / `locate`), none of which can write it.
> The anticipated mechanism ("the projector writes it") **contradicts EOS-D6's central
> holding**: the store owns the per-session layout and serialization, and every write is
> path-guarded. A projector writing the file itself would compose paths outside the store,
> duplicate layout knowledge, and bypass the lexical/symlink guards ‚Äî reintroducing exactly
> the split EOS-D6 rejected. The file is in the frozen plan; only the mechanism was left
> ambiguous. This task settles it the way EOS-D6 implies.

---

# Objective

Add a **fifth** operation to the store, in the shape of the existing four:

- **`ReviewStore.saveReviewPackage(sessionId: string, reviewPackage: ReviewPackage): Promise<void>`**
- Implemented in `createFilesystemReviewStore` ‚Äî writes `reviewPackage.markdown` to
  `pending/<sessionId>/review-package.md`, reusing the existing `assertSegment` /
  `resolveInRoot` / `assertNoSymlinkEscape` / `mkdir -p` idioms verbatim.

The completed per-session layout:

```
<destination>/pending/<session-id>/
    candidates/<candidate-id>.json     canonical CandidateKnowledge (EOS-302)
    report.json                        the SessionReport            (EOS-302)
    review-package.md                  the human-readable projection ‚óÑ‚îÄ‚îÄ this task
    log.md                             append-only execution log     (EOS-302)
```

# Persistence Invariant (unchanged, and reaffirmed)

This task adds a *file*, never a *permission*. The store still:

- **never invokes git** ‚Äî no commit, no staging (ADR-002 ¬ß4, AJS-005 ¬ß7);
- **writes only inside the non-canonical destination** ‚Äî the new write goes through the
  same guards as every other; the non-canonical construction guard is untouched;
- **does not interpret** ‚Äî it writes `markdown` as given. It does not render, re-render,
  validate the prose, parse it back, or reconcile it against the candidates. **Consistency
  between the package and the candidates is guaranteed by construction** (the projector
  derives it from them each run; EOS-D4), never by the store checking;
- **is deterministic in layout** ‚Äî one fixed path per session.

**Why markdown, when the store's other artifacts are JSON:** the store is domain-aware ‚Äî it
knows the *artifacts* (EOS-D6). `ReviewPackage` **is** a markdown projection by contract
(EOS-D4); persisting it as `.md` is honouring its type, not weakening the store's JSON
convention. Serializing the projection as JSON would produce a file no human could read ‚Äî
defeating the artifact's entire purpose.

# Scope

## Included

- `saveReviewPackage` on the `ReviewStore` interface (`store/ReviewStore.ts`) and in
  `createFilesystemReviewStore` (`store/FilesystemReviewStore.ts`).
- Behaviour tests, extending the existing `tests/end-of-session/review-store.test.ts`
  (temp dirs, no real vault ‚Äî the EOS-302 precedent):
  - writes `review-package.md` at the documented path with the package's `markdown` verbatim
    (byte-for-byte round trip);
  - **overwrite, not append** ‚Äî re-saving a session's package replaces it (the package is
    regenerable and single-valued, unlike `log.md`); contrast with the `appendLog` test;
  - the `sessionId` single-segment guard applies (`..`, separators, NUL rejected);
  - the operation is present on the frozen handle, and the surface is now **exactly five**
    operations (the existing exact-surface test is updated deliberately).
- No new public operation: `saveReviewPackage` is a method on an already-exported
  interface, so the EOS-007 drift-guard manifest is **unchanged** (operation count stays
  20) ‚Äî worth stating at review.

## Not Included

- Rendering the package ‚Äî **EOS-403** (the store persists what it is given).
- Calling this operation ‚Äî **EOS-406** (orchestration).
- Reading the package back, listing packages, or any SPEC-004 read API ‚Äî SPEC-004 reads the
  layout when it is built; this task does not speculate about its needs.
- Any change to `saveCandidates` / `saveReport` / `appendLog` / `locate`, the non-canonical
  guard, or the path guards ‚Äî all frozen and untouched.

# Design Notes / Open Decisions (for the Planning Review)

- **Store operation vs. projector-writes-the-file ‚Äî RATIFIED: store operation
  ([EOS-D8](../decisions/EOS-D8-review-store-save-review-package.md)).** The reviewer
  accepted the store operation at the M5 Planning Review, recorded as a distinct decision
  that **extends** EOS-D6. The reasoning is preserved below and canonically in EOS-D8.
  - **(a) `saveReviewPackage` on the store (recommended).** Consistent with EOS-D6: one
    owner of the layout, one guarded write path, a clean call site in EOS-406
    (`store.saveReviewPackage(id, pkg)` alongside `saveCandidates`/`saveReport`), and
    SPEC-004 depends on a named contract for the whole session directory rather than for
    three-quarters of it. Cost: a fifth operation on a frozen interface.
  - **(b) The projector (or EOS-406) writes the file directly**, as EOS-302's prose hinted.
    Cost: layout knowledge leaks out of the store; the write skips the store's guards; the
    projector gains a filesystem dependency and stops being pure (undoing EOS-403's central
    property). **Rejected** ‚Äî it contradicts EOS-D6's holding and EOS-403's invariant at
    once.
  - **Ratified: (a).** The store already *hosts and guards* the session directory (EOS-302's
    own words); owning the fourth file in it is the smaller, more consistent change. The
    reviewer chose a **distinct decision record (EOS-D8)** over an EOS-D6 amendment; EOS-D6
    carries a forward pointer to it and remains Accepted, since EOS-D8 extends rather than
    revises it.
- **Overwrite vs. append.** The package is single-valued and regenerable per session;
  `review-package.md` is overwritten (like `report.json`), unlike the append-only `log.md`.
  Called out because the store's only other markdown file is append-only, and the asymmetry
  should be deliberate.
- **Should the store validate the package (`parseReviewPackage`) on write?** **No** ‚Äî the
  store does not interpret (Persistence Invariant), and the projector already validated it
  (EOS-403). Re-validating would move contract enforcement into persistence and duplicate an
  owner. Consistent with `saveCandidates`, which trusts its input.
