# Purpose

Define the Context Package contract.

The Context Package is the primary output of the Context Builder and the primary input to AI coding agents.

This task establishes a stable, immutable data contract that future milestones will populate but not redesign.

No collection, ranking, or generation logic is implemented.

---

# Objective

Create the Context Package contract.

The implementation should provide:

- runtime validated package schema
- strongly typed package model
- immutable package structure
- public package exports

The task defines **what** a Context Package contains, not **how** it is created.

# Design Principles

The Context Package should be:

### Immutable

A generated package never changes.

---

### Self-Contained

Everything required by the coding agent exists within the package.

---

### Explainable

Every context section should be traceable to its source.

---

### Deterministic

The same collected information always produces the same package structure.

---

### Portable

The package contract is independent of output format.

# Worklog Summary

**2026-07-07**

Implemented the public **Context Package contract** under
`src/context-builder/package/`, extending the CB-001/CB-002 module. No provider,
collection, ranking, token, assembly or rendering logic was added — the task
defines *what* a package is, not *how* it is produced.

- Added `package/schema.ts` — Zod `contextPackageSchema` (`.strict()`) with the
  five top-level elements from AJS-002 Appendix B / CB-003 (`metadata`,
  `sections`, `references`, `explainability`, `summary`); the component schemas
  (metadata, section, source reference, explainability, entry); the
  `SECTION_KINDS` (12 canonical Appendix B sections) and `REFERENCE_TYPES` (9
  AJS-002 source categories) const enums; a `superRefine` enforcing unique
  reference ids, unique section kinds and referential integrity; a private
  `deepFreeze`; and `parseContextPackage()` (validate + deep-freeze).
- Added `package/types.ts` — a `DeepReadonly` utility plus the package types
  (`ContextPackage`, `ContextPackageMetadata`, `ContextSection`,
  `SourceReference`, `ContextExplainability`, `ExplainabilityEntry`,
  `ContextSectionKind`, `ReferenceType`), all inferred from the schema.
- Added `package/index.ts` — Context Package barrel.
- Updated `src/context-builder/index.ts` — public exports and scope note.
- Updated `src/context-builder/README.md` — Context Package contract section.
- Added `decisions/CB-003-context-package-contract.md`.

Validation: `npm run typecheck` and `npm run build` pass. A runtime smoke test
(17 assertions) confirmed: a valid package parses and is deeply frozen; strict
mutation throws; and each invalid form fails — unknown top-level/metadata key,
bad datetime, missing required field, unknown section kind, unknown reference
type, duplicate reference id, duplicate section kind, dangling section reference,
dangling explainability reference.

**Decision** (see `decisions/CB-003-context-package-contract.md`): reify AJS-002
Appendix B into a portable, serialization-independent contract; sections as a
`kind`-tagged ordered array with opaque `content`; first-class typed references;
score-free explainability; deep immutability; structural invariants enforcing the
*Self-Contained* and *Explainable* principles.

---

# Lessons Learned

- Appendix B is written as a `Context.md` schema; the *Portable* principle forces
  it to be reified into a serialization-independent structure. Keeping section
  bodies as opaque `content` strings cleanly defers rendering to M4 while keeping
  the contract stable.
- Modelling references as first-class typed records (with ids) is what turns
  "explainable" and "self-contained" from prose principles into *validated*
  invariants (referential integrity), which also produced the strongest
  "invalid package fails validation" test cases.
- "Avoid filesystem paths" and Appendix B's "Files Likely to Change" are in
  tension; resolving it as *logical* references (domain content) vs.
  *implementation* paths (excluded) kept the contract clean without dropping a
  required section. Documented as an interpretation, not a redesign.
- Deep-freeze + `DeepReadonly` is the correct nested analog of CB-002's shallow
  freeze; a nested package needs immutability at every level to honour the
  *Immutable* principle.
