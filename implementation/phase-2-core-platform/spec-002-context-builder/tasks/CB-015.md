# CB-015 — Selection Policy

**Status:** Planned

---

# Goal

Define the deterministic Selection Policy used by the Selection Engine.

The Selection Policy specifies how collected knowledge is evaluated, prioritized, filtered, ordered, and when exact duplicate KnowledgeItems are eliminated.

This task defines only the deterministic policy.

It does not execute selection, integrate with the Context Builder, assemble Context Packages, or introduce behaviour.

---

# Scope

This task includes:

- deterministic selection policy (executable platform behaviour)
- evaluation predicates
- filtering predicates
- prioritization and ordering as an ordered comparator chain
- duplicate elimination policy
- module documentation

The Selection Policy is implemented as executable platform code — pure, deterministic functions (evaluation and filter predicates, and an ordered comparator chain). It is not merely documentation.

This task does not execute the pipeline, construct a SelectionResult, or integrate with the Context Builder. That runtime execution is CB-016.

# Purpose

The Selection Policy defines the deterministic platform rules governing knowledge selection.

The policy determines:

- which KnowledgeItems continue
- which KnowledgeItems are excluded
- the canonical deterministic ordering of selected KnowledgeItems, via an ordered comparator chain
- when exact duplicate KnowledgeItems are removed

The policy never modifies KnowledgeItems.

---

# Inputs

The Selection Policy evaluates immutable KnowledgeItems contained within a CollectionResult.

The policy does not communicate with providers.

The policy does not execute pipeline stages.

---

# Outputs

Deterministic selection decisions suitable for Selection Engine execution.

The policy itself does not construct a SelectionResult.

---

# Design Principles

## Deterministic

Identical inputs always produce identical decisions.

---

## Reproducible

Selection decisions must be reproducible across executions.

---

## Ordered Comparator Chain

Ordering is expressed as an ordered comparator chain, not as scores or numeric priorities.

Each comparator is deterministic. The chain terminates with an immutable platform identifier (for example `KnowledgeItem.id`) to guarantee a stable total ordering of selected KnowledgeItems.

The policy introduces no scoring algorithms, no numeric priority values, and no business-specific ranking heuristics. It specifies deterministic guarantees, not ranking heuristics.

The resulting canonical ordering is the platform guarantee; SelectionResult exposes no explicit priority field.

---

## Platform-Owned

Selection decisions belong exclusively to the platform.

Providers never determine selection priority.

---

## Preserve Knowledge Identity

KnowledgeItems are never rewritten, merged, summarized, or transformed.

---

## Exact Duplicate Elimination

Only exact duplicate KnowledgeItems may be eliminated.

Knowledge consolidation is explicitly out of scope.

---

## Independent Evaluation

KnowledgeItems are evaluated using deterministic platform policy.

Selection policy does not depend on runtime state.

# Notes

The Selection Policy defines the deterministic decision-making model of the Context Builder.

Selection execution is intentionally deferred to CB-016.

This separation allows the policy to be reviewed, tested, and evolved independently of the Selection Engine implementation.

Implementation note: Define "exact duplicate" explicitly before implementing duplicate elimination. The definition must not rely on KnowledgeItem.id, since the identifier serves as the deterministic tie-breaker rather than the duplicate identity.
