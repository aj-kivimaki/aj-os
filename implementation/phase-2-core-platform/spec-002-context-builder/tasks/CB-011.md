# Purpose

Integrate the Collection Engine into the Context Builder.

This task connects the platform services established during Milestone 1 and the collection behaviour introduced during Milestone 2.

After this task, the Context Builder is capable of collecting deterministic knowledge from registered providers and returning a CollectionResult that contains **both** successful KnowledgeItems and CollectionErrors.

No knowledge selection, ranking, Context Package generation, or explainability is implemented.

## Integration Point

The Context Builder **composes** a Collection Engine constructed with the Provider Registry (`createCollectionEngine(registry)`). The Context Builder does not own the registry — it owns the engine.

The Context Builder exposes a collection entry point that receives a `KnowledgeRequest`, delegates to the Collection Engine, and returns the resulting `CollectionResult`. Both successful knowledge and collection errors are integrated into that single result.

# Design Principles

The collection pipeline should be:

### Deterministic

The same request should always produce the same CollectionResult.

---

### Composable

The Context Builder coordinates services.

Business logic remains inside the Collection Engine.

---

### Stateless

Collection should not create persistent runtime state.

---

### Minimal

Only behaviour required for deterministic collection is introduced.

---

### Platform-Oriented

The Context Builder orchestrates platform services.

It does not implement provider-specific behaviour.

# Worklog Summary

_2026-07-08_

Integrated the first end-to-end collection pipeline. The Context Builder now
composes a Collection Engine (`createCollectionEngine(registry)`) at construction,
owns it, and exposes `collect(request)` — a direct delegation that returns the
engine's `CollectionResult` unchanged. It remains a thin orchestration layer: no
inspection, mutation, filtering, ranking, deduplication or enrichment; all
collection business logic stays in the Collection Engine (CB-007/CB-010).

To let a single immutable handle own the engine and expose an unconditional,
request-only `collect`, the registry must be injected at construction. This
required an **approved architectural evolution** of the frozen CB-002 factory —
from `createContextBuilder(config)` to `createContextBuilder(config, registry)`
(required registry) — and adding `collect(request)` to the `ContextBuilder`
interface. The change was reviewed under the Implementation Guardrail before any
code was written; the rationale is recorded in
`decisions/CB-011-context-builder-integration.md`. No other public contract
changed (ContextBuilderConfig, ProviderRegistry, CollectionEngine,
KnowledgeRequest, CollectionResult, CollectionError, KnowledgeItem are all
untouched).

Validation: `npm test` (105 passed), `npm run typecheck`, and `npm run build` all
pass. Only the three CB-002/CB-005 factory-test call sites that depend on the old
signature were updated; behaviour tests remain CB-012's scope.

---

# Lessons Learned

- **A frozen factory signature must evolve when a service has to own a
  collaborator at construction.** Preserving the signature via a second factory or
  a staged/optional injection would have produced a partial or duplicate public
  API (conditional `collect`, hidden behaviour) — protecting the letter of the
  contract while degrading the coherence it exists to provide. The smallest
  *coherent* change (a required second parameter) was superior to any additive
  workaround.
- **Run the contract-change review before writing code.** Surfacing the CB-002
  evolution as an architectural decision (not an implementation detail) kept the
  contract-first discipline intact and produced an explicit, approved record.
