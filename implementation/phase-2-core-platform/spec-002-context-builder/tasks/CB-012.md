# Worklog Summary

_2026-07-08_

Completed Milestone M2's permanent collection behaviour tests on the existing
CB-006 Vitest foundation — no new framework, no new platform behaviour. The
contract-first tasks (CB-007…CB-010) had already landed their own per-task
behaviour tests as they were implemented (`collection.test.ts`,
`collection-errors.test.ts`, `collection-result.test.ts`,
`collection-execution.test.ts`), so CB-012 **completed** the suite rather than
re-authoring existing coverage (the "do not duplicate" guardrail):

1. **Context Builder integration** — new `context-builder-collection.test.ts`.
   CB-011 deliberately deferred its behaviour tests to CB-012, so nothing
   exercised `ContextBuilder.collect` end-to-end. The new suite asserts
   delegation to the owned engine, that the `CollectionResult` is returned
   **unchanged** (proved equal to a standalone engine over the same registry — no
   filtering, ranking, deduplication or enrichment), integration of items and
   errors in one partial result, determinism (registry order authoritative under
   delayed completion), and deep immutability.
2. **Deterministic error ordering** — extended `collection-execution.test.ts`
   with delayed-**rejecting** fixtures proving that provider *completion* order
   never influences *error* ordering (the existing suite only proved this for
   *item* ordering), plus an interleaved items+errors ordering case.

Result: the full suite grew from 105 to 119 tests across 10 files. Tests remain
fast, deterministic and public-API-only — no filesystem, network, randomness or
timing-based assertions (delays only prove that completion order is *ignored*,
never asserted against a clock).

Validation: `npm test` (119 passed), `npm run typecheck`, and `npm run build` all
pass. No platform contract was modified; CB-002…CB-011 remain frozen.

# Lessons Learned

- **In a contract-first milestone, the dedicated test task consolidates rather
  than recreates.** Because each CB-007…CB-010 task shipped with its own
  behaviour tests, CB-012's real value was closing the *remaining* gaps (the
  CB-011 integration seam and deterministic error ordering) and validating the
  whole suite — not re-authoring the enumerated coverage. Reproducing existing
  tests would have violated the "do not duplicate" guardrail while adding no
  protection.
- **Determinism must be proved for every ordered collection independently.** The
  suite proved completion order was ignored for *items* but not for *errors*; a
  contract that aggregates two ordered collections needs the ordering guarantee
  asserted for both. Delayed-rejecting fixtures are the deterministic way to do
  this (they prove completion order is irrelevant without timing assertions).
- **Test the integration seam through the public entry point, not by re-testing
  the collaborator.** `ContextBuilder.collect` is validated by asserting its
  result equals a standalone engine's — proving "thin orchestration, returned
  unchanged" without duplicating the CB-010 provider-execution mechanics.
