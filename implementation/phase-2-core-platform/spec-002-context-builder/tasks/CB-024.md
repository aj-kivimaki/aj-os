# CB-024 — Permanent Assembly Behaviour Tests

**Status:** Planned

> Milestone 4 — Context Assembly. Inherits the frozen M4 Planning Package and the
> frozen architecture (AD-001–AD-010). This document plans the task only; it
> introduces no product code. CB-024 authors the permanent regression suites for
> the Assembly stage, on the CB-006 testing foundation (mirrors CB-018).

---

# 1. Purpose

Author the permanent behaviour tests that lock in the Assembly stage's guarantees —
determinism, immutability, order preservation, empty-section and minimal-field
presence, contract conformance, and the `build()` pipeline — validated **only
through the public surface** (AJS-007 §6.1 Public-Surface Validation).

---

# 2. Scope

- **Engine boundary:** `createAssemblyEngine()` returns an immutable, stateless
  handle.
- **Determinism:** repeated `assemble` / `build` on identical inputs deep-equal
  (full-package equality, since `generated_at` is an injected input).
- **Immutability:** output is deep-frozen; the input `SelectionResult` is untouched
  (immutability by divergence).
- **Order preservation:** `references` and section contents mirror the canonical
  `selectedItems` order (RC-6).
- **Structure:** the four named sections are present and empty; minimal
  `explainability`/`summary` present; referential integrity holds.
- **Contract conformance:** `parseContextPackage()` accepts every produced package;
  deliberately malformed inputs are rejected (duplicate ids/kinds, dangling
  references).
- **Pipeline:** `build(request)` runs Collection → Selection → Assembly and returns
  `ContextPackage`; thin-orchestration equality (`build == assemble(select(collect))`).
- **Scope-negative:** no rendered output; explainability entries empty; profile inert.

---

Test fixtures — `KnowledgeItem`s and `SelectionResult`s built through the **public**

# 6. Implementation Notes

- Validate exclusively through the public API (`createAssemblyEngine`/`assemble`,
  `build`, `parseContextPackage`); import no internal mapping or policy — the
  implementation stays free to evolve (CB-018 precedent).
- Determinism asserted by identical-inputs deep-equality; immutability by divergence
  (out-of-canonical-order input, order-preserved output; frozen output).
- Prove thin orchestration by equality with a manual stage composition.
- Include scope-negative assertions so excluded capabilities cannot silently appear
  (no rendering, empty named sections, no computed explainability, profile inert).

---

# 7. Explicit Exclusions

- No product or behaviour code; no new contracts.
- No rendering, semantic-validation, explainability, or profile tests (out of M4
  scope, AD-009).
- No internal-module imports.
- No modification of any frozen contract or of the frozen Planning Package.
