# Purpose

Define the public contracts that allow knowledge to enter the Context Builder.

This task establishes the platform contracts that define the complete provider interface:

- **KnowledgeRequest** — the public request contract supplied to providers.
- **KnowledgeProvider** — the abstraction implemented by every knowledge source.
- **KnowledgeItem** — the canonical unit of knowledge returned by providers.

Together these contracts define **how knowledge enters the platform**, not how it is collected or processed.

No provider implementations, provider registry, ranking, collection logic, or Context Package generation are implemented.

---

# Objective

Create stable, runtime-safe contracts for requesting and supplying knowledge.

The implementation should provide:

- KnowledgeRequest contract
- KnowledgeProvider interface
- KnowledgeItem schema
- runtime validation (Zod)
- strongly typed contracts
- immutable data structures
- public exports

The goal is to establish the platform's input contract.

# Design Principles

The contracts should be:

### Minimal

Providers expose only the capabilities required by the Context Builder.

---

### Stateless

Providers should not maintain mutable runtime state.

---

### Deterministic

The same provider request should produce the same KnowledgeItem structure.

---

### Immutable

KnowledgeRequest and KnowledgeItems are immutable once created.

---

### Provider-Agnostic

The Context Builder must not know whether knowledge originated from:

- Handbook
- Wiki
- Project documentation
- Notion
- GitHub
- Jira
- MCP
- Future providers

Only the provider knows how knowledge is obtained.

# Public Contracts

## KnowledgeRequest

Represents a request for knowledge.

The request contract defines the public input to every KnowledgeProvider.

The request should contain only stable platform concepts and remain provider-agnostic.

Future milestones may extend the request contract without changing the provider interface.

---

## KnowledgeProvider

Represents a source capable of contributing knowledge to the Context Builder.

Responsibilities:

- identify itself
- describe itself
- receive a KnowledgeRequest
- contribute KnowledgeItems

Providers do **not**:

- build Context Packages
- rank information
- format output
- estimate tokens
- know about Context Builder configuration

---

## KnowledgeItem

Represents a single unit of knowledge.

A KnowledgeItem should contain only stable platform concepts.

Examples include:

- unique identifier
- source reference
- content
- metadata

The exact schema should follow the approved standards while remaining independent of any specific provider implementation.

# Implementation Notes

Use:

- Zod for runtime validation
- TypeScript type inference
- immutable structures
- explicit public exports
- a single immutable request object

Avoid:

- provider-specific fields
- implementation-specific metadata
- ranking information
- package information
- transport-specific concerns
- filesystem implementation details
- multiple primitive method parameters

The public API should favour stable contracts over expanding method signatures.

If additional fields appear necessary, justify why they represent stable platform concepts before introducing them.

# Worklog Summary

_2026-07-08_

Implemented the platform's knowledge **input contracts** in a new
`src/context-builder/providers/` module (`schema.ts`, `types.ts`, `index.ts`),
mirroring the `config/` and `package/` module layout for codebase consistency:

- **`KnowledgeRequest`** (`knowledgeRequestSchema`, `parseKnowledgeRequest`) —
  `{ project, task, branch?, commit?, issue? }` from SPEC-002 §7. The Context
  Profile and workflow type were deliberately excluded as Context Builder
  configuration/ranking concerns providers must not see.
- **`KnowledgeItem`** (`knowledgeItemSchema`, `parseKnowledgeItem`) —
  `{ id, source, content }`, where `source` **reuses the CB-003
  `sourceReferenceSchema`** so the same citable-source contract flows from
  provider output into Context Package references. No freeform metadata bag.
- **`KnowledgeProvider`** interface + **`providerMetadataSchema`** /
  `ProviderMetadata` — providers identify/describe themselves and expose
  `provide(request): Promise<readonly KnowledgeItem[]>` (single immutable request
  object, async-ready).

All contracts are `.strict()`, validated with Zod, and immutable (parse helpers
deep-freeze). Public exports added to `src/context-builder/index.ts`; module
README updated. Validation: `npm run typecheck` and `npm run build` pass; an
ad-hoc runtime harness exercised 11 checks (valid parses, strict rejection of
unknown keys, empty-content rejection, invalid source-type rejection, deep
immutability, and an implementable provider) — all green.

# Lessons Learned

- **Reuse across foundation contracts pays off.** `KnowledgeItem.source`
  reusing CB-003's `SourceReference` avoids a parallel source model and makes
  the eventual collection→package mapping trivial. Composition, not coupling.
- **"Metadata" in a task description is a trap.** The instinct to add a generic
  `metadata` bag would have re-introduced provider-specific data through the
  back door. Modelling explicit, stable fields keeps the contract
  provider-agnostic.
- **A single immutable request object future-proofs the interface.** New
  request inputs (issue type, workflow) can be added later without touching the
  `KnowledgeProvider` signature.
- **Minor duplication noted:** a small `deepFreeze` helper and the `DeepReadonly`
  type now exist in both `package/` and `providers/`. Left as-is to avoid
  refactoring reviewed CB-003 code; see the process recommendation below.

> **Engineering Rule**
>
> Providers contribute knowledge—not context.
>
> Knowledge flows through immutable platform contracts:
>
> **KnowledgeRequest → KnowledgeProvider → KnowledgeItem → Context Builder → Context Package**
>
> New knowledge sources should integrate by implementing the KnowledgeProvider contract without requiring changes to the Context Builder.
>
> Public interfaces should exchange immutable contracts rather than expanding primitive parameter lists.
