# Purpose

Implement deterministic execution of registered Knowledge Providers.

This task introduces the first runtime behaviour of the Collection Engine.

Providers are executed against the registry injected during CB-007. Successful providers contribute KnowledgeItems; failing providers contribute CollectionErrors. The engine assembles both into a CollectionResult.

Collection is **partial**: a single provider failure never aborts collection.

No knowledge selection, ranking, filtering, or Context Package generation is performed.

---

# Objective

Implement deterministic provider execution using the contracts established during Milestone 1 and CB-008 / CB-009.

The implementation should provide:

- provider execution
- deterministic execution order
- KnowledgeItem aggregation
- CollectionError surfacing (using the CB-008 contract)
- CollectionResult assembly (items + errors)
- documentation

The goal is to establish deterministic knowledge collection behaviour.

# Design Principles

Provider execution should be:

### Deterministic

Providers execute in a predictable order.

Registry order is authoritative. Provider completion order must never influence CollectionResult ordering.

---

### Partial

A single provider failure never aborts collection.

A failing provider yields a CollectionError; successful providers still contribute their KnowledgeItems.

---

### Stateless

Execution should not create persistent runtime state.

---

### Provider-Agnostic

The Collection Engine should know only the KnowledgeProvider contract.

It should not contain provider-specific logic.

---

### Immutable

Collected KnowledgeItems and CollectionErrors become immutable before being returned.

---

### Contract-Reusing

Failures are surfaced using the CollectionError contract from CB-008.

This task does not define a new error contract and does not introduce retry, recovery, logging, or policy.

# Behaviour

The Collection Engine should:

1. receive a KnowledgeRequest
2. execute registered providers in registry order
3. collect KnowledgeItems from successful providers
4. surface a CollectionError for each failing provider (using the CB-008 contract)
5. preserve deterministic ordering (registry order authoritative; completion order irrelevant)
6. assemble and return an immutable CollectionResult containing both items and errors

No further processing is performed.

# Lessons Learned

- **Concurrency without sacrificing determinism.** Running providers with
  `Promise.all` (or `allSettled`) preserves *input* order in the resolved array
  regardless of *completion* timing, so deterministic ordering needs no sorting —
  just aggregate in registry order. Pairing each provider with its own
  `try/catch` inside the mapped async function keeps the provider associated with
  its outcome and sidesteps `noUncheckedIndexedAccess` index lookups.
- **The engine sees only opaque rejections.** At execution time a failure is just
  a rejected promise; the engine cannot deterministically distinguish
  `invalid-request` vs `provider-unavailable`, so every rejection maps to the
  provider-agnostic catch-all `provider-error`. Finer classification belongs to
  providers that can classify their own failures, not the engine.
- **Construct-through-contract.** Building the result with `parseCollectionResult`
  (rather than a bespoke freeze) reuses the CB-009 contract as its own constructor
  — validation and deep-freeze come for free and the output cannot drift from the
  contract.
