# CB-014 — SelectionResult Contract

**Status:** Completed

---

# Goal

Define the immutable SelectionResult contract for Milestone 3.

SelectionResult represents the complete deterministic outcome of knowledge selection.

This task establishes only the data contract. It does not implement selection behaviour, evaluation, prioritization, ordering, duplicate elimination, Context Builder integration, or tests.

SelectionResult becomes the canonical output of the Selection Engine and the sole input to Context Assembly (Milestone 4).

# Purpose

SelectionResult represents the complete deterministic outcome of selection.

It preserves both:

- selected KnowledgeItems
- excluded KnowledgeItems

No KnowledgeItem is modified, rewritten, merged, summarized, or transformed.

SelectionResult preserves knowledge identity.

---

# Inputs

SelectionResult is constructed from:

- CollectionResult
- deterministic selection decisions

Selection behaviour is introduced in later tasks.

# Outputs

SelectionResult

containing:

- metadata
- selectedItems
- excludedItems

All collections are immutable.

`selectedItems` is an ordered collection. Its order is the canonical deterministic contract consumed by Assembly. SelectionResult exposes no explicit priority field.

# Design Principles

## Complete Outcome

SelectionResult represents the complete deterministic outcome of selection.

Both selected and excluded KnowledgeItems are preserved.

---

## Immutable Contract

SelectionResult is validated and deep-frozen.

---

## Preserve Knowledge Identity

KnowledgeItems remain unchanged.

SelectionResult contains references to original immutable KnowledgeItems.

---

## Deterministic Contract

Identical selection inputs always produce identical SelectionResult values.

---

## Composition

SelectionResult composes existing platform contracts.

No KnowledgeItem fields are duplicated.

No CollectionResult fields are redefined.

---

## Ordering Is The Contract

`selectedItems` is ordered, and that order is the public platform guarantee.

SelectionResult exposes no explicit priority field. Any priority used to derive the ordering is an implementation detail of the Selection Policy (CB-015). Assembly consumes `selectedItems` exactly in the order provided and performs no reordering.

# Worklog Summary

_2026-07-09_

Implemented the **SelectionResult contract** (CB-014) — the canonical, complete,
deterministic outcome of knowledge selection and the immutable boundary between
the Selection Engine and Context Assembly. The contract aggregates three stable
platform concepts: `metadata` (provenance — the answered `KnowledgeRequest`,
carried forward from CB-009), `selectedItems` (ordered `KnowledgeItem[]`, CB-004)
and `excludedItems` (`KnowledgeItem[]`, CB-004). It is a pure **data** contract:
no Selection Policy, Selection Engine execution, evaluation, prioritization,
ordering logic, filtering, duplicate elimination or Context Builder integration.

Followed the established data-contract pattern (CB-009): a new `selection/result/`
module with a `.strict()` `schema.ts` + module-local `deepFreeze` +
`parseSelectionResult()` (validate + deep-freeze), a `DeepReadonly` `types.ts`, and
an `index.ts` barrel, re-exported through `selection/index.ts` and the top-level
`context-builder/index.ts`. `selectedItems`, `excludedItems` and `metadata`
**compose** the existing CB-004/CB-009 schemas rather than redefining them —
`selectionResultMetadataSchema = collectionResultMetadataSchema`. There is **no**
priority/score/ranking field: ordering of `selectedItems` is the public contract,
enforced by `.strict()`. KnowledgeItems are preserved unchanged (identity is not
rewritten).

Added `tests/context-builder/selection-result.test.ts` — 16 contract tests
covering validation, composition, ordering-as-contract, leaked-priority rejection,
strictness, deep immutability and determinism.

Validation: `npm run typecheck`, `npm test` (135 passing across 11 files — up from
119; 16 new, none modified) and `npm run build` — all green. No frozen M1/M2
platform contract changed; no Contract Change Proposal required.

# Lessons Learned

- **Compose contracts; do not redefine them.** `selectedItems`, `excludedItems`
  and `metadata` all reuse existing platform schemas (`knowledgeItemSchema`,
  `collectionResultMetadataSchema`). Reusing the collection's metadata as
  provenance guarantees the selection outcome can never drift from the request it
  answers — the same "reuse the source contract" move CB-009 made for its own
  metadata.
- **Expose the guarantee, not the mechanism.** Selection needs a deterministic
  order; the ordered `selectedItems` array *is* that guarantee, so the contract
  publishes the order and withholds the priority/score that produced it. A ranking
  number would leak Selection Policy internals into a frozen public contract and
  invite drift between the number and the order. `.strict()` enforces the exclusion.
- **A contract validates shape, not policy.** Selected/excluded disjointness and
  id-uniqueness were deliberately *not* enforced — that is selection behaviour
  (CB-015/CB-016), out of scope here.
