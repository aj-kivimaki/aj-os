# Purpose

The Selection Engine executes the deterministic Selection Policy against a CollectionResult.

The result of execution is a SelectionResult containing:

- selected KnowledgeItems
- excluded KnowledgeItems
- deterministic metadata

Selection execution never modifies KnowledgeItems.

---

# Inputs

Selection Execution consumes:

- CollectionResult
- Selection Policy

The Selection Engine does not communicate with providers.

The Selection Engine does not access external services.

---

# Outputs

SelectionResult

containing:

- metadata
- selectedItems
- excludedItems

constructed using the immutable SelectionResult contract introduced in CB-014.

---

# Behaviour

The Selection Engine shall:

- expose the `select(collectionResult)` stage operation
- evaluate every KnowledgeItem using the Selection Policy
- determine whether each KnowledgeItem continues through the pipeline
- apply the Selection Policy's ordered comparator chain to produce the canonical deterministic ordering
- eliminate exact duplicate KnowledgeItems using the approved **Exact Duplicate Definition** in CB-015 (retain the first occurrence in canonical order; move every subsequent duplicate to `excludedItems`)
- construct an immutable SelectionResult whose `selectedItems` are in canonical order

The Selection Engine shall not:

- modify KnowledgeItems
- merge KnowledgeItems
- summarize KnowledgeItems
- rewrite KnowledgeItems
- construct Context Packages

# Worklog Summary

_2026-07-09_

Implemented **selection execution** (CB-016) — the first runtime behaviour of the
Selection Engine (CB-013) — as a stateless internal function the engine delegates
to, mirroring how CB-010 added `collectKnowledge` behind `CollectionEngine.collect`:

- **`src/context-builder/selection/selectKnowledge.ts`** — `selectKnowledge(collectionResult)`,
  a pure application of the CB-015 Selection Policy that sequences the policy's
  decisions and constructs the contract:
  1. **filter** — partition items with `isRetainedKnowledgeItem` (retain vs exclude);
  2. **order** — sort the retained items into canonical order with
     `compareKnowledgeItems`, over a **copy** (the frozen input array is never
     mutated);
  3. **dedupe** — eliminate exact duplicates in canonical order with
     `partitionExactDuplicates` (first occurrence retained; every subsequent
     duplicate routed to `excludedItems`);
  4. **assemble** — construct the SelectionResult via `parseSelectionResult` (CB-014),
     which validates and deep-freezes it.
  Execution embeds no evaluation, filtering, ordering or duplicate rule of its own —
  the policy owns every decision. It never modifies KnowledgeItems, communicates with
  no provider or external service, holds no runtime state, and exposes no
  priority/score/ranking value.

- **`src/context-builder/selection/createSelectionEngine.ts`** — extended the CB-013
  boundary with the anticipated `select(collectionResult): Promise<SelectionResult>`
  stage operation, which delegates to `selectKnowledge`. This is **additive** — the
  CB-013 interface was empty and its documentation anticipated exactly this method —
  so no frozen public contract is changed. `select` is a method on the existing
  handle, so the module's public **export** surface is unchanged.

Doc comments in `selection/index.ts` and `context-builder/index.ts` were updated to
record that `select` is now implemented (Selection Policy and selection-execution
behaviour stay internal, applied through `select`).

**Decisions.** (1) `excludedItems` — which the SelectionResult contract does **not**
order — is ordered with the same policy comparator purely to keep output fully
deterministic; this reuses `compareKnowledgeItems` and introduces no new comparator.
(2) `select` is `async` (returns `Promise<SelectionResult>`) to mirror the CB-010
`collect` stage operation and the frozen anticipated usage `await engine.select(...)`,
keeping both stage operations uniform for the CB-017 `build(request)` pipeline;
selection performs no I/O, so the promise resolves synchronously. Both are recorded
in the CB-016 decision record.

Validation: `npm run typecheck`, `npm test` (135 passing across 11 files — unchanged;
CB-016 adds no behaviour tests, per scope — those are CB-018) and `npm run build`
(emits `dist/context-builder/selection/selectKnowledge.js`) — all green. Behaviour was
additionally exercised with a throwaway script (filter, canonical ordering, exact-
duplicate elimination, determinism, deep-freeze, input untouched) before removal. No
frozen contract changed.

# Lessons Learned

- **Execution sequences policy; it never re-decides.** With evaluation, filtering,
  ordering and duplicate identity all owned by CB-015, the engine reduced to four
  ordered steps and a contract construction. Keeping the decision model out of the
  executor is exactly what let CB-016 be small — the discipline the CB-015/CB-016
  split exists to create.
- **Determinism demands an order even where the contract guarantees none.**
  `excludedItems` has no ordering guarantee, but "identical inputs → identical
  outputs" still requires a deterministic sequence. Reusing the policy's existing
  total-order comparator satisfied determinism without inventing new ordering logic.
- **Never sort the input in place.** The CollectionResult's `items` array is frozen;
  ordering must copy first (`[...items].sort(...)`) so immutable inputs stay
  immutable.
