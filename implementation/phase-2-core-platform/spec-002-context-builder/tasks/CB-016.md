# CB-016 — Selection Execution

**Status:** Planned

---

# Goal

Implement deterministic knowledge selection within the Selection Engine.

The Selection Engine executes the Selection Policy defined in CB-015 against the immutable CollectionResult and produces a validated, immutable SelectionResult.

This task introduces only selection behaviour.

It does not define the Selection Policy, introduce new contracts, integrate with the Context Builder, assemble Context Packages, or implement explainability.

---

# Scope

This task includes:

- Selection Engine `select(collectionResult)` execution
- deterministic evaluation
- deterministic filtering
- canonical deterministic ordering via the Selection Policy comparator chain
- exact duplicate elimination
- SelectionResult construction
- module documentation

This task introduces the runtime behaviour of the Selection Engine. It executes the Selection Policy defined in CB-015 and defines no ordering, filtering, or evaluation rules of its own.

---

# Explicitly Excluded

Do not implement:

- Context Builder integration
- Context Package generation
- explainability
- optimisation
- provider execution
- behaviour tests

Do not modify:

- Selection Policy
- SelectionResult
- CollectionResult
- Collection Engine
- Provider Registry

---

# Purpose

The Selection Engine executes the deterministic Selection Policy against a CollectionResult.

The result of execution is a SelectionResult containing:

- selected KnowledgeItems
- excluded KnowledgeItems
- deterministic metadata

Selection execution never modifies KnowledgeItems.

---

# Inputs

Selection Execution consumes:

- CollectionResult
- Selection Policy

The Selection Engine does not communicate with providers.

The Selection Engine does not access external services.

---

# Outputs

SelectionResult

containing:

- metadata
- selectedItems
- excludedItems

constructed using the immutable SelectionResult contract introduced in CB-014.

---

# Behaviour

The Selection Engine shall:

- expose the `select(collectionResult)` stage operation
- evaluate every KnowledgeItem using the Selection Policy
- determine whether each KnowledgeItem continues through the pipeline
- apply the Selection Policy's ordered comparator chain to produce the canonical deterministic ordering
- eliminate exact duplicate KnowledgeItems
- construct an immutable SelectionResult whose `selectedItems` are in canonical order

The Selection Engine shall not:

- modify KnowledgeItems
- merge KnowledgeItems
- summarize KnowledgeItems
- rewrite KnowledgeItems
- construct Context Packages

---

# Design Principles

## Deterministic Execution

Identical CollectionResults always produce identical SelectionResults.

---

## Stateless

Selection execution stores no runtime state.

---

## Immutable Inputs

CollectionResult is never modified.

---

## Immutable Outputs

SelectionResult is created through the CB-014 contract and returned as an immutable value.

---

## Policy-Driven

Selection behaviour is governed exclusively by the Selection Policy.

Execution never embeds additional decision rules.

---

## Preserve Knowledge Identity

KnowledgeItems remain unchanged throughout execution.
