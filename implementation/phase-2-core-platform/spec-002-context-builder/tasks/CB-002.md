# Purpose

Define the public configuration contract for the Context Builder.

This task establishes the immutable configuration model and public factory API that future Context Builder functionality will rely upon.

The objective is to define stable interfaces—not implement Context Builder behavior.

---

# Objective

Create the public configuration contract for the Context Builder.

The implementation should provide:

- a runtime validated configuration schema
- strongly typed configuration
- a minimal public factory API
- public exports

No collection, provider, ranking or context generation logic is implemented.

# Design Principles

The configuration contract should be:

### Immutable

Configuration is supplied once during construction and never mutated.

---

### Deterministic

The same configuration always produces the same behavior.

---

### Explicit

Every configurable capability must be represented in the schema.

Avoid hidden defaults.

---

### Minimal

Configuration should model platform capabilities rather than implementation details.

Do not configure features that do not yet exist.

# Worklog Summary

**2026-07-07**

Implemented the Context Builder's public configuration contract and factory API
under `src/context-builder/`, extending the CB-001 module boundary. No provider,
collection, ranking or context-generation logic was added.

- Added `config/schema.ts` — Zod `contextBuilderConfigSchema` (`.strict()`) with
  three required fields (`profile`, `explainability`, `outputFormat`), the
  `CONTEXT_PROFILES` / `OUTPUT_FORMATS` const enums, and
  `parseContextBuilderConfig()` (validate + `Object.freeze`).
- Added `config/types.ts` — `ContextBuilderConfig` (Readonly, inferred from the
  schema), `ContextProfile`, `OutputFormat`.
- Added `config/index.ts` — configuration barrel.
- Added `createContextBuilder.ts` — the `ContextBuilder` interface (single
  `readonly config` member) and the `createContextBuilder()` factory, which
  validates, freezes and returns an immutable handle.
- Updated `src/context-builder/index.ts` — public exports and scope note.
- Updated `src/context-builder/README.md` — Configuration & factory section.

Validation: `npm run typecheck` and `npm run build` pass. A runtime smoke test
confirmed: valid config accepted and frozen; invalid profile → `ZodError`;
unknown keys rejected (strict); missing fields rejected (no hidden defaults);
returned handle and config both frozen.

**Decision** (see `decisions/CB-002-configuration-contract.md`): three required
fields with no hidden defaults, spec-derived capabilities only, runtime + type
immutability, functional factory hiding internals.

---

# Lessons Learned

- Deriving config types from the Zod schema (as `src/api/schemas.ts` already
  does) gives a single source of truth and satisfies "strongly typed +
  runtime validated" without duplication.
- "Avoid hidden defaults" and Zod `.default()` are in direct tension; making all
  fields required keeps the contract explicit and makes the "invalid
  configuration fails validation" acceptance check meaningful.
- `.strict()` is what turns "explicit" from a types-only guarantee into a runtime
  one — it rejects speculative/unknown keys that would otherwise pass silently.
