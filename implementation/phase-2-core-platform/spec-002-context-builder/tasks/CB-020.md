# CB-020 — Section Composition Strategy

**Status:** Done

> Milestone 4 — Context Assembly. Inherits the frozen M4 Planning Package and the
> frozen architecture (AD-001–AD-010). This document plans the task only; it
> introduces no implementation and no code. CB-020 is a **decision task**: it fixes
> a deterministic rule and records it; the rule is realized in code by CB-022.

---

# 1. Purpose

Fix the deterministic strategy by which Assembly composes the `ContextPackage`
`sections` from its inputs (AD-010). This is the more fundamental structural
contract decision of Milestone 4 and precedes metadata composition (CB-021).

The strategy governs two things:
- how ordered selected knowledge maps to the **knowledge-derived** Appendix B
  section kinds, and
- the construction of the four **non-knowledge-derived** sections as structurally
  valid empty sections (Reviewer Decision A).

---

# 2. Scope

- A **total, deterministic, purely structural** mapping from a `KnowledgeItem`'s
  `source.type` (CB-004 `REFERENCE_TYPES`) to a knowledge-derived section `kind`
  (CB-003 `SECTION_KINDS`).
- A **stable, order-preserving** partition of `selectedItems` into sections: the
  canonical Selection order is preserved within each section and across sections
  (RC-6; AD-010).
- Construction of **Objective, Success Criteria, Constraints, Open Questions** as
  present-but-empty sections (Decision A) — never derived, generated, or inferred.
- Section-to-reference linking (`referenceIds`) that preserves referential
  integrity by construction (CB-003 invariant).
- The strategy recorded as `decisions/CB-020-section-composition-strategy.md`,
  **reviewer-approved before CB-022**.

---

# 3. Inputs

Described at the level of the strategy (no runtime execution here):
- `SelectionResult.selectedItems` — ordered `KnowledgeItem`s (CB-014).
- Each item's `source.type` (CB-004) and `source.id`.
- The canonical `SECTION_KINDS` enum (CB-003).

---

# 4. Outputs

A specified, deterministic section-composition rule: the `source.type → kind`
mapping, the order-preserving partition, the empty-section policy, and the
section→reference linking rule. No `ContextPackage` is produced.

# 6. Implementation Notes

- The mapping must be **total over `REFERENCE_TYPES`** and deterministic — every
  source type resolves to exactly one section kind, with no dependence on timing,
  randomness, or item content.
- **Purely structural**: the decision uses only frozen structural fields
  (`source.type`); it performs **no semantic evaluation**, keyword analysis, or
  inference (AD-002, AD-010).
- Ordering is the contract (CB-014): the partition is stable and preserves the
  canonical `selectedItems` order; sections carry no priority/score.
- Section `kind`s are unique (CB-003 invariant): items sharing a target kind
  compose one section.
- The four named sections are always present and empty (Decision A).
- This task **specifies** the rule; the executable mapping lands in CB-022. This
  spec/implementation split mirrors CB-015 (Selection Policy) → CB-016 (execution).
