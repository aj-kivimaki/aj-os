# Purpose

Establish the Collection Engine as an independent platform service.

The Collection Engine is responsible for coordinating deterministic knowledge collection.

This task establishes only the service boundary and public API.

The Collection Engine is **constructed with** the Provider Registry, but it does not execute providers, collect knowledge, or implement any business logic in this task.

---

# Objective

Create a stable public service contract for the Collection Engine.

The implementation should provide:

- Collection Engine module
- public factory `createCollectionEngine(registry)`
- public interface
- Provider Registry injected at construction
- immutable service handle
- public exports
- documentation

The goal is to establish the platform boundary that future collection behaviour will extend.

---

# Scope

## Included

- Collection Engine module
- public interface
- `createCollectionEngine(registry)`
- Provider Registry injection (construction only)
- immutable service handle
- public exports
- documentation

## Not Included

- provider execution
- knowledge collection
- CollectionError
- CollectionResult
- error handling
- ranking
- Context Package generation
- business logic

The registry is **held** by the engine after construction; it is **not executed** until CB-010.

---

# Design Principles

The Collection Engine should be:

### Minimal

Expose only the behaviour required by future collection tasks.

---

### Composed

The engine receives its Provider Registry at construction.

It does not own, discover, or configure providers.

---

### Stateless

Do not maintain mutable runtime state.

---

### Deterministic

Construction should always produce the same public service.

---

### Immutable

The returned service handle should be immutable.

---

### Platform-Oriented

The Collection Engine coordinates collection.

It is not itself a provider.
