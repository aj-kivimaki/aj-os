# CB-017 — Extend Context Builder Pipeline

**Status:** Implemented — `ContextBuilder.build(request)` composes the Collection Engine and the Selection Engine, runs Collection → Selection, and returns the SelectionResult unchanged; `ContextBuilder.collect` is superseded (see Worklog Summary)

---

# Goal

Extend the Context Builder pipeline by integrating the Selection Engine into the existing deterministic processing pipeline.

This task composes the Selection Engine with the existing Collection Engine and establishes the next stage of the Context Builder pipeline.

The Context Builder becomes responsible for orchestrating both deterministic collection and deterministic selection.

This task introduces orchestration only.

It does not implement selection behaviour, define selection policy, assemble Context Packages, or implement explainability.

# Purpose

The Context Builder owns the deterministic processing pipeline.

Following this task the pipeline becomes:

KnowledgeRequest
│
▼
Collection Engine
│
▼
CollectionResult
│
▼
Selection Engine
│
▼
SelectionResult

The Context Builder performs orchestration only.

The Context Builder exposes a single public entry point, `build(request)`, which executes this pipeline and returns the highest-level implemented result (SelectionResult at Milestone 3). Internally it invokes the stage operations `CollectionEngine.collect(request)` and `SelectionEngine.select(collectionResult)`; those stage operations and the intermediate CollectionResult remain internal to the pipeline.

## Approved Public API Evolution

`build(request)` is the Context Builder's single public entry point. This supersedes the Milestone 2 era `ContextBuilder.collect` public method — an approved public API evolution recorded in MILESTONES.md.

No frozen Milestone 1 or Milestone 2 platform contract is modified: the Milestone 2 collection behaviour is preserved as the `CollectionEngine.collect(request)` stage operation, and CollectionResult, the Collection Engine, the Provider Registry, and the configuration contract are unchanged.

## Regression Suite Reconciliation

Because `ContextBuilder.collect` is no longer a public API, its **builder-level** regression suite (the CB-011/CB-012 `context-builder-collection.test.ts`) is **retired** as part of this approved public API evolution. The permanent regression strategy evolves with the public surface:

- Collection behaviour remains permanently covered at the engine level by the Collection Engine regression suite (`collection-execution.test.ts`, CB-010) — no collection behaviour coverage is lost.
- The permanent `ContextBuilder.build(request)` **pipeline** regression suite is owned by CB-018, which validates the Collection → Selection pipeline through the new public entry point.

CB-017 introduces **no** replacement or bridge tests (behaviour tests are excluded from this task); permanent `build(request)` behaviour testing is left entirely to CB-018.

# Inputs

The Context Builder receives, through its public `build(request)` entry point:

- KnowledgeRequest

Internally it orchestrates:

- Collection Engine
- Selection Engine

Pipeline stages communicate only through immutable platform contracts.

---

# Outputs

SelectionResult

returned from `build(request)`, unchanged from the Selection Engine.

The Context Builder does not inspect, modify, filter, reorder, or enrich the SelectionResult.

---

# Responsibilities

The Context Builder shall:

- expose the single public `build(request)` entry point
- compose the Collection Engine
- compose the Selection Engine
- orchestrate deterministic pipeline execution by invoking `CollectionEngine.collect(request)` then `SelectionEngine.select(collectionResult)`
- return the resulting SelectionResult

The Context Builder shall not:

- implement selection policy
- perform evaluation
- execute providers directly
- construct Context Packages
- modify pipeline contracts

---

# Design Principles

## Pipeline Orchestration

The Context Builder owns the processing pipeline.

Pipeline stages remain independent.

---

## Composition

The Context Builder composes pipeline stages exactly once during construction.

---

## Stage Independence

Collection and Selection communicate only through immutable contracts.

Neither stage depends on the other's implementation.

---

## Thin Orchestrator

The Context Builder performs orchestration only.

Business logic remains inside individual pipeline stages.

---

## Deterministic Pipeline

Pipeline execution remains deterministic from KnowledgeRequest to SelectionResult.

# Worklog Summary

_2026-07-09_

Extended the Context Builder pipeline with the Selection stage and reconciled the
public entry point to `build(request)` — orchestration only, composing the
previously completed engines:

- **`src/context-builder/createContextBuilder.ts`** — the factory now composes
  **both** engines exactly once at construction: the owned Collection Engine
  (`createCollectionEngine(registry)`, CB-007/CB-010) and a Selection Engine
  (`createSelectionEngine()`, CB-013/CB-016). The `ContextBuilder` interface exposes
  a single public entry point, `build(request): Promise<SelectionResult>`, which
  invokes `collectionEngine.collect(request)` then
  `selectionEngine.select(collectionResult)` and returns the SelectionResult
  **unchanged**. The Milestone 2 era `collect(request)` public method is removed
  (superseded) — the approved public API evolution. The Context Builder implements no
  selection policy or collection behaviour, and does not inspect, modify, filter,
  reorder, deduplicate or enrich the result; determinism and immutability are
  inherited from the engines. The handle holds no runtime state and is frozen.

- **`src/context-builder/index.ts`** — updated the module narrative and the factory
  export comment to record the `build(request)` entry point and the `collect`
  supersession. No export was added or removed: `ContextBuilder`, `SelectionResult`
  (CB-014) and both engine factories were already public; `build` is a method on the
  existing handle.

- **`tests/context-builder/context-builder-collection.test.ts`** — **removed**. It
  exercised the now-superseded `ContextBuilder.collect`; its collection coverage is
  retained by the engine-level `collection-execution.test.ts` (CB-010), and the
  permanent `build(request)` pipeline suite is owned by CB-018 (see Regression Suite
  Reconciliation). No bridge tests were added.

No new contract was introduced: CB-017 composes CB-009, CB-010, CB-013, CB-014,
CB-015 and CB-016 unchanged. No frozen Milestone 1 or Milestone 2 platform contract
was modified beyond the approved `collect` → `build` public entry-point evolution.

Validation: `npm run typecheck`, `npm test` (123 passing across 10 files — 135 − 12
retired with the obsolete builder-level `collect` suite; no behaviour coverage lost),
and `npm run build` (emits the updated `dist/context-builder/createContextBuilder.js`)
— all green. The pipeline was additionally exercised with a throwaway script
(out-of-order ids, an exact cross-provider duplicate, and a failing provider):
`build` returned `selectedItems` in canonical order with the duplicate eliminated to
`excludedItems`, output identical to a manual two-engine composition, deeply frozen,
deterministic across runs, with no public `collect` method. Script removed after
verification.
