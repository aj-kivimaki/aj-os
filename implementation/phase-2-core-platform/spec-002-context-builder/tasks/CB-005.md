# Purpose

Define the immutable Provider Registry used by the Context Builder.

The Provider Registry is responsible for validating, storing, and exposing the collection of KnowledgeProviders available to the Context Builder.

This task establishes the registry contract and implementation.

It does **not** implement provider execution, provider discovery, dependency injection, plugin loading, collection, or Context Package generation.

---

# Objective

Create a simple, deterministic registry responsible for constructing and exposing an immutable collection of KnowledgeProvider implementations.

The implementation should provide:

- `createProviderRegistry()` factory
- ProviderRegistry contract
- immutable provider collection
- provider lookup
- duplicate provider validation
- public exports

The registry acts solely as a catalogue of providers.

# Design Principles

The registry should be:

### Deterministic

The same providers always produce the same registry.

---

### Immutable

The registry is created once and never mutated.

---

### Minimal

The registry manages providers only.

---

### Provider-Agnostic

The registry must not know anything about provider implementation details.

---

### Explicit

Duplicate providers are rejected immediately during registry construction.

# Public Contract

## ProviderRegistry

Represents the immutable collection of KnowledgeProviders available to the Context Builder.

Responsibilities:

- validate providers
- reject duplicate providers
- expose registered providers
- retrieve providers

The registry is created once through a factory function and never mutated.

The registry does **not**:

- execute providers
- discover providers
- load providers
- configure providers
- rank providers
- build Context Packages

# Worklog Summary

_2026-07-08_

Implemented the immutable **Provider Registry** in a new
`src/context-builder/registry/` module, following the `createContextBuilder()`
factory-based service pattern:

- **`createProviderRegistry(providers)`** — validates providers during
  construction (every provider must carry a non-empty string `id`; duplicate
  `id`s are rejected with an `Error`), then returns a frozen handle.
- **`ProviderRegistry`** contract — minimal surface: `providers` (readonly,
  insertion-ordered catalogue) and `get(id)` (lookup → `KnowledgeProvider |
  undefined`). No `has`/`size`/`filter` — those would be speculative.
- **Immutable & deterministic** — the returned handle and its `providers` array
  are frozen; ordering preserves the caller's insertion order so the same input
  always yields the same registry. Provider objects themselves are caller-owned
  and are not frozen (they carry a `provide` method).

Files: `registry/createProviderRegistry.ts` (interface + factory) and
`registry/index.ts` (barrel). Public exports (`createProviderRegistry`,
`ProviderRegistry`) added to `src/context-builder/index.ts`; module README
updated with a Provider Registry section.

Validation: `npm run typecheck` and `npm run build` pass. An ad-hoc runtime
harness (no test runner exists yet — CB-006) exercised 12 checks — construction,
lookup, unknown-id → `undefined`, insertion-order determinism, empty-registry
validity, duplicate-id rejection, empty-id rejection, and frozen handle/array —
all green.

# Lessons Learned

- **Not every foundation module needs a `schema.ts`.** The suggested layout
  (`schema.ts`/`types.ts`/`index.ts`) fits *data* contracts (config, package,
  providers) because they define Zod schemas. The registry is a *service* that
  composes the existing CB-004 `KnowledgeProvider` and adds no new data shape, so
  a schema file would be empty ceremony. Co-locating the interface with its
  factory (as `createContextBuilder.ts` does) was the more consistent choice.
- **Don't `.strict()`-parse a provider as its metadata.** `providerMetadataSchema`
  is `.strict()`, but a real provider also carries a `provide` method — strict
  parsing would wrongly reject valid providers. The registry validates only the
  `id` it keys on, keeping it provider-agnostic.
- **Deterministic ≠ sorted.** Preserving the caller's insertion order is already
  deterministic and respects the caller's explicit intent; sorting would silently
  reorder the catalogue. The simplest rule that satisfies the spec won.
- **Minor duplication noted (unchanged):** insertion-order + `Object.freeze` is a
  recurring immutability idiom across `config/`, `package/`, `providers/` and now
  `registry/`. Left as-is to avoid refactoring reviewed code; see the process
  recommendation.
