This task extends the permanent Vitest foundation established in CB-006.

(the approved public API evolution reconciled in CB-017), the obsolete builder-level

remains covered by the CB-010 Collection Engine regression suite and is **not**
re-tested or moved here.

# Worklog Summary

_2026-07-09_

Completed Milestone M3's permanent Selection behaviour tests on the existing CB-006
Vitest foundation — no new framework, no new platform behaviour, no contract change.
Unlike CB-012 (where every CB-007…CB-010 task had shipped its own per-task tests),
the M3 contract-first tasks split differently: CB-014 shipped its contract suite
(`selection-result.test.ts`), but **CB-013 (engine boundary), CB-015 (policy) and
CB-016 (execution) deferred their behaviour tests to CB-018**. CB-018 therefore
**authored** the deferred Selection Engine and pipeline suites while **consolidating**
(not recreating) the CB-014 contract suite:

1. **Selection Engine boundary** — new `selection.test.ts`. Validates the CB-013
   service boundary: the factory takes no construction dependency, returns a plain
   frozen handle exposing only `select`, is stateless (fresh and reused engines
   agree), and is deterministic in construction.
2. **Selection execution + policy** — new `selection-execution.test.ts`. Validates
   the CB-016 `select` behaviour and the internal CB-015 Selection Policy **only
   through the public API**: canonical deterministic ordering (stable total order via
   the terminal `KnowledgeItem.id` tie-breaker, independent of content and collection
   order), M3 filtering (every well-formed item is eligible; empty-content items are
   unreachable through the contract), exact-duplicate elimination with first-in-
   canonical-order retained and the rest routed to `excludedItems`, non-duplicate
   discrimination (same content ≠ same source; same source ≠ same content), metadata
   preservation, knowledge-identity preservation, input immutability (the frozen
   `CollectionResult` is sorted over a copy), deep immutability, determinism across
   runs, and SelectionResult contract conformance (schema round-trip; no priority
   field).
3. **Context Builder pipeline** — new `context-builder-pipeline.test.ts`. Validates
   `build(request)` as the single public entry point by asserting it **equals a manual
   two-engine composition** (`select(collect(request))`) over the same registry —
   proving thin orchestration that adds no behaviour. Also covers end-to-end canonical
   ordering, cross-provider duplicate elimination, metadata provenance, the empty
   registry, stage independence (a failing provider yields a partial collection that
   selection still consumes; errors never enter the SelectionResult), determinism and
   deep immutability. No engine internal is re-tested — engine-level collection
   behaviour stays owned by the CB-010 suite.

Result: the full suite grew from 123 to 160 tests across 13 files. Tests remain fast,
deterministic and public-API-only — no filesystem, network, randomness or timing-based
assertions, and no policy comparator/predicate/private helper is imported.

Validation: `npm run typecheck`, `npm test` (160 passed), and `npm run build` all pass.
No platform contract was modified; CB-002…CB-017 remain frozen. **Milestone M3 is
complete.**

# Lessons Learned

- **A dedicated validation task's shape depends on how the milestone's per-task tests
  were split.** CB-012 mostly *consolidated* existing per-task suites; CB-018 had to
  *author* the CB-013/CB-015/CB-016 behaviour suites that were deferred to it, while
  still avoiding re-authoring the CB-014 contract suite. The guardrail is the same —
  do not duplicate — but the work is larger when behaviour tests are deferred rather
  than shipped per task.
- **Internal policy is validated through the public surface, not by importing it.**
  The Selection Policy (comparators, predicates, duplicate helpers) is module-private.
  Its guarantees — canonical ordering, filtering, duplicate elimination — are all
  observable in the output of `select`/`build`, so testing through the public API
  proves the behaviour while leaving the implementation free to evolve (e.g. M5
  profile comparators prepended to the chain).
- **Prove orchestration by equality with a manual composition.** The cleanest way to
  show `build(request)` is a thin orchestrator that "re-decides nothing" is to assert
  it deep-equals `select(collect(request))` over the same registry — no engine
  mechanics are re-tested, and any future drift (the builder filtering, reordering or
  enriching) fails immediately.
- **Input immutability is best proven by divergence, not just `Object.isFrozen`.**
  Feeding an out-of-canonical-order `CollectionResult` and asserting the input array
  stays unordered while the output is sorted proves `select` copies before sorting —
  a stronger guarantee than a frozen-check alone.
