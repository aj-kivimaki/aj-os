# CB-023 — build(request) Integration

**Status:** Planned

> Milestone 4 — Context Assembly. Inherits the frozen M4 Planning Package and the
> frozen architecture (AD-001–AD-010). This document plans the task only; it
> introduces no code.

---

# 1. Purpose

Extend the Context Builder pipeline to Collection → Selection → **Assembly** so the
single public entry point `build(request)` returns a `ContextPackage`, wire the
construction-time timestamp injection (Decision B), and prove the Context Builder
remains a thin orchestrator (CB-017 evolution; PIPELINE-ARCHITECTURE §Public Entry
Point).

---

# 2. Scope

- Compose the `AssemblyEngine` in `createContextBuilder` alongside the existing
  Collection and Selection engines, once at construction.
- `build(request)` invokes `assemble` after `select`, returning the resulting
  `ContextPackage` unchanged. The entry-point **return type advances**
  `SelectionResult` → `ContextPackage` — the pre-anticipated evolution recorded in
  PIPELINE-ARCHITECTURE and the CB-017 docstrings (not a redesign).
- Wire the **construction-time injected timestamp** and supply it to `assemble` as
  the explicit `generated_at` input (CB-021 / Decision B).
- Prove **thin orchestration by equality**: `build(request)` deep-equals
  `assemble(select(collect(request)))` over the same registry and injected inputs.

---

# 3. Inputs

- `KnowledgeRequest` (the `build` request) — public signature unchanged.
- Construction-time dependencies: `config`, `registry`, and the injected timestamp
  source (Decision B).

---

# 4. Outputs

`build(request)` returns an immutable `ContextPackage` (the highest-level
implemented pipeline output at Milestone 4).

# 6. Implementation Notes

- The Context Builder adds **no behaviour** — it composes and sequences engines and
  returns the Assembly output unchanged; all determinism and immutability are
  inherited from the engines (thin orchestrator, AD-002).
- `build(request)`'s **public input signature is unchanged**: the timestamp is
  injected at Context Builder construction (Decision B), not added as a `build`
  parameter — so no public-input contract evolution is required. Only the **return
  type** advances, which is the pre-approved CB-017 evolution.
- Prove thin orchestration by equality with a manual stage composition (the M3
  technique) — any future drift (the builder filtering, reordering, enriching, or
  re-timestamping) fails immediately.
- Engines are composed exactly once at construction; the returned handle stays
  immutable.

---

# 7. Explicit Exclusions

- No `assemble` logic (CB-022); no section/metadata rules (CB-020/CB-021).
- No rendering, semantic validation, explainability computation, or profiles
  (AD-009).
- No permanent behaviour tests (CB-024).
- No change to Collection or Selection behaviour; no frozen-contract change beyond
  the pre-approved return-type advance.
