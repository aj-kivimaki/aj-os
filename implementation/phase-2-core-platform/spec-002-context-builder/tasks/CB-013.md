# CB-013 — Establish Selection Engine Service

**Status:** Completed

---

# Goal

Establish the Selection Engine service boundary for Milestone 3.

The Selection Engine is responsible for deterministic knowledge selection.

This task establishes only the service boundary. It does not implement selection behaviour, evaluation, prioritization, ordering, duplicate elimination, Context Builder integration, or tests.

The Selection Engine becomes the platform service responsible for transforming a CollectionResult into a future SelectionResult.

---

# Scope

This task includes:

- Selection Engine public interface
- createSelectionEngine(...) factory
- immutable SelectionEngine handle
- public exports
- module documentation

This task does not implement any selection behaviour.

---

# Explicitly Excluded

Do not implement:

- SelectionResult
- selection rules
- selection execution
- prioritization
- filtering
- duplicate elimination
- Context Builder integration
- behaviour tests

Do not modify any existing Collection Engine contracts.

Do not modify any Milestone 2 public contracts.

---

# Inputs

The Selection Engine is constructed during Context Builder composition.

Selection execution is not introduced in this task.

No runtime inputs are accepted.

---

# Outputs

An immutable SelectionEngine service handle.

No SelectionResult is produced.

---

# Public Contract

Introduce:

```ts
createSelectionEngine(...)
```

returning

```ts
SelectionEngine;
```

The interface exposes only the service boundary required by future milestones.

No behaviour is introduced.

No selection method is implemented.

The stage operation the Selection Engine will expose is `select(collectionResult)`, introduced with its behaviour in CB-016. This task names the boundary only; it does not add the `select` method, mirroring how the Collection Engine boundary (CB-007) preceded `collect` (CB-010).

---

# Ownership

The Context Builder owns the Selection Engine.

The Selection Engine owns deterministic selection behaviour.

The Selection Engine does not own:

- Collection Engine
- Provider Registry
- Knowledge Providers

The Selection Engine consumes CollectionResult through its future public interface.

---

# Design Principles

Establish the Selection Engine service boundary only.

---

## Immutable Service

The returned SelectionEngine handle is immutable.

---

## Stateless

The Selection Engine stores no runtime state.

---

## Composition

The Selection Engine is composed by the Context Builder.

The Selection Engine does not compose other pipeline stages.

---

## Pipeline Independence

The Selection Engine communicates only through immutable platform contracts.

No pipeline stage depends on another stage's implementation.

# Worklog Summary

_2026-07-09_

Established the **Selection Engine** service boundary, opening Milestone M3. Added
`src/context-builder/selection/` with `createSelectionEngine()` and the public
`SelectionEngine` interface (interface co-located with its factory, no `schema.ts`
— the engine introduces no new data contract), mirroring the CB-007 layout. The
factory takes **no arguments** and returns a frozen handle: unlike the Collection
Engine, which holds an injected Provider Registry, the Selection Engine has no
construction-time dependency — its `CollectionResult` input arrives at the future
`select(collectionResult)` argument (CB-016) and its Selection Policy arrives in
CB-015. The handle exposes **no members** and no `select` method (no placeholder
for later behaviour). Wired public exports through `src/context-builder/index.ts`
and documented the service in the module `README.md`. No behaviour, no
SelectionResult, no tests introduced (per scope); the frozen Milestone 1/2
contracts (CollectionResult, Collection Engine, Provider Registry) are unchanged.

Validation: `npm run typecheck`, `npm test` (119 passing across 10 files —
unchanged from M2), and `npm run build` all pass; the build emits the `selection/`
module.

# Lessons Learned

- **A boundary can be genuinely empty — and that is the honest design.** CB-007
  exposed its held `registry` for observability, but the Selection Engine has no
  construction-time dependency to hold: forcing an artificial member (or a
  `select()` stub) purely to make it "observable" would be a placeholder the task
  forbids. The empty, frozen handle is the correct minimal boundary; its members
  arrive with the behaviour that needs them (CB-014…CB-016).
- **The factory-service pattern absorbed a zero-dependency service unchanged.**
  The same co-located-interface, no-`schema.ts`, `Object.freeze` pattern applied
  even though there was nothing to inject — a sign the M1/M2 service precedent
  generalises across services with and without held dependencies.
- **Naming the future stage operation in docs (not code) preserves the seam.**
  Documenting `select(collectionResult)` as the operation CB-016 will add keeps
  the boundary's intent legible without shipping a stub, exactly as CB-007 named
  `collect` ahead of CB-010.
