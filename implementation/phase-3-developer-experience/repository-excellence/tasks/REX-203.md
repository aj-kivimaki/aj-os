# Purpose

**Why does this task exist?**

**This is the highest-value and highest-risk task in the entire package.**

REX-202 makes 40 type errors visible. They are **real strictness violations** under the repository's
own `noUncheckedIndexedAccess` and `exactOptionalPropertyTypes` — guarantees the project chose,
enforced on exactly half its code.

**The precedent is unambiguous: this gap hid three real defects during SPEC-003.** Not style issues —
a broken stub, an `exactOptionalPropertyTypes` violation, and an import of a type that does not
exist. Each passed every check the repository had.

**And two of the 40 are `TS2352` — `readonly`→mutable casts the compiler flags as *"may be a
mistake"* — in a repository whose immutability is contract-level**, where `deepFreeze` and
`DeepReadonly` are load-bearing and EOS-005 ratified their duplication rather than weaken them. **If
a real defect hides among the 40, that is where it is.**

---

# Objective

Resolve all 40 errors. **Triage each as *defect* or *type-narrowing* before fixing it** — and record
every defect as a finding rather than silencing it.

## ⚠️ The reviewer's standing instruction for this task (M2 Planning Review, 2026-07-17)

> **When reviewing fixes, optimise for preserving intent over minimising diagnostics. If resolving
> one diagnostic exposes a deeper design question, prefer surfacing that question over forcing the
> repository back to zero warnings as quickly as possible. Repository Excellence exists to improve
> repository quality, not merely repository metrics.**

And on what success means here:

> *The objective is not "make TypeScript happy." The objective is: **determine whether each
> diagnostic represents a typing deficiency or a behavioural defect.*** … *If one of those casts
> turns out to hide a genuine behavioural defect, **discovering that is success, not failure**.*

**This reframes the acceptance criteria below.** "0 errors" is the *floor*, not the goal — a run that
reaches zero without a recorded verdict per diagnostic has **failed this task**. A run that surfaces
a design question and defers it with evidence has **succeeded**, even if a diagnostic remains and
must be narrowly suppressed with the question named.

# Implementation Notes

**Triage before fixing. Every error, individually.** The temptation is a sweep of `!` and
`?.` that turns 40 into 0 in an afternoon. **That would silence precisely the signal this task
exists to surface**, and it would do it invisibly.

For each error ask: **what is this error telling me about the code under test?**

**Three possibilities** — the taxonomy the reviewer fixed at the REX-201 review:

| Verdict | Meaning | Action |
|---|---|---|
| **1. An incorrect type** | The test indexes an array the contract guarantees is populated; the compiler cannot know. | Add the guard or assertion. **Low risk.** |
| **2. A real behavioural defect** | The test asserts something that cannot hold, stubs a shape that no longer matches, or casts past a contract. | **Record as a finding. Fix failing-test-first** — make it fail for the right reason before making it pass. |
| **3. A design question requiring explicit judgement** | The diagnostic is correct, and answering it means deciding something about the design. | **Surface it.** Do not answer it by silencing the diagnostic. Record; escalate if it touches frozen work. |

> *"If resolving a diagnostic uncovers a deeper architectural concern, surface it. **Do not optimise
> for reaching zero diagnostics at the expense of repository truth.**"* — reviewer, REX-201 review.

**Verdict 3 is the one that gets skipped**, because verdicts 1 and 2 both end in a fix and feel like
progress. A design question ends in a *record*, which feels like not finishing. **It is finishing.**

**Start with the two `TS2352` casts** (`collection-result.test.ts:138`,
`selection-result.test.ts:174`). A `readonly`→mutable cast in a suite that exists to prove deep
immutability is **either a test that cannot fail, or a real hole in the guarantee**. SPEC-003's most
common review finding was *tests that cannot fail*. **Ask of each: what would make this fail?**

**`git-collection-integration.test.ts` carries 7 and `git-change-analyzer.test.ts` 5** — the two
suites over real git. `TS2532`/`TS18048` there mean the test assumes git returned something. **That
assumption is the test's actual subject**, so a bare `!` erases the thing being tested.

**Never weaken to go green.** No test removed, skipped, or `todo`'d; no assertion loosened; no
`expect` deleted. **[REX-D10](../decisions/REX-D10.md) makes this a milestone-level falsifier**, and
this task is where it would be violated.

**If a fix requires changing `src/`, stop.** No `src/` errors exist. A fix that reaches into `src/`
means a real defect in production code — **halt and report**; that is an FPCP conversation, not a
quiet edit.

# Worklog Summary

_2026-07-17._

All 40 resolved; `tsc -p tsconfig.test.json` → **0**. **Verdict: 40× incorrect type, 0 behavioural
defects, 2 design questions surfaced.**

**The task's stated highest risk was clean, and proving that was the work.** The two `TS2352`
readonly→mutable casts sit in suites whose job is proving deep immutability. **Verified empirically
that the tests can fail**: a frozen array's `push` throws `TypeError`; an unfrozen one does not — so
the assertion discriminates. Intent sound; the compiler named the fix (`as unknown as`). *Recording
that conclusion was as valuable as a defect would have been.*

**The `related` errors were the real find.** `relatedSchema` has `.default([])`, so a parsed
`SourceExtraction` **always** carries `related` — but the fixtures, typed as the Zod *output*, were
written as *input* and omitted it. **That is why `createWikiRenderer.ts:62` types it `| undefined`
and guards `?? []`: the tests taught production to defend against a case production cannot produce.**
Fixtures made faithful; renderer tests **still pass**, confirming the guard is dead. **Guard left in
place per the reviewer's ruling** — no diagnostic demands the `src/` change (REX-D10). → **DQ-1**.

**A fourth instance of the documented mechanism.** `96bc19d` (EOS-303) made `reviewPath` **required**
on the shared `AjConfig`, silently breaking a **SPEC-005** fixture. SPEC-003's retrospective logged
three instances of this gap hiding defects; the tool that closes it found the fourth. → **DQ-2**.

**Per-case, not by sweep.** 30 `noUncheckedIndexedAccess` errors split two ways: **10 already had a
length assertion** guarding the index — `!` is honest there, the guard fails first with a clear
message. **20 did not** — a bare `!` would have reported a missing element as *"cannot read
properties of undefined"* instead of stating what the test expected, so the guard went in.
**Assertions rose 387 → 395.**

# Lessons Learned

- **The most valuable diagnostic was the one that wasn't a defect.** The `related` errors looked like
  six stale fixtures. They were **one modelling error with a production consequence**: fixtures typed
  as parser output but written as parser input, which grew a defensive guard in `src/` for an
  impossible case. **Fixing the count would have hidden that; triaging it surfaced it.** The
  reviewer's instruction — *preserve intent over minimising diagnostics* — is the only reason it was
  found.

- **I wrote a vacuous check while validating this task.** My first assertion-count comparison ran
  `paste | bc` on an empty file list, compared empty strings, and **reported ✅**. It could not have
  failed. Caught it, rebuilt it against the working tree (not `..HEAD` — the edits were uncommitted),
  and proved it can fail. **This is the defect class the task exists to find, committed by the person
  looking for it**, three steps after writing that a proof which cannot fail is not evidence.

- **`noUncheckedIndexedAccess` is not noise — two-thirds of its complaints were tests missing a
  guard.** Twenty sites indexed an array without first asserting it had an element. Each would have
  failed with `TypeError` instead of a stated expectation. **The flag was right and the tests were
  thin**; the fix made them better, and the assertion count is the evidence.

- **Zero behavioural defects is a real result, not an anticlimax.** SPEC-003's precedent set the
  expectation that this gap hides defects — it hid three. This time it hid none, **and hid two design
  questions instead**. Reporting that plainly matters more than finding what was predicted.
