# Purpose

**Why does this task exist?**

`implementation/backlog/SPEC-003-specification-hygiene.md` records seven specification updates
agreed at the SPEC-003 planning review, with a self-declared deadline:

> *"…and **no later than the SPEC-003 implementation Freeze Review** (AJS-007 Documentation
> Synchronization)."*

That Freeze Review happened on 2026-07-17. **All seven boxes are still unchecked** (F-019).

The result is that `docs/specifications/SPEC-003-End-of-Session-Workflow.md` — the Draft that
*defines* the End-of-Session Workflow — **diverges from the frozen implementation it specifies**.
It names no CLI command though the command ships. §17 says only "Review package location" though
the location is a ratified decision (EOS-D2). §8 does not say which of its two outputs is
authoritative though EOS-D4 settles it. The backlog exists precisely to prevent *"silent divergence
between the specification and the frozen implementation plan"*, and the divergence happened anyway.

---

# Objective

Apply all seven backlog items to the SPEC-003 Draft as a **single batched revision**, bump its
version and change log, and close the backlog.

# Scope

## Included

The seven items exactly as recorded — no more:

1. **Related Specifications — add SPEC-004** (its direct downstream consumer). *(CONTRACTS.md)*
2. **CLI entry point** — document `aj session end` with `--since <ref>` and `--notes` under
   §6 Triggers / §13 Interfaces. *(EOS-006, M5)*
3. **Review storage location** — record `<handbook-vault>/knowledge-review/pending/<session-id>/`
   and `AjConfig.handbook.reviewPath` (default `knowledge-review`) in §17. *(EOS-D2)*
4. **Agreed v1 scope** — a capture-only vertical slice: one Git analyzer, manual trigger, no-op
   notification; **no git commit, no wiki generation** (deferred); pluggable seams. *(package README)*
5. **Canonical output vs. projection** — clarify §8: `CandidateKnowledge` is the canonical
   machine-readable output; `ReviewPackage` is a deterministic human-readable projection rendered
   from it; SPEC-004 consumes the structured candidates, not markdown. *(EOS-D4)*
6. **First-class Session** — a stable **opaque id**; trigger, branch, and git state are metadata,
   not identity. Relates to §7 Inputs and §12 State Model. *(EOS-D3)*
7. **Boundary contract reference** — reference `docs/architecture/CONTRACTS.md` and note that
   `CandidateKnowledge` is producer-owned by SPEC-003. *(EOS-D1, EOS-D5)*

Plus: bump the spec version and change log; close the backlog document.

# Implementation Notes

**Apply as one batched revision**, as the backlog itself directs. Seven small commits against a
Draft specification would produce seven version bumps for one coherent act.

**Verify each item against the code, not against the decision record.** The decision says what was
agreed; the code says what shipped. Item 2's flags must match `src/cli/index.ts:49-61`; item 3's
default must match `ConfigService.ts:138`. **Where a decision and the code disagree, halt and report
(§3)** — do not paper over it in the spec.

**On item 4 and the roadmap sentence.** The backlog's item 4 says to *"Note that the roadmap's 'owns
git commits' role is deferred beyond v1."* Within this task that means **adding the scope note to
the SPEC** — the actual corrections to `docs/README.md:74` and `ROADMAP.md:26` are **REX-106's**,
because those are different documents with a different blocker. Recording the boundary here so the
two tasks do not both edit ROADMAP.md, or both assume the other did.

**Respect SPEC-000's structure.** These are edits to §6, §7, §8, §12, §13, §17 and the frontmatter
of a document that conforms to a mandated 17-section template. **Do not restructure** — populate.

**The spec stays Draft.** README:98-100 defines Draft as *"approved for implementation, not yet
frozen"*. Nothing here promotes it, and REX has no authority to.
