# Purpose

**Why does this task exist?**

`README.md`, `ROADMAP.md`, and `CHANGELOG.md` are the three documents a newcomer reads first, and
all three describe a repository that no longer exists. README says SPEC-003 is *"Next"* (it
shipped). ROADMAP's single **"Resume Here"** pointer — the one place a reader looks to find the
next task — aims at completed work. CHANGELOG lists SPEC-003 under `[Unreleased] → Planned` and
**never records it as `Added`**, despite 26 tasks, five milestones, 11 decisions, and a shipped
command across PRs #6–#11.

This task also establishes the **assertion inventory**: the reusable artifact that turns "the docs
are accurate" from a judgement into a re-runnable test. That artifact is what makes M1 repeatable
rather than a one-off tidy-up.

---

# Objective

Make every claim in the three root documents true, and produce the assertion inventory that keeps
them true.

# Implementation Notes

**Every correction cites the code that falsifies the old text.** Not *"updated README"* — *"README
said SPEC-003 was next; `9bd051d` shipped it."* The citation is what makes the correction reviewable
rather than trusted.

**The assertion inventory is the real deliverable.** Extract every falsifiable claim these documents
make about the code into a table: the claim, the file:line, and **the command that proves or
disproves it**. This is what reframes "docs are accurate" from Judgement to Measurable — and that
reframing is the load-bearing move of the whole classification. Without the inventory, M1 is a tidy-up
that decays; with it, drift becomes detectable.

Keep it **honest about its limits**: the inventory covers claims that are *mechanically checkable*.
"The README is welcoming" is not one, and pretending otherwise would corrupt the classification. If
a claim resists mechanical checking, that is **evidence about the boundary** and belongs in the
retrospective.

**On F-010.** The fix is **removal, not update**. The count is the defect, not its value — updating
it re-arms the same trap. A document that cannot drift needs no synchronization. Replace it with a
description of the guarantee, or delete it.

**On F-009.** SPEC-003 is a substantial body of shipped work with no `Added` entry. Write the entry
from `MILESTONES.md` v1.33 and the PR history — **and describe capabilities, not counts.**

**On ROADMAP's "Resume Here".** The real next item is **REX** (this package), not SPEC-004 — the
reviewer's decision, recorded at the package Planning Freeze. `MILESTONES.md` v1.33 is explicit that
SPEC-004 is *not* to be started.

**Scope guard check.** Every edit here corrects a false statement about an existing implementation.
Nothing changes what the platform is, promises, or does. **In scope, cleanly.**
