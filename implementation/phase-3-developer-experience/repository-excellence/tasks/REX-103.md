# Purpose

**Why does this task exist?**

`README.md`, `ROADMAP.md`, and `CHANGELOG.md` are the three documents a newcomer reads first, and
all three describe a repository that no longer exists. README says SPEC-003 is *"Next"* (it
shipped). ROADMAP's single **"Resume Here"** pointer — the one place a reader looks to find the
next task — aims at completed work. CHANGELOG lists SPEC-003 under `[Unreleased] → Planned` and
**never records it as `Added`**, despite 26 tasks, five milestones, 11 decisions, and a shipped
command across PRs #6–#11.

This task also establishes the **assertion inventory**: the reusable artifact that turns "the docs
are accurate" from a judgement into a re-runnable test. That artifact is what makes M1 repeatable
rather than a one-off tidy-up.

---

# Objective

Make every claim in the three root documents true, and produce the assertion inventory that keeps
them true.

# Implementation Notes

**Every correction cites the code that falsifies the old text.** Not *"updated README"* — *"README
said SPEC-003 was next; `9bd051d` shipped it."* The citation is what makes the correction reviewable
rather than trusted.

**The assertion inventory is the real deliverable.** Extract every falsifiable claim these documents
make about the code into a table: the claim, the file:line, and **the command that proves or
disproves it**. This is what reframes "docs are accurate" from Judgement to Measurable — and that
reframing is the load-bearing move of the whole classification. Without the inventory, M1 is a tidy-up
that decays; with it, drift becomes detectable.

Keep it **honest about its limits**: the inventory covers claims that are *mechanically checkable*.
"The README is welcoming" is not one, and pretending otherwise would corrupt the classification. If
a claim resists mechanical checking, that is **evidence about the boundary** and belongs in the
retrospective.

**On F-010.** The fix is **removal, not update**. The count is the defect, not its value — updating
it re-arms the same trap. A document that cannot drift needs no synchronization. Replace it with a
description of the guarantee, or delete it.

**On F-009.** SPEC-003 is a substantial body of shipped work with no `Added` entry. Write the entry
from `MILESTONES.md` v1.33 and the PR history — **and describe capabilities, not counts.**

**On ROADMAP's "Resume Here".** The real next item is **REX** (this package), not SPEC-004 — the
reviewer's decision, recorded at the package Planning Freeze. `MILESTONES.md` v1.33 is explicit that
SPEC-004 is *not* to be started.

**Scope guard check.** Every edit here corrects a false statement about an existing implementation.
Nothing changes what the platform is, promises, or does. **In scope, cleanly.**

## Recorded deviation — F-004 and F-008 occupy the same two lines

**The acceptance criterion *"`ROADMAP.md:26` untouched — REX-106 owns it"* is not physically
satisfiable, and the M1 plan should not have contained it.**

`ROADMAP.md:25-26` is a single unit — **Resume Here item 1**:

> *1. **End-of-Session Workflow (SPEC-003)** — the orchestration layer that decides when to run the
> generator and **owns git commits** (the engine never commits).*

F-008 (this task's: *"Resume Here points at completed work"*) and F-004 (REX-106's: *"owns git
commits"*) are **the same two lines**. Removing the stale item — which F-008 requires — necessarily
removes the false claim. Leaving the item in place to preserve REX-106's boundary would mean
knowingly shipping a "Resume Here" whose first step is completed work: **the finding this task
exists to close.**

**Resolved by the reviewer's intent-preservation principle** (ruled at the REX-102 review):
*deletion removes the error but not the omission.* So:

- **REX-103** removed the stale item. The falsehood went with it. **F-008 closed.**
- **F-004 is not thereby closed.** REX-106 still owns **positively recording** that the commit role
  is deferred (ADR-002 / AJS-005 §7) and that **no component owns it** — which is the part that
  deletion loses, and the part REX-106's ruling actually governs. Its authority-citation requirement
  is untouched; **only the location moved.**

**Not an FPCP.** No milestone scope, objective, sequencing, acceptance criterion, or reviewer
decision changed — F-004 still closes inside M1, by the task that carries the ruling. This is task
ownership, which the reviewer ruled below the threshold at the REX-101 review.

# Lessons Learned

- **Findings are not a partition of the text.** The M1 plan assigned findings by ID and assumed the
  IDs carved the documents into disjoint pieces. **They don't** — F-004 and F-008 name the same two
  lines from different angles ("this claim is false" vs. "this section is stale"). This is the
  **second** boundary collision in M1, after F-020. Both were found by **validation, not review**,
  because both surfaced as an acceptance criterion that could not be met honestly.

- **"Delete the false claim" is rarely the whole fix.** Removing *"owns git commits"* makes the
  ROADMAP silent about who commits — and the honest answer, *nobody, deliberately, and the
  orchestration layer that would doesn't exist yet*, is **more useful than the sentence it
  replaces**. The reviewer's principle earned its place within one task of being recorded.

- **The assertion inventory's value showed up immediately, and not where expected.** Its first real
  use was not proving REX-103's own work — it was **proving A-06 still fails**, which is what
  confirmed the F-004/F-008 resolution left REX-106's job intact rather than silently absorbing it.
  **A checklist that can only report success cannot report a boundary.**

- **Stale "next steps" documentation has a compounding cost.** ROADMAP's "Resume Here" is described
  in its own header as *"the single 'where do I pick up' pointer"*, and it pointed at finished work.
  Of every stale claim in M1, this is the one most likely to have wasted someone's afternoon.
