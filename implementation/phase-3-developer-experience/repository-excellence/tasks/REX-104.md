# Purpose

**Why does this task exist?**

`docs/guides/` is where README sends anyone who wants to actually run AJ-OS, and it **actively
misleads them**.

`installation.md:51-56` tells a user that the wiki generator is *"implemented but **not yet wired to
a runnable command**"* and that seeing *"no generated wiki"* is *"a **known limitation**, not a setup
mistake."* `README.md:58` documents `aj wiki build`. **The two documents directly contradict each
other**, and the guide's advice is to accept a broken state that a shipped command fixes.

`configuration.md` calls itself *"Reference for AJ-OS settings"* and omits two of the three shipped
`handbook` keys — including `generatedWikiPath`, which README:68 calls *"one configuration
contract"*, and `reviewPath`, without which `aj session end` cannot be configured. It documents no
command beyond `aj ask`.

---

# Objective

Make the guides describe the commands and configuration that actually ship, so that following them
produces a working system.

# Implementation Notes

**Document configuration from `ConfigService.ts`, never from memory or the decision records.** The
service is what runs. Its defaults are the defaults. If `aj.config.example.json` and `ConfigService`
ever disagree, **halt and report (§3)** — that is a defect, not a documentation choice.

**F-002 is the highest-value fix in this task.** It doesn't merely omit a feature; it tells the user
that a broken state is expected and instructs them to tolerate it. A user following this guide today
concludes AJ-OS is half-built. Remove the limitation text, remove the Troubleshooting entry, and
document `aj wiki build` as the answer.

**Cross-check flags against `src/cli/index.ts` character by character.** A guide with a wrong flag is
worse than a guide with no flag: the first fails confusingly, the second sends the reader to `--help`.

**`aj session end` needs a home, not a mention.** It is the newest shipped capability and is
documented nowhere a user will look. It writes to `<vault>/knowledge-review/pending/<session-id>/`,
takes `--since <ref>` and `--notes <text>`, and **never touches canonical knowledge** — a property
worth stating, since it is the thing a cautious user will want to know before running it against
their vault.

**Scope guard check.** Every edit documents an existing implementation. Nothing changes behaviour.
**In scope, cleanly.**

## End-to-end validation — the guide was still wrong, and following it proved it

**The acceptance criterion said *follow* the guide, not *read* it. That distinction found a defect
the whole task would otherwise have shipped.**

Built a scratch handbook matching the guide's own description — *"a directory of notes"* — and ran
the documented flow. **`aj wiki build` crashed with an uncaught stack trace:**

```text
SourceConnectorError: Source directory does not exist: foundation (…/vault-test/foundation).
```

**Root cause:** `createKnowledgePipeline.ts:41` hard-codes
`HANDBOOK_SOURCES = ["foundation", "library"]`. The handbook **must** contain both directories, and
**no document said so** — not README, not `installation.md`, not `configuration.md`. Worse, this
task's own first draft had made it *more* wrong by asserting *"a directory of notes… you do not need
to bring one."*

**Fixed in the docs** (this task's half): `installation.md` gained a *"What the handbook must look
like"* section with the required layout, README gained a pointer to it, and Troubleshooting gained
the exact error string a user will paste into a search box.

**The second half is a code defect and is NOT this task's** — M1 changes no source:

> `SourceConnectorError` is **not** in `wiki.ts:45`'s catch list (`ConfigError || AIError`), so a
> first-time user gets a **raw stack trace** where the code has a friendly-message path sitting right
> there. **This is F-060 reproduced live** — the finding predicted that the enumeration was fragile;
> this proves it is **already incomplete**. Recorded as evidence on F-060 for **M5**. The
> Troubleshooting entry is the doc-side mitigation until then.

**After the fix, the full documented flow works from the guide's *minimum* config:**

| Step | Result |
|---|---|
| Config: `{ handbook: { path } }` only | Loads; defaults resolve to `wiki-generated` + `knowledge-review` — **proving the guide's claim that the other two are optional** |
| `aj wiki build` | ingested 1, **5 pages**, wiki at `<handbook>/wiki-generated/` incl. `index.md` |
| `aj ask "How does the Context Builder work?"` | Correct answer from the generated wiki, **4 citations** |
| `aj session end --notes "…"` | `completed`; wrote `<handbook>/knowledge-review/pending/<id>/` |

**The three guarantees the guide now states were proven, not asserted** — snapshot the canonical
sources + wiki, run `aj session end`, re-snapshot:

- **never commits** — `HEAD` byte-identical (`003ff9b` → `003ff9b`); the new file stayed untracked.
- **never generates the wiki / never modifies canonical** — combined hash identical
  (`9a578e53…` → `9a578e53…`).
- **the proof can fail** — tampering with a canonical file changes the hash, so the guarantee is
  **not vacuous**. *(An earlier check used a 2-minute mtime window and was **unsound**: it could not
  distinguish `session end` from the `wiki build` minutes earlier. Replaced rather than trusted.)*

# Lessons Learned

- **"Follow the guide" and "read the guide" are different validations, and only one of them works.**
  Every prior REX task validated by grep and diff, and those were sufficient because the claims were
  *about text*. This task's claims were **about what happens when you do the thing** — and the guide
  failed on step one, on a handbook built from its own description. **The finding existed because
  nobody re-ran the guide after the generator shipped; reading it again would have repeated exactly
  that failure.** This mirrors SPEC-003's own hardest-won lesson: *only the real thing finds real
  gaps.*

- **The fix made the document worse before it made it better.** My first draft asserted *"a directory
  of notes — you do not need to bring one"* — more confident, more welcoming, and **more wrong** than
  the stale text it replaced. A documentation pass that only corrects known findings can introduce
  new falsehoods with total sincerity. **The end-to-end run is what caught it.**

- **A prediction in the inventory came true within a day.** F-060 said the error enumeration was
  fragile and *"a new user-facing error silently breaks the friendly-message path."* The reality is
  worse and already shipped: `SourceConnectorError` was **never** in the list, so the very first
  thing a new user does — build a wiki with the wrong folder layout — bypasses the friendly path
  entirely. **The finding was right about the mechanism and too optimistic about the tense.**

- **An unsound check is worse than no check, and I wrote one.** My first "never generates the wiki"
  test used a 2-minute mtime window that also caught the `wiki build` from minutes earlier — it
  could not have failed for the right reason. Caught it, replaced it with a before/after hash, and
  **verified the replacement can fail** by tampering. This is precisely the class of defect SPEC-003
  logged as its most common review finding: *ask of every guard — what would make this fail?*
